## 你已经确认过的定义（北极星）

以下内容你已经亲口说过并在 2026-04-22 确认"没问题"，本文档在此基础上展开。

### 要解决的问题
老年用户使用信息服务 APP 的**交互问题**与**数字鸿沟**。

### 现有方案不足
1. 适老化改造只停留在页面层，未触及交互逻辑
2. APP 内置 AI 助手能力局限，只给文字建议
3. 老年人最终仍倾向线下求助工作人员

### 设计定位
**以智能代理为核心**，围绕它实现适老化设计与多模态交互。

### 能力隐喻
**模拟线下客服** —— 复刻老年人向窗口工作人员求助的场景。

### 主交互方式
**语音对话**。

### 核心流程
用户与代理对话 → 代理确认需求 → 自动跳转对应页面 → 引导 + 操作完成易失误环节。

### 三项机制
- **容错（草稿箱）**：流程状态和填写内容随时保存，意外中断后快速恢复
- **信任分级推进（L1/L2/L3）**：从低级别起步，使用中建立信任后适时推荐升级
- **操作记录（子女视角）**：记录代理+用户操作，子女可查

### 命名含义
- **受控** = 权限受控 + 行为受控
- **响应型** = 不主动挑事，只在用户有需求时才介入

## 🔑 横切全局原则（对话中定稿，对所有组生效）

### 【原则 1】代理永远不主动（2026-04-22 组 E 定稿）

代理在用户主动唤醒前，**完全沉默**；未被唤醒期间不说话、不出现、不弹窗、不响铃。
任何系统层面的提醒需求（升级、草稿、记录等）→ 统一走**静态按钮入口**，不走主动推送。

### 【原则 2】"确定性操作"代理止步（2026-04-22 组 B 讨论中定稿）

**定义**："确定性操作" = 一旦按下就会**对外产生真实结果、不可轻易撤销**的动作。
典型例子：**登录**、**提交表单**、**去支付**、**同意条款**、**确认修改**、**删除**、**发送申请**。

**规则**：代理可以**引导、填充、准备**这类操作的前置条件，但**按下那一下必须由用户亲手完成**。

**推论**：
- 这条原则**横跨所有 L 等级**——哪怕 L3 也不能替用户按确定性按钮
- L3 的"更高权限"仅指"能代填更敏感字段"（如身份证号），**不是"能替用户做决定"**
- "确定性" vs "非确定性"的区分是场景设计的核心：查询类操作（提交查询请求）虽有提交动作，但因**无对外后果、可重复**，不算确定性操作——代理可代做

### 【原则 3】权限一事一授（2026-04-22 组 C 定稿）

**定义**：代理权限**不是持久化的全局等级**，而是**任务级的当场授权**。

**工作方式**：
1. 用户发起任务（"帮我登录"/"帮我查养老金"等）
2. 代理在对话中**遇到需要升级能力的点当场问一句**（如"这次我帮您填验证码可以吗"/"接下来要填身份证号，这次我帮您填还是您自己填"）
3. 用户当场回答，**授权只对当前任务生效**
4. 任务结束，授权自动失效
5. 下次同类任务**重新问一次**

**推论**：
- **用户不再看到"权限等级"这个前台概念**——他们只会听到"这次要不要我帮您做 X"之类的情境化问题
- L1/L2/L3 能力矩阵（组 C C0 表）仍在代码中存在，但**用途变为"能力敏感度分类"**——决定什么时候该停下来问；不是用户可见的状态
- 消解了"设置里切换等级"的认知负担——老年人不需要理解、记住、维护"当前等级"
- 信任推进 → 由"一次次说'可以'的累积"自然实现，与开题报告里"信任由用户一次次建立"契合
- 原组 E 中"权限等级按钮"的具体设计**被本原则作废**——若后续需要"常做任务免问"的偏好记录，再另外讨论是不是要一个"代理使用偏好"入口

## 组 B · 代理的能力边界（业务范围）

### B1. 代理能操作哪些页面？

**答案（2026-04-22 对话中定稿）：聚焦 4 个核心场景做深**

| # | 场景 | 代理能力上限 | 代理止损点 |
|---|---|---|---|
| 1 | **账号登录 · 刷脸分支** | L1（语音引导） | 摄像头权限弹窗前语音解释；刷脸动作（眨眼/摇头）永远用户自己做 |
| 2 | **账号登录 · 验证码分支** | L2（代读短信 + 代填验证码） | 填完后停在"登录"按钮前，**用户亲手按**（确定性操作）|
| 3 | **医保缴费** | L2–L3（代填表单、代确认缴费对象）| 止步于"去支付"按钮前，**用户亲手按**；后续支付链路（支付宝/微信/输密码）不碰 |
| 4 | **医保查询** / **养老金查询** | L1–L2 全链路 | 纯查询无确定性操作，代理可从唤醒→对话→跳转→读取→播报一气到底 |

**两条登录分支都做**的学术价值：刷脸分支展示"纯 L1 多模态引导"；验证码分支展示"L2 代填 + 确定性按钮不代做"——两条合起来把"受控响应型"在两种输入模态下的分级控制都演示到位。

**不做的页面/功能**：
- 浙里办长辈版中 4 场景之外的页面（"我的"设置页、标准版首页、搜索结果页的其它服务入口等），代理被唤醒也答"这个我不会"（具体话术见 B2）
- 支付环节（明确退出代理能力范围）
- 纯生物识别动作（刷脸、指纹等肉身动作）

### B2. 代理不会的事情怎么办？

**答案（2026-04-22 对话中定稿）：区分情景应答**

| 情景 | 代理反应 |
|---|---|
| 用户问的事**浙里办根本没有**（订火车票、看天气、发微信） | 明说"浙里办没有这个服务"，不诱导用户去别的系统 |
| 用户问的事**浙里办有但代理不会**（例如"我的"设置页改头像）| 指路 + 坦白"这个功能在这里（指路），我暂时帮不了您，您可以自己试试" |

**隐喻**：像线下政务大厅的窗口工作人员——"我们这个窗口不办这个业务"（没有的业务）vs "那个业务您去那边窗口"（有的业务但不是本窗口职责）。

**待定细节**（到"代理操作范式"时细究）：
- 是否在第一次对不上话时报一次菜单（"我只会：查养老金、查医保、缴医保、帮登录"）
- 具体措辞（prompt 层面）

### B3. 代理理解错、听错时的兜底？

**答案（2026-04-22 对话中定稿）：每次执行前先复述确认**

代理收到用户指令后，执行**任何操作**前（无论是否属确定性操作、无论权限等级），**先复述一句**"我帮您查 XX，对吗？"等用户确认后才动。

**依据**：
- 老年用户存在方言不清晰、发音含糊等普遍情况，代理听错率不可忽视
- 默默执行跳转到错的页面，会触发老年人"我的指令没被理解"的焦虑
- 线下客服也是这样——核对一下再去办，是"服务感"的重要来源
- 多这一秒，心理成本值得

**待定细节**（到"代理操作范式 / prompt 设计"时细究）：
- 复述的具体话术
- 复述后的等待时间、默认超时行为
- 是否支持"说'不对'"语音打断

## 组 C · L1/L2/L3 的可执行定义

### C0. 权限等级能力矩阵（2026-04-22 对话中定稿）

| 能力 | L1 | L2 | L3 |
|---|:-:|:-:|:-:|
| 语音对话、理解意图 | ✓ | ✓ | ✓ |
| 复述确认（执行前必做） | ✓ | ✓ | ✓ |
| 页面跳转（帮用户找入口） | ✓ | ✓ | ✓ |
| 语音引导用户操作（"请您点这个绿色按钮"） | ✓ | ✓ | ✓ |
| 屏幕高亮目标元素 | ✓ | ✓ | ✓ |
| 代读外部内容（短信、屏幕文字） | ✗ | ✓ | ✓ |
| 代填**非敏感字段**（查询条件、缴费金额、手机号、地址等） | ✗ | ✓ | ✓ |
| 代填**敏感字段**（身份证号、密码、银行卡号） | ✗ | ✗ | ✓ |
| 按"登录 / 提交 / 去支付"等**确定性按钮** | ✗ | ✗ | ✗（横切原则 2）|

**记忆口诀**：
- **L1 = "只用嘴和眼"**（代理只说话、只高亮；手始终是用户的）
- **L2 = "嘴眼手，但手只碰普通信息"**（可代读、代填非敏感字段）
- **L3 = "嘴眼手，手可以碰敏感字段，但不能按 Send"**（可代填身份证号，但确定性操作仍用户按）

### C2 派生：敏感字段白名单（硬编码）

"敏感"是**硬编码白名单**，不由 LLM 判断：
- 身份证号
- 密码
- 银行卡号

其余（手机号、地址、验证码、查询条件等）一律算非敏感。

**依据**：LLM 判断敏感性不稳定，白名单更可控；3 项白名单覆盖绝大多数真正敏感场景。如后续需扩展（社保号、银行卡预留手机号等），直接加进白名单。

### C3 派生：L2 / L3 的边界问题（已由 C0 + 横切原则 2 消解）

原"L2 能填下拉但点查询要用户"这类纠结被两条规则干净切开：
- 代理**填字段**的边界 = 看字段是否在敏感白名单（C0 表 + C2 白名单）
- 代理**按按钮**的边界 = 看按钮是否是"确定性操作"（横切原则 2）
- 两条独立判断，不再需要"L2 能不能点查询"这种级别化纠结

查询类按钮（"查询"、"搜索"、"下一步"等非对外提交）→ 不是确定性操作，代理可按
确定性按钮（"登录"、"提交"、"去支付"、"同意"等）→ 代理永远不按

### C1. 级别切换是"一次设定终身"还是会话内可临时动？

**答案（2026-04-22 对话中定稿）：问题本身被作废——不存在"等级切换"**

原问题假设了"用户处于某个全局等级"这一前提。经对话讨论，**这个前提本身是老年用户认知负担的源头**：老人不会说"我现在在 L2 模式"，他们只说"这次你帮我"或"这次我自己来"。

**改为走**"权限一事一授"模型（已升为横切原则 3）：
- 不存在"当前等级"这个全局状态
- 代理在任务中遇到升级点当场问一次，用户一次次授权累积成信任
- 用户完全不需要理解 L1/L2/L3 这三个字——这只是代码里的分类

**原组 E "升级按钮" 被本答案作废。** 未来若需要"常做任务免问"之类的偏好记录，再讨论是否需要独立入口。

## 组 D · 草稿箱与操作记录（数据与查看）

### D1. 草稿保存什么粒度？

> 存储结构由这里决定。
> - **字段级**（"身份证号已填 3 位"）
> - **页面级**（"当前在缴费页，表单完整状态"）
> - **流程级**（"养老金查询流程走到第 3 步 / 共 5 步"）
> - **会话级**（"完整对话历史 + 所有填过的字段"）
> - **组合**：比如"页面级 + 流程级"

**答案（2026-04-27 对话中定稿）：页面级草稿**

只存**已完成字段的快照**——把表单中所有已填完的字段值打包保存。不存"填到一半"的字段（如身份证号只输了 6 位），不存对话历史。

**适用范围**：4 个核心场景中，真正需要草稿的主要是**医保缴费**（有表单要填）。查询类场景（医保查询、养老金查询）中断后重查成本极低，不需要草稿。登录流程本身很短，重来也不费事。

**依据**：
- 字段级（存半截输入）实现复杂且收益低——恢复半截身份证号没有实际意义
- 会话级（存完整对话历史）太重——重新说一句"帮我缴医保"就能开始
- 页面级在实现成本和用户体验之间取得平衡

### D2. 意外退出后下次进入，草稿怎么找回？

**答案（2026-04-27 对话中定稿）：草稿箱按钮常驻 + 代理被唤醒时顺带提醒**

- 界面上常驻**"草稿箱 (N)"按钮**，用户随时可自己点进去恢复
- 用户唤醒代理时，代理在**问候语之后顺带提一句**"您上次有个未完成的缴费，要继续吗？"

**不违反横切原则 1**：代理是被用户主动唤醒后才提的，属于被唤醒后的正常应答，不是主动弹出。

### D3. 子女怎么查看记录？

**答案（2026-04-27 对话中定稿）：子女拿老人手机直接查看**

场景假设：子女拿到老年用户的手机，在 APP 内点进"操作记录"页面直接查看。

- 不涉及多账号、不涉及远程查看、不涉及扫码共享
- 不需要独立子女端
- 实现最简：APP 内有"操作记录"入口，任何人拿到手机都能看

### D4. 子女"只看"还是"能介入"？

**答案（2026-04-27 对话中定稿）：只看记录，不能介入**

子女拿手机看操作记录是**纯被动监护**——了解代理帮老人做了什么、老人自己做了什么。不涉及远程通知、远程批准、远程操作。

### D5. 老人知不知道自己被子女查看？

**答案（2026-04-27 对话中定稿）：不需要额外告知机制**

场景是老人把手机递给子女看，**递手机这个动作本身就是知情同意**。不需要界面上额外提示"您的家人可查看记录"，也不需要开关。

## 组 E · 结构性矛盾 · **关键组·先答这里**

### E1. "响应型 = 不主动" vs "代理三处主动"

> 你第 6 段定义："**响应型 = 不主动挑事，只有用户有需求时才介入**"。
> 但前面几段里出现了三处代理**主动**的点：
>
> 1. 第 4 段：**"系统在适当时机推荐升级信任等级"** —— 这是主动推荐
> 2. 第 3 段（间接）：意外退出后代理主动问"要不要恢复草稿"
> 3. 第 2 段（边缘）：代理主动复述确认意图 —— 这是被唤醒后的，勉强算不违反
>
> **需要拍板**：
>
> - **选项 1：硬派响应型** —— 代理**从不主动说话**。升级只能用户去设置里升，草稿只能用户去草稿箱找。信仰"零打扰"。
> - **选项 2：软派响应型** —— 代理**主动但仅限安全信号白名单**，比如：
>   - 用户连续失败 3 次（可以主动提示）
>   - 流程中断后下次进入（可以主动问要不要恢复）
>   - 进入敏感页前（可以主动确认权限）
>   - 白名单外一律不主动
> - **选项 3：分级响应型** —— 代理的主动性随信任级别变化：
>   - L1：极度克制，绝不主动
>   - L2：可主动问恢复草稿
>   - L3：可主动推荐"这个也可以我帮您做"
>
> ⚠️ 这个问题是**所有其他组的源头**：
> - 如果选 1（硬派），组 C1 选项 3、组 D2 选项 1、组 C3 的任何"代理问"都作废
> - 如果选 2（软派），需要**逐条列白名单**
> - 如果选 3（分级），L 三级的主动性规则需要一张表

**你的答案（2026-04-22 对话中定稿）：**

**核心原则：代理不主动说话、不主动出现、不主动弹窗。**

依据：老年用户不喜欢不确定性，代理"突然冒出来"本身就是信任杀手。用户想用代理，必须**自己主动**唤醒。

三个"主动嫌疑点"的归一化处理：

| 原矛盾点 | 新处理 |
|---|---|
| 第 4 段"适时推荐升级信任等级" | ⚠️ **已被横切原则 3 彻底作废**——不再存在"升级"这个用户可见概念。原拟议的"权限等级按钮"也随之作废。信任推进改为"任务级授权的累积"自然发生。 |
| 第 3 段隐含"意外退出后代理询问恢复草稿" | 不询问。助手界面常驻一个**简单按钮**（"草稿箱 (N)"），用户自己点进去恢复。 |
| 第 2 段"代理复述确认" | **不算主动**——发生在用户唤醒后，是正常应答链的一部分，保留。 |

**原则的完整表述**：
- 代理**未被唤醒时** → 完全沉默，不出现、不提示、不响铃
- 代理**被唤醒后** → 正常应答、复述、确认、执行、反馈（这些都属"用户主动开场后的对话"，不违反原则）
- 任何系统层面的提醒需求（升级、草稿、操作记录等）→ **统一用 UI 上的静态按钮入口**表达，用户看得到、够得着、想用才点

**派生设计纪律**（本组定下的、适用于所有其他组）：
- 未来任何"代理应该主动告诉用户什么"的设计提议，默认答案都是"不"；改用"静态按钮入口 + 用户点击查看"。
- 代理的**措辞/话术 (prompt)** 细节留到"组 · 代理操作范式"再定。

## 组 F · 错误恢复（代理出错怎么办）

### F1. 代理执行中出错（跳错页、填错字段），用户怎么改？

**答案（2026-04-27 对话中定稿）：不需要专门的撤销机制**

B3 的复述确认机制已经解决了"代理理解错意图"的问题——跳错页面、做错任务在执行前就被拦住。

代理填字段前也会逐个确认（横切原则 3 的一事一授），填错的概率已经很低。万一还是填错了（如用户口音导致代理听错数值，用户又没注意就确认了），处理方式：

- **用户手动改**：直接点击填错的字段自己修正
- **语音告诉代理改**：如"缴费年度选错了，改成 2026"，代理重新填

不做逐步撤销、不做整个任务回滚——实际场景中填错一个字段改一个字段就够了。

### F2. 代理没听清 / 网络卡住时的兜底？

**答案（2026-04-27 对话中定稿）：常规兜底，不做复杂重连**

**没听清**：
- 代理说"不好意思没听清，您再说一遍"
- 连续 3 次仍听不清 → 提示"您也可以试试手动操作"，代理退出语音识别等待

**网络断开**：
- 界面显示"网络已断开"提示
- 代理停住，不做任何操作
- 网络恢复后用户重新唤醒代理继续

不做自动重连、不做离线缓存。
