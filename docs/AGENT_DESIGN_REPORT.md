## 一、代理定位：它是什么，不是什么

### 1.1 设计出发点

本项目的起点来自一个调研发现：老年用户并非不愿意线上办事，而是**不愿意在线上自己冒险办事**。社区访谈中多位受访者直接表达了这一心理：

> "我觉得自己弄肯定弄不好，还是去街道办保险。"  
> "一上来就要身份证号，我不敢填，怕是骗子。"

这种"信任人胜过信任系统"的心理特征，正是当前信息服务APP的适老化改造无法触及的深层障碍——仅仅放大字体、简化界面，并不能解决老年人对线上操作的根本不信任。智能代理"小浙"的设计目的，是将线下"向窗口工作人员求助"的体验**复现到线上**。

### 1.2 小浙是什么

**受控响应型智能代理**。

这个命名本身就是设计主张的浓缩：

- **响应型**——代理只在用户主动唤醒后才介入，不会自动弹出、不会主动提示、不会发通知。沉默是默认状态，开口是被邀请的结果。
- **受控**——代理的每一步操作都经过用户当场授权，用户始终保持最终控制权；涉及敏感信息时需要单独征得同意；真实决策动作（登录、支付）由用户自己完成。

**能力隐喻**：模拟线下客服。就像去街道办事处的柜台工作人员——帮你找表格、帮你填非关键信息、告诉你该点哪里，但最后签字盖章的是你本人。

### 1.3 小浙不是什么

| 对比对象 | 根本区别 |
|---|---|
| Siri / 小爱 / 微信 AI | 那些是通用助手，主动推送、主动回应各类需求；小浙聚焦 4 个政务场景，不接受范围外的请求，也不主动介入 |
| 传统 RPA（机器人流程自动化） | RPA 全权代执行，不与用户交互；小浙每个关键步骤都需要用户确认或亲手完成 |
| 无障碍辅助工具 | 辅助工具替用户输入/点击；小浙在确定性操作前明确止步，用户保留最后一步的决策权 |
| 老年版 APP 本身 | 老年版只是 UI 放大；小浙叠加在老年版之上，提供交互引导层 |

### 1.4 学术创新点定位

开题报告指出，当前研究存在一个尚未解决的张力：**智能代理的自主性 vs 老年用户对操作可控性的强烈需求**。传统智能代理的设计逻辑是"代理替用户做决定"，老年人既不愿独自操作、又不放心委托代理，形成双重拒绝。

本研究的学术贡献在于提出并实现了一种**分级授权的协助型代理模式**：通过能力矩阵（L1/L2/L3）和"权限一事一授"机制，在保留代理核心能力的同时，将用户对控制权的掌握制度化而非口头承诺——这是对现有适老化智能代理研究的结构性补充。

## 二、代理能力全景

### 2.1 能力分级矩阵

代理能力分为三个等级，等级决定代理"手"可以碰什么：

| 能力 | L1 | L2 | L3 |
|---|:-:|:-:|:-:|
| 语音对话、理解意图 | ✓ | ✓ | ✓ |
| 复述确认（执行前必做） | ✓ | ✓ | ✓ |
| 页面跳转（帮用户找入口） | ✓ | ✓ | ✓ |
| 语音引导（"请点这个绿色按钮"） | ✓ | ✓ | ✓ |
| 屏幕高亮目标元素 | ✓ | ✓ | ✓ |
| 代读外部内容（读短信验证码） | ✗ | ✓ | ✓ |
| 代填非敏感字段（手机号、缴费年度、金额等） | ✗ | ✓ | ✓ |
| 代填敏感字段（身份证号、密码、银行卡号） | ✗ | ✗ | ✓ |
| 按确定性按钮（登录 / 提交 / 去支付等） | **✗（永远不）** | **✗（永远不）** | **✗（永远不）** |

**记忆口诀**：
- L1 = 只用嘴和眼（说话 + 高亮，手始终是用户的）
- L2 = 嘴眼手，手只碰普通信息
- L3 = 嘴眼手，手可碰敏感字段，但不能按 Send

"敏感字段"不由 AI 判断，由**硬编码白名单**定义：身份证号、密码、银行卡号。白名单之外的一律视为非敏感。

### 2.2 能力边界：4 个场景的能力上限

| 场景 | 能力等级 | 代理止损点 |
|---|---|---|
| 登录 · 刷脸分支 | L1 | 刷脸动作（眨眼 / 摇头）永远用户自己做；代理只引导和高亮 |
| 登录 · 验证码分支 | L2 | 代填验证码后止步于"登录"按钮，用户亲手按 |
| 医保缴费 | L2–L3 | 表单全部填完后止步于"去支付"按钮，用户亲手按；支付链路代理不跟入 |
| 医保查询 / 养老金查询 | L1–L2（全链路） | 纯查询无确定性操作，代理可一气到底 |

### 2.3 能力之外的场景处理

代理对自己能力范围之外的请求有明确应对规则：

| 情况 | 代理应对 |
|---|---|
| 浙里办本身没有这个服务（如订火车票） | "浙里办没有这个服务，您可以试试 12306" |
| 浙里办有但代理不会（如改头像） | "这个功能在'我的'页面里，我暂时帮不了您，您可以自己试试"，同时高亮对应入口 |

## 三、交互流程设计：4 个核心场景

### 3.1 通用交互结构

所有场景的代理交互遵循统一骨架：

```
用户唤醒 → 代理问候
  → 用户说需求（语音）
    → 代理复述确认（执行前必做）
      → 用户确认
        → 代理执行（跳转 + 引导 + 代填）
          → 遇敏感操作：当场征得授权
          → 遇确定性操作：高亮止步，用户亲手按
            → 代理播报完成，面板自动关闭
```

复述确认是整套设计的安全阀：代理对理解错的指令，会在执行前被用户拦截，不留后患。

### 3.2 场景 1：登录 · 刷脸分支（L1 纯引导）

**典型用户**：不知道刷脸认证怎么操作的老年用户（68 岁，王阿姨）

**代理价值**：摄像头权限弹窗是老年用户恐慌的最高风险点。在旧版无引导的情况下，弹窗突然出现会让用户怀疑被攻击。代理**提前语音解释**弹窗的性质（"马上会弹出使用摄像头的请求，这是正常的，请点同意"），将这一情感低点化解。

**全程代理只"说话和高亮"**：勾选同意条款→高亮引导；点登录按钮→高亮引导；摄像头权限→语音解释；刷脸动作→语音指导，**用户自己完成**；登录成功→语音播报，面板自动关闭。

**情感峰值**：用户第一次独立完成了以往"绝对弄不好"的刷脸登录，掌控感与成就感同时达到峰值。

### 3.3 场景 2：登录 · 验证码分支（L2 代读代填）

**典型用户**：刷脸失败多次、不会切换 APP 查看短信的老年用户（72 岁，陈大爷）

**代理价值**：验证码流程需要在两个 APP 之间来回切换，是老年用户放弃率最高的环节。代理通过"一事一授权"请求当场获得用户同意，代为读取并填写验证码，消除跨 APP 切换的认知负担。

**关键设计决策**：代理填完验证码后，明确说"已帮您填好了，您看一下对不对"，让用户有机会检查，而非直接提交。"登录"按钮由**用户亲手按**，强化"这一步是我自己决定的"的掌控感。

### 3.4 场景 3：医保缴费（L2-L3，最复杂场景）

**典型用户**：表单字段多、不敢随便填身份证号的老年用户（65 岁，李奶奶）

**代理价值**：表单字段多且用户不清楚填什么。代理逐字段代填（带语音播报），遇到身份证号时**单独弹出授权卡片**，明确说出"想帮您填【身份证号】，这次可以吗"——明确标注敏感信息的名称，而非模糊授权。填写后显示脱敏结果（330…\*\*…6），既告知已填又保护隐私感。

**草稿箱子旅程**：若用户中途离开（如接电话、切换 APP），前端自动将已填字段保存至 IndexedDB。下次唤醒代理时，代理在问候后顺带提示"您上次有个未完成的医保缴费，要继续吗？"，大幅降低重来成本。

**止损点**：表单填完后，代理高亮"去支付"按钮并口头说明，然后停下来等用户。支付链路不跟入。

### 3.5 场景 4：养老金查询（L1-L2 全链路）

**典型用户**：找不到查询入口的老年用户（70 岁，张大爷）

**代理价值**：说一句话，直接到达查询结果，中间零操作。这是 4 个场景中情感曲线最平滑的一个——用户无需做任何决策，代理从导航到代填查询条件到代按"查询"（非确定性操作）一气完成。

**双通道播报**：结果出现时，屏幕大字卡片展示数字，同时语音播报"您的养老金本月发放了 XXX 元，已经到账了"。老年人即便没听清数字，抬头还能看到屏幕；即便没记住屏幕，耳朵里还有播报——两个通道互为补充，确保信息有效送达。

## 四、三项行为准则的落地机制

代理的三条横切原则不是口头约定，而是在产品层面和技术层面都有对应的落地机制。

### 4.1 原则一：代理永远不主动

**落地方式**：
- 代理面板默认折叠，不存在。用户必须主动点击底部"助手"按钮才能召唤
- 任何系统层面的提醒（草稿、操作记录）统一走**静态按钮入口**（"我的"页面中的"草稿箱"和"操作记录"），用户看得到、够得着、想用才点
- 草稿箱恢复提示唯一的例外：代理被唤醒后，**在用户先开口的前提下**，在问候语之后顺带一提。不违反"永远不主动"——这是被唤醒后的正常应答，不是主动打扰

**派生纪律**：所有"代理应该主动告诉用户什么"的设计提议，默认答案是"不"——改用静态入口 + 用户点击查看。

### 4.2 原则二："确定性操作"代理止步

**定义**：确定性操作 = 一旦按下就会对外产生真实结果、不可轻易撤销的动作（登录、提交、去支付、同意条款、删除等）。

**落地方式**：
- 确定性按钮在代码层面**不注册为代理工具**——代理物理上无法调用，不依赖提示词约束
- 只有非确定性操作（"查询"、"下一步"等无对外后果的按钮）注册为 `cmd_press_button` 工具，且前端维护安全白名单校验
- 跨越所有 L 等级——哪怕是用户明确授权的 L3 场景，代理也不能按确定性按钮

**设计价值**：这条原则是老年用户"不放心委托代理"的最直接回应——用户知道，无论代理获得什么授权，最终那一下都是自己按的。

### 4.3 原则三：权限一事一授

**落地方式**：
- 代理没有持久化的"全局权限等级"，用户永远不会看到抽象的权限等级界面
- 每次需要升级能力时（如代填身份证号、代读短信），代理**当场弹出授权卡片**，用口语化语言说明要做什么（"这次我帮您填身份证号可以吗？"），用户用语音或点击当场决定
- 授权**仅对当前任务生效**，会话结束后自动失效，下次同类任务重新询问
- 超时 20 秒无响应，自动视为拒绝，代理跳过该步骤

**设计价值**：通过一次次"您说可以，我才做"的交互积累，用户对代理的信任自然增长——不是系统单方面宣称"我是安全的"，而是用户在使用过程中亲历了代理始终守住边界。

## 五、多模态交互设计

### 5.1 多模态协同策略：冗余互补

本设计的核心多模态策略是**冗余互补**：同一信息通过多个通道同时传递，确保任一感官退化不影响信息接收，同时让用户选择最自然的通道。

| 通道 | 作用 | 设计细节 |
|---|---|---|
| **语音输入（ASR）** | 用户表达需求 | Web Speech API（浏览器原生）；按住麦克风说话，松开发送 |
| **语音播报（TTS）** | 代理反馈、引导、播报结果 | Edge TTS；语速慢约 15%；数字类信息重复一遍（"4800元，四千八百元"） |
| **视觉高亮** | 引导用户目光，告诉他"点这里" | 橙色描边脉冲动画，最多同时高亮 1 个元素；对应蒙层挖空，高亮元素透出可点击 |
| **代填操作** | 代替用户输入表单内容 | 字段逐位填充动画，让用户"看着字自己进去"而非突然变化 |
| **触控保留** | 用户任何时候可以自己操作 | 面板打开时上方业务页面虽变暗，但在代理发出高亮指令时挖空对应位置，用户可直接点击 |

### 5.2 语音通道的适老化设计

- **语速**：TTS 输出为正常语速的 85%（慢约 15%），适合老年用户理解
- **用词**：避免技术术语，口语化表达（"帮您查一下"而非"执行查询操作"）
- **关键数字重复**：金额、日期等关键信息重复一遍，给用户二次确认机会
- **等待反馈**：ASR 识别期间，状态行显示"正在听您说话…"；LLM 推理期间，显示"小浙正在想…"配三点跳动动画，防止老年用户误判为死机
- **失败恢复**：听不清时说"不好意思没听清，您再说一遍"，最多重试 3 次后提示可切换手动操作

### 5.3 视觉通道的适老化规范

- 主色调：橙色 `#FF6D00`，对比度强，长辈版整体沿用
- 最小字号：正文 18sp，代理气泡 20sp，授权卡片权限名称 24sp 加粗
- 最小触控区：48dp（底部 Tab）、88dp（助手按钮 + 麦克风大按钮）
- 按钮间距：≥ 16dp，防止老年人误触相邻按钮
- 高亮动画周期：1000ms，不过快也不过慢，符合老年用户视觉追踪速度

## 六、代理 UI 形态

### 6.1 面板结构与空间关系

代理面板从屏幕底部滑上，占屏幕高度 55%，上方业务页面始终可见（变暗 40%）。这一设计实现了两个重要目的：

1. **上下文不丢失**：老年用户在与代理对话时，仍能看到当前业务场景，不会因切换到全屏助手而"忘了自己在做什么"
2. **执行时透明可见**：代理发出高亮或代填指令时，效果直接在上方业务区域体现，用户能实时看到代理"动了哪里"

```
┌──────────────────────────────┐
│    上方业务页面（变暗 40%）    │  ← 45%，可见，高亮挖空后可交互
├──────────────────────────────┤ ← 拖拽条
│  [浙] 小浙              ×    │  ← 48dp 标题栏
├──────────────────────────────┤
│    对话气泡区（可滚动）        │  ← 小浙气泡左对齐，用户气泡右对齐
├──────────────────────────────┤
│   [状态行] 正在听您说话… 🎙️   │
├──────────────────────────────┤
│         ●  按住说话          │  ← 72dp 麦克风大按钮，橙色
└──────────────────────────────┘
```

### 6.2 代理状态流转与 UI 对应

| 代理状态 | 触发事件 | 可见 UI 变化 |
|---|---|---|
| **沉默**（默认） | — | 面板不存在；底部 Tab 中央仅有"助手"按钮 |
| **唤醒** | 用户点击"助手"按钮 | 面板从底部滑入（300ms）；上方页面渐暗；TTS 播报问候语 |
| **聆听** | 用户按住麦克风 | 状态行变橙"正在听您说话…"；麦克风按钮外圈出现声波扩散动画 |
| **思考** | ASR 结果返回，等待 LLM | 状态行灰色"小浙正在想…"；气泡区出现三点跳动；麦克风变灰 |
| **执行** | 用户确认意图 | 业务页面跳转 / 字段填充动画 / 高亮脉冲依次出现；代理气泡同步展示操作文字 |
| **等待授权** | 遇到敏感操作 | 面板内弹出授权卡片（"小浙想帮您填【身份证号】"）；TTS 播报授权请求 |
| **完成** | `task_done` | TTS 播报完成语；1.5 秒后面板自动关闭；写操作记录 |
| **网络断开** | WebSocket 断连 | 面板顶部出现红色横幅"网络已断开"；麦克风变灰不可用 |
| **超出能力** | `agent_out_of_scope` | 代理气泡显示应对话术（有无服务两种子情况） |

### 6.3 授权卡片交互

授权卡片是"权限一事一授"的前台界面，设计重点：
- **不脱离面板**：卡片从面板底部弹出，不打断上方业务页面
- **明确标注要碰什么**：权限名称 24sp 加粗 + 橙色，"【身份证号】"清晰可见
- **两个大按钮**：" 可以 "（橙色填充，主操作）和"不用了"（白底灰边，次操作），触控区各 ≥ 120×56dp，按钮间距 ≥ 16dp
- **超时自动拒绝**：20 秒无响应视为拒绝，代理语音说"好的，这一步就不帮您填了"，流程继续

## 七、代理专属页面与独立设置分析

### 7.1 问题背景

当前小浙没有独立主页——它以底部面板的形式叠加在业务页面之上。这引出一个产品问题：**代理是否需要一个专属的独立页面，以及配套的独立设置功能？**

这不是一个单纯的 UI 选题，它触及对代理本质的理解：小浙到底是一个需要"去找"的功能区，还是一个随时陪伴在场景里的助手？

---

### 7.2 代理专属页面的必要性分析

**如果做专属主页，可以放什么？**

| 内容模块 | 说明 | 是否值得做 |
|---|---|---|
| 助手介绍 / 能力说明 | 图文说明小浙能做什么（4 个场景）、怎么唤醒 | 值得做，但可以用更轻量的方式（见 7.4） |
| 历史对话记录 | 跨 session 的对话文本 | **不做**——当前架构会话状态存于内存、页面关闭即丢弃，这是刻意设计（防止授权状态跨 session 残留），重新持久化引入不必要复杂度 |
| 操作记录入口 | 小浙帮用户做过什么 | 已有，在"我的"页面；不需要专属主页，设置入口即可 |
| 草稿箱入口 | 未完成的表单 | 同上 |
| 设置（语音、语速等） | 见 7.3 节 | 值得做，但不需要独立主页，设置子页即可 |
| 从主页发起对话 | 直接在主页开始和小浙说话 | **存在根本问题**：从主页发起任务，执行时需要跳转到业务页面，用户体验上会经历"离开主页 → 进入业务页面"，上下文断裂 |

**核心矛盾**：专属主页对代理最有价值的功能（发起对话、引导操作）来说，本身就是一个位置错误的起点。代理的价值发生在业务页面上，而不在代理的"家里"。

---

### 7.3 独立设置功能分析

设置的需求是合理的，但不需要依附于专属主页，**放在"我的"页面的子入口即可**。

**建议纳入的设置项：**

| 设置项 | 优先级 | 说明 | 适老化设计要求 |
|---|---|---|---|
| 语音引导开关 | P0 | 全局开关，关闭后代理仅文字回复，不播放 TTS；**实施前提**：需新建 `AgentSettingsService` 单例，`_speakHint()` 中增加开关判断 | 大型切换开关，附带"已开启 / 已关闭"文字说明，不只靠颜色区分 |
| 语速选择 | P0 | 两档：慢 / 标准，默认"慢"；**实施前提**：`_speakHint()` 当前硬编码 `rate=0.9`，需改为读取 `AgentSettingsService` 的配置值 | 用大按钮而非滑块；避免让老年用户在连续刻度上选择 |
| 使用说明 / 帮助 | P1 | 静态图文，分场景卡片（登录 / 缴费 / 查询） | 每张卡配一句话说明，不用长段文字 |
| 操作记录快捷入口 | P1 | 复用已有 `operation_logs_page`，直接链接，不另起页面 | — |
| 草稿箱快捷入口 | P1 | 复用已有草稿箱，直接链接 | — |
| 唤醒词开关 | P2 | 热词"小浙"唤醒的开 / 关（当前默认关闭） | 留待热词功能正式启用后再暴露此设置 |

> **代码依赖说明**：语音引导开关和语速选择两项功能落地前，需先由 frontend 新建 `AgentSettingsService` 并修改 `_speakHint()` 的调用方式。UI 设置项本身可以先做，但在 Service 接入完成前按钮无实际效果。

**不建议纳入的设置项：**

| 设置项 | 原因 |
|---|---|
| 历史对话记录 | 架构设计冲突（见 7.2）；若要做需重新评估 session 持久化方案 |
| 隐私授权管理（预设哪些字段允许代填） | 与"权限一事一授"原则冲突——预设授权等于用批量授权替代每次当场同意，绕过了信任建立机制的核心 |
| 代理形象 / 语音包选择 | 毕设范围外；老年用户不需要面对不必要的选择 |

---

### 7.4 专属页面 vs 悬浮面板：优劣对比

| 评估维度 | 悬浮面板（当前方案）| 专属主页 |
|---|---|---|
| **上下文连贯性** | ✅ 面板弹出时业务页面仍可见，高亮 / 代填就在眼前发生 | ❌ 从主页跳转到业务页面，用户经历两次页面切换 |
| **进入成本** | ✅ 随时召唤，无需额外导航 | ❌ 需先切换到助手 Tab，再发起任务 |
| **认知负担（老年用户）** | ✅ 代理只有一种状态，用户位置感清晰 | ❌ 需区分"在助手主页里"和"在业务页面里"两种状态 |
| **发现性（首次使用）** | ❌ 不直观，用户需自行发现底部"助手"按钮的用途 | ✅ 专属页面可展示引导说明，降低初次理解成本 |
| **信息聚合** | ❌ 操作记录 / 草稿箱散落在"我的"页面 | ✅ 所有代理相关内容集中展示 |
| **与"响应型"定位契合** | ✅ 代理无"地盘"，只在被需要时出现 | ❌ 暗示代理是需要"去找"的目的地，与定位相悖 |
| **导航层级（老年用户）** | ✅ 不增加层级 | ❌ 新增一个 Tab，老年用户要记住的入口增加 |

---

### 7.5 适老化视角的推荐方案

**推荐：不做独立主页 Tab，改用"两者兼有"的轻量组合方案。**

"两者兼有"不意味着两套完整系统，而是：

- **面板做实时交互**（保持现有设计不变）——代理在场景里响应，高亮 / 代填 / 引导发生在用户正在看的页面上
- **"我的"页面做静态管理**——新增"小浙助手"入口（设置 + 使用说明 + 操作记录 + 草稿箱），让用户有地方"了解代理、管理代理"，而不是"使用代理"

这个方案的逻辑是：**实时交互在业务发生的地方，管理和了解在一个集中的地方**——两者角色不重叠，也不产生"代理住在哪里"的认知歧义。

```
长辈版底导 →  "助手"按钮 → 弹出面板（实时交互）
"我的"页面 → "小浙助手" → 设置子页（管理 + 了解 + 查看记录）
```

**发现性问题的解决**（专属主页的唯一合理诉求）：
- 首次打开 APP，底部"助手"按钮旁增加一次性提示气泡（"点这里唤醒小浙"），自动消失，不再重复
- "我的 → 小浙助手"的使用说明以分场景卡片形式呈现，让用户在非操作时可以了解代理能力

**这样做的适老化理由**：
- 老年用户的记忆负担是"再多记一个入口"，面板按钮 + 管理入口各司其职，不引入第三套心智模型
- 管理功能（语音开关、语速）放在"我的"页面，与手机设置习惯一致——老年人已经学会了"在'我的'里改设置"这个模式

### 8.3 已识别、建议优先补充的 3 项改进

**改进 1：场景 1 语音提示时机前移**（N5 Prompt 调优）

当前剧本中，"马上会弹出摄像头请求"的语音提示，可能在弹窗已经出现后才到达用户耳朵，反而让用户先慌到。建议在跳转到刷脸页后，代理**立即先说解释**，再等弹窗出现。

**改进 2：身份证号脱敏处理明确在前端**（N4 真机测试验证）

身份证号脱敏显示（330…\*\*…6）应由前端在渲染时处理，后端不通过 WebSocket 明文传输敏感值。需确认 `agent_command_executor.dart` 中对 `is_sensitive: true` 字段的渲染逻辑。

> **安全声明（答辩备注）**：当前 Demo 环境 WebSocket 使用明文传输（ws://），敏感字段值在传输链路上未加密。真实部署场景须使用 WSS（WebSocket over TLS）并对敏感字段值进行加密或令牌化处理。本原型仅用于毕设功能演示，不具备生产级安全标准。

**改进 3：面板关闭延迟计时起点需修复**（**N4 前必须完成，非仅验证**）

场景 4 的面板 2 秒延迟关闭，应在 TTS 音频开始播放、屏幕结果卡片渲染完成后才开始计时，否则用户可能还没听完就看到面板消失。

> **根本原因**：当前 `AudioPlayer` 未监听 `onEnded` 回调，延迟计时从"消息到达"而非"播放完成"开始，属于代码缺陷而非环境差异。此问题在真机上必然复现，需在 N4 真机测试前完成修复。

### 8.4 N4 真机测试须覆盖的额外风险

| 风险 | 描述 | 影响 |
|---|---|---|
| `cmd_press_button` 兼容性 | 当前实现通过 `GestureDetector` 查找触发点击，若目标按钮为 `ElevatedButton` 或 `InkWell` 则静默失败（无报错、无响应） | 场景 4 查询按钮代按功能可能在部分 widget 上失效；N4 须用真实养老金查询页面验证 |

---

### 8.5 明确不做（毕设范围）

| 内容 | 原因 |
|---|---|
| 支付环节（支付宝 / 微信 / 输密码） | 超出代理能力范围，涉及第三方支付安全 |
| 生物识别代做（替用户刷脸 / 按指纹） | 肉身动作，技术和原则上均不可代做 |
| 4 场景之外的政务服务 | 毕设范围聚焦，超出范围时代理坦诚说明 |
| 方言识别 | 技术复杂度高，留待未来 |
| 热词唤醒（"小浙小浙"免按钮） | 功能已实现基础，但产品层面当前版本默认关闭 |
| 生产级 WS 加密 | Demo 原型，真实部署须 WSS + 字段加密（见 8.3 安全声明） |
