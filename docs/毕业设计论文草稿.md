## 1 设计背景

### 1.1 绪论

#### 1.1.1 设计的发展趋势

信息服务 APP 适老化设计方面，国外研究起步较早，形成了较系统的理论体系。在感官退化与界面设计层面，Farage 等经典综述系统梳理了老年人在视觉、听觉、触觉、运动和记忆方面的功能变化，并提出具体设计指南，如提高对比度、控制声音频率范围、提供多通道反馈、简化任务流程等。Amouzadeh 等系统综述了适老化移动 APP 设计，梳理了字体、色彩对比、触控区域、导航层级等视觉与交互规范，确认老年用户在认知负荷、视觉感知和运动精度三个维度上存在显著的特殊需求。

在多模态与 AI 适配层面，Bhattacharya 等的叙事性综述提供了关键数据支撑：相比单一模态系统，78% 的老年人更倾向于支持多种输入方式的界面；AI 驱动的预测模型在自适应系统中预测用户偏好的准确率可达 95%；针对认知老化的个性化多模态界面能将老年人可用性提升 30%。

国内方面，张晓宇等构建了包含系统易用性、系统包容性、情感支持度、服务有效度四个维度的适老化设计评价指标体系，指出当前适老化改造多停留在放大字体、简化界面等表层优化，而交互逻辑、容错机制、情感化设计仍有不足。李梦云对移动政务 APP 适老化使用的影响因素进行了实证研究，发现感知有用性和感知易用性显著影响老年人的持续使用意愿，而现有适老化改造对"感知安全性"（即老年人对操作失误风险的担忧）关注不足。侯文军则指出适老化改造并非简单做"加减法"，字体放大、界面简化是必要条件而非充分条件，深层问题在于服务流程的重构与交互逻辑的适老化。

多模态适老化交互方面，Wang 等国际人机交互领域综述分析了 31 项关于老年人智能交互界面的研究，将交互形式分为手势交互、语音交互、多模态交互和人工智能交互四类。研究发现，语音交互因其能够绕开视觉与操作障碍，被认为是适老化设计的重要突破口，但目前许多产品只是形式上接入语音模块，却未形成真正可用的交互闭环——语音输入缺乏视觉反馈确认、视觉引导缺少语音解说、触控操作无法获得多通道响应，模态间的割裂导致老年人需在不同交互方式间频繁切换，反而增加了认知负担。

王斯薇子等提出以交互情境、模态信息、用户意图三要素为导向的闭环设计模型，通过建立多模态信息间的动态映射关系实现场景与用户状态的智能适配。该研究特别强调，多模态协同的关键不在于通道数量的叠加，而在于通道间的语义融合与时序对齐——同一信息须在多个通道中以相互强化而非相互干扰的方式传递。Liu 等的 Mediguide 系统通过多模态引导显著降低了老年患者在医疗场景中的认知负荷，验证了多模态设计在降低操作门槛方面的实际效果。

智能代理研究方面，Wooldridge 等早期定义了智能代理的四要素：自主性（autonomy）、社会性（social ability）、反应性（reactivity）和主动性（pro-activeness）。Jaiswal 等对近年代理式人工智能（Agentic AI）的综述梳理了从规则型代理到大语言模型驱动代理的演进脉络，指出当前代理研究以"最大化自主完成任务"为核心目标，在老年用户这一特殊群体的适老化应用层面研究较为薄弱。

Lee 等提出的任务导向型多模态代理系统（TOMAS）通过简化用户界面和整合语音命令，帮助老年人更独立地使用智能手机，但该研究中代理的边界设计较为模糊，对"代理可以替用户做什么"缺乏明确约束。CareBuddy 等多智能体协同系统在老年照护领域展现出良好的可用性，但其应用场景为被动监护，与信息服务 APP 的主动操作场景存在本质差异。Cuadra 等开发的多模态大语言模型代理系统用于健康数据录入，揭示了多模态交互如何以互补方式为老年用户提供替代路径，特别指出代理在处理敏感健康数据时需要明确的用户授权机制。张诗棋等通过用户访谈深入了解了老年人对代理型适老软件的需求和体验，发现老年用户在使用智能代理时对"代理帮我做了什么、我是否还在掌控"存在明确的知情需求，这一洞察对代理透明性设计具有重要参考价值。

综合以上三个领域的研究现状，当前主要存在三方面不足：

第一，表层改造与深层需求脱节。 现有适老化改造多局限于放大字体、简化首页等视觉层面调整，对交互逻辑、容错机制和情感化设计缺乏深入考量，未能有效降低老年人的认知负荷与操作门槛。

第二，信任与控制的双重缺失。 传统智能代理强调自主性和主动性，其设计逻辑是代理替用户做决定、替用户执行。然而老年人对智能代理的失控存在深层焦虑——担心代理理解错误、操作错误、泄露信息，却又无法干预。这种"不信任系统"叠加"不放心代理"的双重心理，导致老年人既不愿独自操作，也不敢委托代理，最终形成对线上服务的抵触和回避。

第三，多模态交互的适老化应用尚不成熟。 在适老化场景中，语音、触控、视觉等模态未能形成有效协同——语音输入缺乏视觉反馈确认、视觉引导缺少语音解说、触控操作无法获得多通道响应。模态间的割裂导致老年人需在不同交互方式间频繁切换，反而增加了认知负担。

针对上述三方面不足，本研究提出受控响应型智能代理这一设计框架，以"分级授权能力矩阵 + 三原则架构固化"为核心创新点，探索从"自助服务"到"协助服务"的范式转变路径，并通过多模态冗余互补策略解决模态协同问题。

#### 1.1.2 研究背景

我国已步入中度老龄化社会。据民政部《2024 年民政事业发展统计公报》，截至 2024 年底，60 岁及以上老年人口达 3.10 亿，占总人口比例 22.0%，预计到 2035 年这一比例将突破 30%，进入重度老龄化阶段。与此同时，移动互联网的普及使各类信息服务 APP（政务、医疗、健康、生活服务等）成为公众日常生活不可或缺的工具。然而，老年群体在使用这些 APP 时面临严峻的"数字鸿沟"困境——据中国互联网络信息中心（CNNIC）第 55 次《中国互联网络发展状况统计报告》，60 岁以上网民中仅约四成能独立完成在线政务服务操作，相当比例的老年人因操作困难而放弃线上渠道，转而依赖线下窗口服务。

政务类 APP 在这一困境中尤为突出。相比购物、社交等消费类应用，政务服务涉及身份认证、资金缴纳、证件申请等高风险操作，一旦误操作可能产生难以撤销的后果。这种"操作失误代价高"的特性，使老年用户在政务 APP 上的焦虑感远高于其他类型应用。浙江省政务服务 APP"浙里办"已覆盖医保缴费、养老金查询、公积金提取等数百项高频政务服务，是老年人日常生活中需求最迫切、使用障碍也最集中的应用场景之一。

数字鸿沟的成因是多层次的。在操作层面，老年人在使用政务 APP 时普遍遭遇字体过小难以辨认、触控区域过小导致误触、多步骤流程中途迷失、验证码切换 APP 后页面失效等具体障碍。在认知层面，面对频繁更新的界面版本和不一致的交互逻辑，老年人的学习迁移成本极高，每次版本更新都可能让已掌握的操作路径失效。在心理层面，对操作失误的恐惧和对个人信息泄露的担忧，形成了比操作障碍更难突破的心理壁垒——这也是为什么即便学会了操作，许多老年人遇到"要紧事"仍会选择去线下窗口办理。

本研究的核心洞察来自于实地调研：老年用户并非不愿线上办事，而是不想在线上自己冒险办事。相比独自操作，他们更倾向于去线下服务大厅寻求工作人员帮助。老年用户最渴望的不是更清晰的界面，而是可信赖的帮助。这种"信任人胜过信任系统"的心理，是阻碍老年用户跨越数字鸿沟的深层障碍，要求信息服务从传统的自助服务模式向协助服务模式转变。近年来智能代理技术的成熟恰好为这一转变提供了技术支撑，如上一节文献综述所示，现有适老化研究在表层改造、信任机制和多模态协同三个方面均存在明显不足，这正是本研究的切入点。

#### 1.1.3 研究目的

本研究以浙江省政务服务 APP"浙里办"为应用场景，面向 60—75 岁老年用户群体，以解决老年人使用信息服务 APP 的"数字鸿沟"问题为核心目标，具体研究目的包括以下三个层面：

其一，提出受控响应型智能代理的设计框架。 针对现有智能代理强自主性设计导致老年用户"失控感"焦虑的问题，本研究提出并构建"受控响应型智能代理"这一交互模式，明确代理在权限受控、行为受控、响应型三个维度的设计边界，形成可操作的分级授权能力矩阵（L1/L2/L3），为适老化智能代理的设计提供理论框架。

其二，探索多模态协同在适老化场景中的有效整合路径。 针对现有研究中语音、触控、视觉等模态在适老化场景中未能形成有效协同的不足，本研究通过"冗余互补"策略设计三模态协同机制，在实际系统中验证多模态协同对老年用户交互体验的提升效果。

其三，实现可运行的适老化高保真原型。 将设计框架落地为可在手机浏览器运行的高保真原型，覆盖账号登录、医保缴费、医保查询、养老金查询 4 个政务服务核心场景，通过真实可用的系统验证受控响应型代理设计的技术可行性与实用价值。

#### 1.1.4 研究流程与方法

本研究遵循"问题发现 → 文献研究 → 设计实践 → 原型实现 → 可用性验证"的整体研究路径，共分五个阶段：

第一阶段：实地调研与问题发现（2025 年 10—11 月）
深入杭州市西湖区三个典型老年社区及政务大厅，对 7 位 60 岁以上老年用户进行半结构式访谈与参与式观察，提炼核心痛点与心理洞察，确立"信任人胜过信任系统"的研究核心命题。

第二阶段：文献研究与理论建构（2025 年 11—12 月）
系统梳理适老化设计、多模态交互、智能代理三个领域的国内外研究，识别三方面核心不足：现有适老化设计研究缺乏智能代理介入机制；现有智能代理设计以"最大化自主"为目标，难以适配老年用户对控制感的需求；多模态交互在适老化场景中的协同机制尚不成熟。在此基础上，提出受控响应型智能代理这一设计框架，归纳三条横切全局的设计原则：其一，代理永远不主动——在用户主动唤醒之前完全静默，不弹窗、不推送；其二，确定性操作代理止步——凡按下即产生不可撤销后果的操作，代理只引导用户亲手完成，不代为执行；其三，权限一事一授——不设持久化全局授权，每个敏感操作在当次任务中单独向用户请求授权。这三条原则在后续设计与开发阶段被固化为架构约束而非运行时约定。

第三阶段：设计方案构思与迭代（2026 年 1—3 月）
基于用户研究洞察，完成用户画像、用户旅程图、信息架构、低保真原型设计，通过快速走查对方案进行迭代修正，最终完成高保真交互与视觉设计方案。

第四阶段：原型开发与实现（2026 年 3—5 月）
以 Flutter Web + FastAPI + Agno Agent 技术栈实现高保真可交互原型，完成 17 个页面、4 个核心场景的全功能实现，接入 DeepSeek-V3 语言模型、讯飞 ASR 语音识别和讯飞 TTS / Edge TTS 语音合成。

第五阶段：可用性测试与总结（2026 年 5—6 月）
招募 5—8 位老年用户进行真机可用性测试，采集任务完成率、SUS 量表、信任感评分等指标，验证设计方案的有效性并进行最终总结。

在研究方法上，本研究综合运用以下四类方法：

（1）质性用户研究

采用半结构式访谈与参与式观察相结合的质性研究方法，对老年用户进行深度访谈，同时设置任务测试环节，以"出声思考"（Think Aloud）协议记录真实操作行为与情绪反应。访谈数据通过主题编码（Thematic Coding）方法进行分析，归纳出导航迷失、认知障碍、情绪冲突三个层面的痛点主题，以及"信任人胜过信任系统"这一核心心理洞察。

（2）以用户为中心的设计（UCD）

遵循以用户为中心的设计方法论，以用户调研洞察驱动每一个设计决策，构建用户画像、用户旅程图等设计工具，通过低保真原型快速走查、高保真原型迭代的方式持续验证设计方案。每个核心场景的交互设计均可追溯至具体用户痛点，设计方案的每一处权衡均有明确的用户研究依据。

（3）原型法（Prototyping）

采用渐进式原型法：纸质草图 → 灰度线框图 → 高保真可交互原型，逐步提升原型的保真度与功能完整性。最终高保真原型基于 Flutter Web 实现，具备完整的交互逻辑、视觉细节和后端功能，支持在真实手机设备上运行和测试，区别于单纯的静态原型工具（如 Figma）。

（4）可用性测试

采用任务式可用性测试方法，设计覆盖 4 个核心场景的测试任务（T1—T4），量化评估任务完成率、操作错误率、任务完成时间，辅以 SUS（System Usability Scale）量表和自定义信任感李克特量表，对设计方案的可用性与用户信任感进行综合评估。

### 1.2 选题意义

#### 1.2.1 理论意义

本研究是对现有适老化设计研究的重要补充。已有研究从影响因素识别、评价体系构建、多模态交互融合以及智能代理应用等维度进行了有益探索，但对智能代理的自主性与老年用户的控制需求之间的矛盾缺乏系统回应。本研究提出受控响应型智能代理这一交互模式，在保留智能代理核心能力的同时，通过分级授权机制让用户始终保持最终控制权，探索从自助服务到协助服务的范式转变。这一设计定位回应了老年人担忧智能代理权限失控的深层心理需求，为人机交互领域的适老化研究提供了新的理论视角。

该框架的核心理论贡献在于三条横切全局的设计原则：其一，**代理永远不主动**——代理在用户主动唤醒之前完全静默，不弹窗、不推送、不预判，将发起对话的权利始终留给用户；其二，**确定性操作代理止步**——凡按下即产生不可撤销后果的操作（如登录、提交、支付），代理只负责引导和告知，不代为执行，保障用户对"最后一步"的绝对掌控；其三，**权限一事一授**——代理不持有持久化全局授权，每个涉及用户隐私或账户的敏感操作，均在当次任务中单独向用户请求授权、任务结束即失效。这三条原则的理论意义在于：它们将"信任保障"从对话层面的口头约定提升为架构层面的技术约束，使代理无法越权成为可验证的工程性质，而非依赖运行时判断——这对消解老年用户对智能代理的深层失控焦虑具有根本性意义。

在多模态协同方面，本研究提出的冗余互补策略——语音、触控、视觉三模态同步传递同一信息——为适老化场景下的多模态设计提供了可操作的方法论框架，补充了现有研究中模态协同机制不足的空白。

#### 1.2.2 实践意义

本研究直接回应了老年用户"信任人胜过信任系统"的核心心理，通过智能代理技术将线下"工作人员帮助"的协助模式复现到线上，有望从根本上降低老年用户的数字焦虑，推动其从依赖线下窗口向主动使用线上服务转变。

研究以浙江省政务服务 APP"浙里办"为载体，覆盖医保缴费、养老金查询等高频政务场景，设计成果可直接为同类政务 APP 的适老化迭代提供可操作的设计指南。受控响应型代理的分级授权能力矩阵（L1/L2/L3）和三原则架构，具有向医疗、金融、生活服务等其他信息服务 APP 场景推广的潜力。

#### 1.2.3 社会意义

我国老龄化进程持续加速，数字鸿沟问题已成为影响老年群体生活质量和社会参与的重要障碍。本研究所探索的协助服务模式，有助于让更多老年人平等享受数字化公共服务，契合国家推进数字包容、消除数字鸿沟的政策导向。同时，本研究通过可运行的高保真原型验证设计方案，为后续信息服务 APP 适老化标准的完善提供了实证依据。

## 2 方案构思

### 2.1 设计创新点及设计源点

#### 2.1.1 核心创新：受控响应型智能代理

本研究的核心创新是提出并实现受控响应型智能代理这一交互模式。

相较于当前信息服务 APP 中常见的 AI 助手，本研究所提出的智能代理在能力上有本质区别。现有 AI 助手多停留在对话式咨询层面，给出相关的流程指导，但最终仍需用户自己去实际操作相关流程。而本研究的智能代理则具备直接的任务执行能力，可以代为完成页面跳转、表单填写等步骤繁琐但规律性强的操作，从而切实减轻老年用户的认知与操作负担。

相较于通用智能代理追求最大化自主完成任务的技术路线，本研究回应老年用户对智能代理信任缺失与控制权的双重顾虑，将智能代理定位为"受控响应型智能代理"：代理仅在获得用户明确授权后协助操作，而非主动替用户决策和执行。这一设计定位包含三层含义：一是响应性，代理根据用户指令启动，不主动打扰；二是受控性，代理的所有操作需经用户授权并保持透明，用户可随时中止；三是可委托性，用户可根据信任程度选择不同级别的权限，具体分为三级：

- **L1 引导级**：代理仅提供步骤提示与语音引导，不实际操作页面；
- **L2 半委托级**：代理代为填写表单中的普通字段，但不提交，最终确认由用户完成；对于敏感字段（如身份证号、短信验证码）需单独逐次授权；
- **L3 全委托级**：代理完成完整流程，但不可逆操作（登录、提交、支付）和密码类凭证字段（如登录密码、支付密码）的填写仍须用户亲手完成——这是本设计的硬性底线，不因委托级别而改变。

需说明的是，上述"敏感信息"特指具有隐私价值的个人标识字段（如身份证号、手机号等），与密码类凭证字段（如登录密码、支付密码）在性质上有本质区别。后者属于账户安全凭证，系统在架构层面即予以屏蔽——代理不读取、不存储、不传输任何密码字段，此为不因委托级别或用户授权而改变的系统级硬性约束。

这一设计定位与传统智能代理的"强自主性"范式形成对比：传统代理追求最大化自主完成任务；受控响应型代理追求最大化用户可信任感与控制感，将"代理能做什么"的边界设计得比技术上限更保守、更透明。

[图：受控响应型代理与传统代理的设计定位对比图（待制作）]

#### 2.1.2 设计源点：线下窗口服务的数字复刻

设计源点来自老年人线下求助行为的观察。在政务大厅，当老年人遇到不会操作的环节时，窗口工作人员的协助方式是：

1. 先听清楚老人要办什么（意图理解）
2. 复述确认"您是要办医保缴费对吧？"（复述确认）
3. 引导老人跟着步骤做，必要时帮忙填写（按需协助）
4. 在提交和支付的关键节点，仍然让老人自己确认和操作（止步于确定性动作）
5. 操作完成后告知结果（结果反馈）

本研究将这一线下服务模式原样映射到线上，通过代理"小浙"实现数字化复刻。代理的交互逻辑完全按照"有耐心、有礼貌、会引导、懂边界"的线下工作人员标准设计。

### 2.2 设计概念草案

#### 2.2.1 产品定位

以浙江省政务服务 APP"浙里办"为应用场景，面向老年用户，提供集成受控响应型智能代理"小浙"的适老化长辈版。通过语音对话为主的多模态交互方式，将线下"向窗口工作人员求助"的体验复现到线上，帮助老年人跨越数字鸿沟。

产品愿景：成为老年用户在政务 APP 中可信赖的线上工作人员——用户只需说出需求，代理就能引导、协助完成操作，同时用户始终保持最终控制权。

#### 2.2.2 代理形象概念

代理命名为"小浙"，取浙里办的"浙"字，名字直白易记，贴近政务服务语境。形象为卡通化"浙"字小人图标：简洁不复杂，橙色圆形底，白色字形小人，既有亲和感又有辨识度，不走拟人化路线以避免老年用户对真实性产生困惑。

[图：小浙形象设计稿（待制作）]

#### 2.2.3 核心场景框定

研究聚焦 4 个老年人高频使用、痛点突出的核心政务服务场景：

| 场景 | 代理能力 | 设计价值 |
|---|---|---|
| 账号登录·刷脸分支 | L1（纯语音引导） | 展示代理对系统弹窗"不介入"的边界 |
| 账号登录·验证码分支 | L2（代读短信、代填验证码） | 展示跨应用操作协助与确定性止步 |
| 医保缴费 | L2-L3（代填表单、敏感字段单独授权） | 展示权限一事一授的完整流程 |
| 医保/养老金查询 | L1-L2 全链路 | 展示查询类无确定性操作的全自动路径 |

4 个场景合起来，系统地演示了受控响应型代理在不同能力等级和不同授权情境下的行为边界。

#### 2.2.4 设计概念关键词

协助而非替代：代理是助手，用户始终是主角；透明而非黑箱：每步操作前都有语音告知，操作后有记录可查；渐进而非全权：信任通过一次次授权自然累积，而非一开始就全盘委托；冗余而非单一：同一信息多通道传递，确保老年人无论习惯哪种方式都能接收。

## 3 初步设计

### 3.1 用户需求

#### 3.1.1 用户调研方法

本研究采用结合参与式观察与半结构式访谈的质性研究方法。

调研场域：杭州市西湖区港湾家园、三坝雅苑、古荡新村三个典型老年社区，以及古荡菜市场（高频数字支付场景）与西湖区政务大厅（政务服务高频场景）。三个社区分别代表不同的老年居住类型：港湾家园为紧邻高校的居住型社区，三坝雅苑为回迁安置小区，古荡新村为建设于 20 世纪 80 年代的老牌居民区，居住着大量本地老年居民。政务大厅作为老年用户办理社保查询、医保报销、养老资格认证等服务的核心场所，每日聚集大量老年群体，能够在真实的政务服务语境下观察老年人的操作行为与情绪反应。

调研样本：7 位 60 岁左右老年用户，通过社区渠道目的性抽样招募，覆盖不同年龄段（60—75 岁）、文化程度（初中至大专）、智能手机使用年限（2—8 年）与居住状况（独居/与子女同住）。

| 编号 | 性别 | 年龄 | 文化程度 | 手机年限 | 核心特征 |
|---|---|---|---|---|---|
| U1 | 女 | 68 | 高中 | 5 年 | 微信熟练，每年医保缴费靠女儿帮忙 |
| U2 | 男 | 72 | 初中 | 3 年 | 验证码环节困难，常去线下政务厅 |
| U3 | 女 | 65 | 大专 | 7 年 | 会用美团，对政务 APP 持怀疑态度 |
| U4 | 男 | 70 | 初中 | 2 年 | 不会切换 APP，曾因操作失误损失过 |
| U5 | 女 | 63 | 高中 | 6 年 | 独居，数字素养中等，对语音助手感兴趣 |
| U6 | 男 | 74 | 初中 | 2 年 | 人脸识别多次失败，放弃线上办事 |
| U7 | 女 | 66 | 高中 | 4 年 | 与子女同住，每次操作需子女在场陪同 |

访谈设计：访谈在受访者熟悉的居住社区进行，单次时长约 30 分钟，全部录音并整理转写为文本。访谈围绕三条主线展开：日常数字生活习惯；对信息服务 APP 的认知与使用体验；操作过程中的主要障碍与需求。访谈同时设置任务测试环节，要求受访者在无辅助状态下完成查找养老金发放记录、查询医保缴费记录两个典型任务，研究者以"出声思考"协议记录操作过程与情绪反应。

数据分析方法：对录音转写文本采用主题编码（thematic coding）方法分析。首先进行开放式编码，从原始访谈文本中归纳初始代码；随后进行聚焦编码，将相关代码合并为主题类别；最终提炼出三个层面的痛点主题（导航入口迷失、认知障碍、情绪冲突）及一个核心心理洞察（信任人胜过信任系统）。

#### 3.1.2 核心痛点发现

调研梳理出三个层面的痛点，通过主题编码共归纳 23 条初始代码，聚焦为以下三类主题：

层面一：导航入口迷失

高频功能入口隐蔽，搜索体验欠佳。老年用户在查找养老金认证功能时，由于首页入口不明显，被迫依赖搜索栏。手写输入识别率低、语音搜索对方言支持不足，搜索功能未将高频功能进行显性前置。7 位受访者中有 6 位在任务测试环节中途迷失，平均探索时间超过 3 分钟。

层面二：认知障碍

- 返回逻辑混乱：多位受访者对页面连续返回逻辑不适应，习惯性点一次返回退出流程，但部分服务需多次返回，导致误认为操作失败；
- 验证码输入流程中断率高：7 位受访者全部在短信验证码环节遇到困难，需切换 APP 查看短信再返回填写，过程中页面可能刷新失效；
- 人脸识别容错性差：部分功能要求完成眨眼或摇头动作，老年人动作幅度小、速度慢，常被识别为失败（U6 曾连续失败 5 次后彻底放弃线上登录）；
- 弹窗打断流程：系统升级提示或跳转应用商店导致流程中断、信息丢失。

层面三：情绪冲突

- 缺乏操作安全网：误触返回键即导致表单数据全部丢失，无容错机制；
- 依赖线下，缺乏引导：系统缺乏有效的引导机制，老年人形成"去线下办才保险"的行为定势。5 位受访者明确表示，即便学会了线上操作，遇到"要紧事"仍会去线下办。

核心心理洞察：通过对访谈文本的跨样本比较分析，本研究识别出一条贯穿所有受访者的深层心理线索——信任人胜过信任系统。老年用户的线上操作障碍并非源于"不会用"，而是源于"不敢用"。他们清楚线上办事更方便，却宁可多走一趟线下，因为线下有"人"——一个可以当场确认、当场纠错、当场承担责任的帮助者。这一洞察直接驱动了"受控响应型代理"的设计定位：代理不是替用户做决定的系统，而是在用户身边扮演"线上工作人员"角色的协助者。

#### 3.1.3 需求优先级

| 需求 | 优先级 | 来源 |
|---|---|---|
| 语音对话唤起代理，代理引导完成操作 | P0 | 用户明确表达 + 痛点层面 2 |
| 代理帮忙处理验证码（代读代填） | P0 | 高频痛点，100% 受访者提及 |
| 代理帮忙找入口（跳转页面） | P0 | 痛点层面 1 |
| 操作有安全感，出错可恢复 | P0 | 痛点层面 3 |
| 关键步骤仍由用户自己完成（不失控感） | P0 | 信任心理需求 |
| 操作记录可查看（子女放心） | P1 | 次要用户需求 |
| 表单中断后可恢复（草稿箱） | P1 | 痛点层面 3 容错延伸 |

```
    └── 我的（/my）
        ├── 草稿箱（/elder/drafts，新增）
        ├── 操作记录（/elder/operation-logs，新增）
        └── 小浙设置（/elder/agent-settings，新增）
```

页面体系的设计遵循"最小改动"原则：在浙里办原有导航骨架基础上，仅新增草稿箱、操作记录、小浙设置三个页面，其余页面均为对原有功能的适老化重构。

#### 3.3.2 助手入口设计

小浙助手不占用独立页面，也不占用底部导航栏位置——底部导航保持"首页"和"我的"两项，与原浙里办长辈版保持一致，避免增加老年用户的认知负担。

助手以**悬浮按钮（AgentFab）**的形式叠加在业务页面之上，具体设计如下：

- 悬浮按钮吸附于屏幕右侧边缘，直径 52dp，橙色圆形，中央为白色"浙"字；
- 按钮可拖动，松手后自动吸附到左右边缘，不遮挡页面核心内容；
- 点击后从右下角弹出气泡聊天窗（300×340dp），支持拖动调整位置；
- 有未读消息时按钮显示红点提示，脉冲动画吸引注意；
- 气泡窗关闭后按钮重新出现，随时可再次唤起。

这一设计的核心考量是：助手是"随叫随到"的辅助角色，而非强制介入的主导界面。老年用户在不需要帮助时，悬浮按钮安静地停在边缘，不干扰正常操作；需要帮助时，一次点击即可唤起，无需记忆任何导航路径。

#### 3.3.3 代理面板叠加机制

气泡聊天窗以浮层形式叠加在业务页面之上，不触发路由跳转，业务页面始终保持可见：

- 聊天窗高度 340dp，宽度 300dp，位于屏幕右下角（可拖动）；
- 上方业务页面完整可见，老年用户始终知道自己在哪个页面操作；
- 代理执行高亮指令时，目标元素通过 `AgentElementRegistry` 定位，在全局 Overlay 上绘制橙色脉冲描边；
- 窗口内含对话气泡区（可滚动）、文字输入框、发送按钮，结构清晰；
- 麦克风录音通过 `MediaRecorder` API 实现，录音数据以 `audio_chunk` 消息发送至后端讯飞 ASR 识别。

与"底部半屏面板"方案相比，浮动气泡窗的优势在于：业务页面完全可见，老年用户可以一边看页面内容一边与代理对话，不会产生"页面被遮住了"的空间迷失感。

### 3.5 交互设计_REPLACE_START

#### 3.5.1 三条横切全局原则

以下三条原则贯穿所有设计决策，优先级最高，不因具体场景而妥协。

原则一：代理永远不主动，未被唤醒时完全沉默。

原则二：确定性操作代理止步，凡按下即产生不可撤销后果的按钮，代理只引导不代按。

原则三：权限一事一授，没有持久化全局授权，每个敏感操作当场问当场用。

这三条原则的关键在于，它们不是对话层面的约定，而是在架构层面被固化：确定性按钮永远不注册为代理工具（物理隔离），权限状态随 WebSocket 会话同步消亡，代理唤醒机制只有一个触发入口。这使得"代理无法越权"成为技术保障而非口头承诺，消解了老年用户对智能代理"失控"的深层焦虑。

#### 3.5.2 能力分级矩阵

受控响应型代理的"受控"程度按照场景需要分为三个能力等级（L1/L2/L3），等级决定代理可以触碰什么边界：

| 能力项 | L1 | L2 | L3 |
|---|:-:|:-:|:-:|
| 语音对话、理解意图 | ✓ | ✓ | ✓ |
| 复述确认（执行前必做） | ✓ | ✓ | ✓ |
| 页面跳转（导航代劳） | ✓ | ✓ | ✓ |
| 语音引导（"请点这个绿色按钮"） | ✓ | ✓ | ✓ |
| 屏幕高亮目标元素 | ✓ | ✓ | ✓ |
| 代读外部内容（如短信验证码） | ✗ | ✓ | ✓ |
| 代填非敏感字段（手机号、缴费年度等） | ✗ | ✓ | ✓ |
| 代填敏感字段（身份证号、密码、银行卡号） | ✗ | ✗ | ✓ |
| 按确定性按钮（登录/提交/去支付） | ✗（永远不） | ✗（永远不） | ✗（永远不） |

能力分级的设计逻辑是：用户授出多少信任，代理就获得多少权限，而不是默认最高授权。敏感字段的判断不依赖 AI 的运行时判断，而是由硬编码白名单定义（身份证号、密码、银行卡号），杜绝隐私泄露风险。确定性操作是一条绝对红线，不存在越界的空间。

4 个核心场景与能力等级的对应关系：

| 场景 | 能力等级 | 代理止步点 |
|---|---|---|
| 登录·刷脸分支 | L1 | 刷脸动作由用户完成；代理只语音引导 + 高亮 |
| 登录·验证码分支 | L2 | 代填验证码后止步于"登录"按钮 |
| 医保缴费 | L2–L3 | 表单填完后止步于"去支付"按钮 |
| 医保/养老金查询 | L1–L2（全链路） | 纯查询无确定性操作，代理可全程自动 |

这一分级矩阵使"代理的边界"可以被用户直观感知，也使测试和验收有了明确的可观测标准。

#### 3.5.3 多模态协同策略

本研究探索语音、触控、视觉三模态的协同整合，核心策略是冗余互补：同一信息通过多个通道同时传递，确保任一感官通道退化不影响信息接收。

- 代理说话的同时高亮对应界面元素（语音 + 视觉同步）；
- 用户可以说话也可以点击表达意图（语音 + 触控可互换）；
- 查询结果语音播报 + 屏幕大字显示（听觉 + 视觉双通道）。

### 3.6 交互设计

#### 3.6.1 代理交互核心流程

所有场景共用同一套核心交互流程：

```
用户点击"助手"按钮
  → 代理面板从底部滑上（300ms easeOutCubic），上方页面变暗
  → 代理语音问候"您好，我是小浙，有什么可以帮您？"
  → 用户按住麦克风说出需求
  → ASR 识别 → LLM 理解意图
  → 代理复述确认（"帮您XXX，对吗？"）
  → 用户确认（语音"对"或点击确认按钮）
  → 代理执行任务（跳转 / 高亮 / 代填…）
  → 涉及敏感操作时当场弹出授权卡片，一事一授
  → 确定性操作前代理停止，高亮按钮提示用户亲手点击
  → 任务完成，代理语音播报结果，面板延迟 1500ms 自动关闭
```

代理面板占屏幕高度 45%，从底部滑上；上方业务页面始终可见（覆盖 40% 透明度蒙层），用户无需离开当前页面即可与代理对话。代理执行高亮指令时，蒙层在目标元素区域挖空，使高亮元素透出可点击。

#### 3.6.2 场景 1：登录·刷脸分支（L1 纯引导）

能力等级：L1（语音 + 高亮，代理不碰任何表单字段或按钮）

代理在此场景中只扮演"解说员"角色：语音引导每一步应做什么，高亮对应元素，但不帮用户点击任何按钮。

设计决策核心：摄像头权限弹窗是老年用户恐慌的最高风险点。在旧版无引导的情况下，弹窗突然出现会让用户怀疑被攻击（U3："一个弹窗说要使用摄像头，我以为是钓鱼，赶紧关了"）。本研究的应对策略是预告式语音引导：在弹窗出现之前，代理先行语音解释"马上会弹出一个使用摄像头的请求，这是为了刷脸认证，请您点同意"，将突发弹窗转变为有预期的步骤，消除情感低点。

刷脸动作引导：代理语音引导"请您看着摄像头，慢慢眨眨眼"，用通俗语言替代系统的技术性提示，降低老年用户对动作幅度和速度的不确定感。

关键节点：登录按钮和同意条款始终由用户自己点，代理只高亮提示。情感峰值设计：用户在代理陪伴下第一次独立完成刷脸登录，掌控感与成就感同步到达峰值——这一情感体验的设计意图是将"代理帮助的成功"转化为"我自己做到了"的记忆。

| 步骤 | 用户动作 | 屏幕变化 | 代理语音 |
|---|---|---|---|
| 1 | 点击"助手"按钮 | 面板从底部滑上，页面变暗 | "您好，我是小浙，有什么可以帮您？" |
| 2 | 按住麦克风说"帮我登录" | 状态行"正在听您说话…" | — |
| 3 | 松开麦克风 | 思考动画；气泡"帮您登录，对吗？" | "帮您登录，对吗？" |
| 4 | 说"对" | 页面跳转到登录页；"同意条款"勾选框橙色高亮 | "请您先勾选下方的同意条款" |
| 5 | 点击勾选框 | 高亮消失；"登录"按钮橙色高亮 | "好，现在请点一下登录按钮" |
| 6 | 点击"登录" | 跳转刷脸验证页 | "马上会弹出使用摄像头的请求，这是为了刷脸认证，请您点同意" |
| 7 | 点"同意" → 点"允许" | 进入刷脸验证页面 | "摄像头已经打开了，请您看着摄像头，慢慢眨眨眼" |
| 8 | 完成刷脸动作 | 刷脸成功，跳转首页；面板 1500ms 后关闭 | "登录成功了，欢迎您！" |

#### 3.6.3 场景 2：登录·验证码分支（L2 代读代填）

能力等级：L2（在 L1 基础上，增加代读外部内容、代填非敏感字段）

代理在此场景中升级为"操作员"：代读短信验证码并代填到输入框，前提是获得用户当场授权。

设计决策核心：验证码流程要求用户在浙里办和短信 APP 之间来回切换，这是老年用户放弃率最高的环节（7 位受访者 100% 遇到此困难）。直接解决方案是代理代读代填，但必须解决授权边界问题——短信是隐私数据，代理不能默认有权读取。

本研究的解法是一事一授权：进入验证码页面后，代理不直接代填，而是先弹出授权卡片"这次我帮您读验证码并填上去，可以吗？"；用户授权后，填充动画逐位显示数字填入过程（透明而非突然出现）；代理填完后主动说"验证码已经帮您填好了，您看一下对不对"，给用户核验空间。

"登录"按钮始终由用户亲手按，代理高亮提示止步于此。这一止步点的意义在于：用户在整个流程中始终保留最后一个决策动作，形成"这次登录是我自己完成的"的主体感。

| 步骤 | 用户动作 | 屏幕变化 | 代理语音 |
|---|---|---|---|
| 1 | 点击"助手"按钮 | 面板滑上 | "您好，我是小浙，有什么可以帮您？" |
| 2 | 说"帮我用验证码登录" | 思考动画；气泡"帮您用验证码登录，对吗？" | "帮您用验证码登录，对吗？" |
| 3 | 说"对" | 页面跳转到验证码页 | "好的，帮您打开验证码登录" |
| 4 | — | 面板内弹出授权卡片 | "这次我帮您读短信里的验证码并填上去，可以吗？" |
| 5 | 点击"可以" | 授权卡片收起；验证码输入框逐位填入动画 | "验证码已经帮您填好了，您看一下对不对" |
| 6 | 确认无误 | "登录"按钮橙色高亮（5 秒） | "对的话，请您点一下登录" |
| 7 | 亲手点"登录" | 登录成功，跳转首页；面板自动关闭 | "登录成功了！" |

#### 3.6.4 场景 3：医保缴费（L2-L3，含敏感字段授权）

能力等级：L2–L3（含敏感字段单独授权，即身份证号需额外确认）

代理在此场景展示完整的权限一事一授流程：普通字段直接代填，敏感字段（身份证号）需单独授权，支付按钮代理永远不代按。

设计决策核心：本场景的核心挑战是老年用户对身份证号填写的高度敏感（U1："一上来就要身份证号，我不敢填，怕是骗子"）。本研究的应对策略是分级授权可见化：普通字段（缴费年度、金额）代理直接填，附语音播报；身份证号字段弹出专门的授权卡片，突出强调字段名称（24sp 粗体），让老年用户清楚知道代理想碰的是什么具体信息，而非抽象授权。身份证号填入后脱敏显示（如"330…\*\*…6"），在面板上对用户可视——既告知已填，又维护隐私感。

逐字段代填时，每填一个字段都有语音播报，让用户实时了解代理在做什么，避免"代理在背后悄悄操作"的失控感。

草稿箱机制：若用户在填写过程中关闭面板或离开页面，已填字段自动快照保存到草稿箱（IndexedDB）。下次唤醒代理时，代理在问候后提示"您上次有个未完成的医保缴费，要继续吗？"——这将调研中普遍存在的"操作中断数据丢失"痛点直接化解。

"去支付"按钮高亮后代理不发出任何进一步指令，静候用户操作。支付链路代理不跟入——这是能力边界的物理保障，而非依赖代理的自我约束。

| 步骤 | 用户动作 | 屏幕变化 | 代理语音 |
|---|---|---|---|
| 1 | 点击"助手"按钮 | 面板滑上 | "您好，我是小浙，有什么可以帮您？" |
| 2 | 说"帮我缴医保" | 思考动画；气泡"帮您缴医保，对吗？" | "帮您缴医保，对吗？" |
| 3 | 说"对" | 页面跳转到医保缴费页 | "好的，帮您打开医保缴费页面" |
| 4 | — | 缴费对象、缴费年度、缴费金额依次填入，每填一项有动画 | "已帮您选择缴费对象：本人；缴费年度：2026年；缴费金额：4800元" |
| 5 | — | 面板内弹出授权卡片：【身份证号】 | "还需要填一下身份证号，这次我帮您填可以吗？" |
| 6 | 点击"可以" | 身份证号填入，脱敏显示（330…**…6） | "身份证号已经帮您填好了" |
| 7 | — | "去支付"按钮橙色高亮（持续） | "表单已经填好了，请您确认一下，然后点橙色的"去支付"按钮" |
| 8 | 亲手点"去支付" | 进入支付页面；面板自动关闭 | "好的，祝您缴费顺利！" |

#### 3.6.5 场景 4：医保 / 养老金查询（L1-L2 全链路）

能力等级：L1–L2（全链路，无确定性操作，代理可独立完成全部步骤）

此场景无确定性操作，代理可以全链路完成——从唤醒到结果播报，无需用户手动操作任何步骤。这是 4 个场景中情感曲线最平滑的：用户说一句话，代理从导航到代填查询条件到代按"查询"按钮（非确定性操作）一气完成，中间零决策负担。

设计决策核心：查询结果的呈现是本场景的关键设计点。查询出来的数字（如"3280 元"）对老年用户的意义重大，必须保证信息有效到达，不能因感官退化而失传。本研究采用双通道冗余策略：屏幕大字卡片展示数字（24sp+），同时语音播报"您的养老金本月发放了 3280 元，已经到账了"。双通道设计逻辑是互为兜底——没听清数字的用户抬头看屏幕，没记住屏幕的用户依赖语音记忆，任意一个通道保证信息送达。

"查询"按钮属于非确定性操作（查询不会产生不可撤销的后果），代理可直接代按（高亮 1 秒提示后自动触发），不需要用户授权——这是能力分级矩阵中"L1-L2 全链路"的直接体现。

| 步骤 | 用户动作 | 屏幕变化 | 代理语音 |
|---|---|---|---|
| 1 | 点击"助手"按钮 | 面板滑上 | "您好，我是小浙，有什么可以帮您？" |
| 2 | 说"帮我查养老金" | 思考动画；气泡"帮您查养老金，对吗？" | "帮您查养老金，对吗？" |
| 3 | 说"对" | 页面跳转到养老金查询页 | "好的，帮您打开养老金查询" |
| 4 | — | 查询条件自动填入；"查询"按钮高亮 1 秒后自动触发 | "帮您查一下" |
| 5 | — | 上方页面渲染查询结果（大字卡片）；面板气泡同步显示结果摘要 | "您的养老金本月发放了 3280 元，已经到账了" |
| 6 | — | 面板 3000ms 后自动关闭 | — |

#### 3.6.6 错误处理与容错设计

| 错误情况 | 代理行为 | 设计原则 |
|---|---|---|
| 没听清（ASR 失败） | 说"不好意思没听清，您再说一遍"，最多重试 3 次 | 不让用户感到责备 |
| 连续 3 次没听清 | 提示"您也可以试试手动操作"，面板关闭 | 优雅退出，不强迫 |
| 理解意图错误 | 复述确认机制拦截：用户说"不对"则重新听取 | 执行前必有确认 |
| 网络断开 | 面板顶部红色横幅"网络已断开"，麦克风不可用 | 明示状态，不静默失败 |
| LLM 推理等待 | 显示"小浙正在想…"三点跳动动画，上限 8 秒 | 避免老年用户误判为死机 |
| 超出能力范围 | 坦白"这个功能我暂时帮不了您，您可以自己试试" | 不假装能做，不沉默 |

### 3.9 技术可行性分析

#### 3.9.1 前端可行性

Flutter Web 已有成熟的生产级应用，本研究已完成 17 个页面的原型实现，技术路径经过验证。GoRouter 路由体系适配代理的页面跳转指令；Widget 树的 GlobalKey 机制支持精确的元素高亮操作；AnimationController 支持面板滑入、授权卡片弹出等所有动效需求。

音频录制使用浏览器 MediaRecorder API，长按麦克风按钮录音，松开后将录音数据经 WebSocket 传至后端讯飞 ASR 识别，前端无需处理音频编解码。代理面板通过页面右下角的悬浮气泡按钮手动唤醒，交互入口明确，无需持续监听麦克风。

IndexedDB 支持草稿箱和操作记录的本地持久化存储，前端无需账号体系即可实现数据保存与查询。

#### 3.9.2 后端与代理可行性

Agno 轻量级代理框架（Python）原生支持 DeepSeek-V3 集成。代理工具按场景静态分配（`SCENE_TOOLS` 字典），每个场景只注册该场景所需的工具子集——例如刷脸场景只有导航和高亮工具，物理上不存在代填敏感字段的能力，从架构层面保障了能力边界。Agno session 机制与 WebSocket 连接生命周期天然对齐，session 关闭时授权自动失效，无状态残留风险。

DeepSeek-V3 API 中文理解与生成质量高，月均用量估计成本 < ¥10，适合毕设场景。

#### 3.9.3 语音合成可行性

语音合成采用讯飞 TTS（主）+ Edge TTS（备选）双方案。讯飞 TTS 通过 WebSocket 接入，语速设置为 40（正常值 50，慢约 15%），音色选用"晓燕"，适合老年用户收听；Edge TTS 为微软免费方案，语速参数 `-15%`，在讯飞 API Key 未配置时自动切换。两者均在后端合成 mp3 后经 WebSocket 以 base64 推送至前端播放，链路一致。

#### 3.9.4 关键技术风险

| 风险项 | 等级 | 应对方案 |
|---|---|---|
| Web Speech API 在 Safari 不可用 | 中 | 主力演示环境为 Android Chrome；文字输入框作全程备选 |
| 讯飞 ASR 对老年口音识别准确率 | 中 | Prompt few-shot 调优 + 文字输入框备选；最多重试 3 次 |
| DeepSeek API 响应延迟 | 低 | `agent_thinking` 状态动画消除等待焦虑，上限 8 秒超时提示 |
| 服务器部署 HTTPS 配置 | 低 | Let's Encrypt 免费证书 + Nginx 反向代理，流程成熟 |

### 3.10 可用性研究

#### 3.10.1 测试设计

测试对象：5—8 位 60 岁以上老年用户，招募来源为调研社区，覆盖不同数字素养层次。

测试方法：任务式可用性测试 + 半结构式访谈。测试者采用"边做边说"协议，观察者记录操作行为与情绪反应。

测试任务（覆盖 4 个核心场景）：

| 任务 | 说明 |
|---|---|
| T1 | 唤醒小浙，让它帮你完成刷脸登录 |
| T2 | 用验证码方式登录，让小浙帮你填验证码 |
| T3 | 让小浙帮你缴今年的医保 |
| T4 | 让小浙帮你查上个月的养老金发放情况 |

评估指标：

| 指标 | 说明 |
|---|---|
| 任务完成率 | 在无辅助情况下独立完成任务的比例 |
| 任务完成时间 | 从唤醒代理到任务结束的时长 |
| 操作错误率 | 偏离预设路径的操作次数 / 总操作次数 |
| 用户满意度 | SUS 量表（System Usability Scale）得分 |
| 信任感评分 | 自定义 5 题李克特量表 |
| 代理控制感评分 | 自定义 3 题李克特量表 |

**量表一：SUS 系统可用性量表（标准版，10 题）**

SUS 由 Brooke（1996）提出，是可用性测试领域使用最广泛的主观评估工具。本研究采用其标准版本，每题为 5 点李克特量表（1=非常不同意，5=非常同意）。

| 题号 | 题目 | 计分方向 |
|---|---|---|
| S1 | 我觉得我愿意经常使用这个系统 | 正向 |
| S2 | 我觉得这个系统过于复杂，使用起来不方便 | 反向 |
| S3 | 我觉得这个系统很容易使用 | 正向 |
| S4 | 我认为我需要技术人员的帮助才能使用这个系统 | 反向 |
| S5 | 我觉得这个系统的各种功能整合得很好 | 正向 |
| S6 | 我认为这个系统存在太多不一致的地方 | 反向 |
| S7 | 我认为大多数人能够很快学会使用这个系统 | 正向 |
| S8 | 我觉得使用这个系统非常繁琐 | 反向 |
| S9 | 我在使用这个系统时感到很有把握 | 正向 |
| S10 | 在正式使用这个系统之前我需要学习很多东西 | 反向 |

计分方法：正向题得分为（原始分 - 1），反向题得分为（5 - 原始分），10 题得分之和乘以 2.5，最终得分范围为 0—100 分。根据 Bangor 等人（2008）的评级标准，≥ 85 分为"优秀"，70—85 分为"良好"，50—70 分为"可接受"，< 50 分为"不可接受"。本研究预期 SUS ≥ 75 分。

**量表二：代理信任感量表（自定义，5 题）**

本量表基于 Madsen & Gregor（2000）的自动化信任量表及 Lee & See（2004）的人机信任框架，结合本研究的代理设计特点定制。每题为 5 点李克特量表（1=非常不同意，5=非常同意）。

| 题号 | 题目 | 维度 |
|---|---|---|
| T1 | 我相信"小浙"不会在我不知情的情况下做出重要操作 | 可预测性 |
| T2 | 当"小浙"请求我的授权时，我理解它想做什么 | 透明度 |
| T3 | 我愿意继续使用"小浙"来帮我办理政务事项 | 接受度 |
| T4 | 我感觉"小浙"的行为在我可以控制的范围内 | 控制感 |
| T5 | 我担心"小浙"会替我做出不该做的操作 | 风险感知（反向）|

计分方法：T1—T4 为正向题，T5 为反向题（5 - 原始分），5 题均值即为信任感得分（1—5 分），得分越高表示信任感越强。

**量表三：代理控制感量表（自定义，3 题）**

本量表聚焦于"受控响应型代理"的核心设计目标——让老年用户始终感到自己是操作的主导者。每题为 5 点李克特量表（1=非常不同意，5=非常同意）。

| 题号 | 题目 |
|---|---|
| C1 | 在整个操作过程中，我感到自己一直在做决定，而不是被系统推着走 |
| C2 | 当我想停止"小浙"的帮助时，我知道如何停下来 |
| C3 | 我能随时看到"小浙"在帮我做什么，不会感到迷糊 |

计分方法：3 题均值即为控制感得分（1—5 分）。

#### 3.10.2 预期测试结论（待实测验证）

基于设计原则推演，预期：任务完成率 ≥ 80%（老年用户在有代理协助下）；用户满意度（SUS）≥ 75 分（良好水平）；信任感评分高于市场上现有适老化 APP 参照组。

如测试发现问题，将优先修复影响任务完成率的核心路径缺陷，其次优化信任感相关的话术与授权弹窗设计。

## 4 方案实现

### 4.1 技术架构

#### 4.1.1 系统整体架构

系统由前端、后端代理服务、外部 API 三层构成，通过 WebSocket 长连接实时通信。

```
用户设备（Chrome 浏览器）
  └── Flutter Web 前端
        ├── 业务页面层（17 个长辈版页面）
        ├── 代理 UI 层（底部面板、麦克风、高亮覆盖层）
        ├── 本地存储层（IndexedDB：草稿箱、操作记录）
        └── WebSocket Client
              │  WebSocket 长连接
后端代理服务（FastAPI on 云服务器）
  ├── WebSocket Handler（消息路由 + 状态机）
  ├── Agno Agent（意图理解、工具调用、HITL 确认）
  ├── ASR 适配器（讯飞 ASR → 文字）
  └── TTS 适配器（讯飞 TTS 主 / Edge TTS 备 → mp3）
              │  HTTPS API
外部服务
  ├── DeepSeek-V3 API（LLM 推理）
  ├── 讯飞 ASR API（语音识别）
  └── 讯飞 TTS / Edge TTS（语音合成）
```

[图：系统架构图（待制作）]

#### 4.1.2 三条原则在架构中的落地

| 原则 | 架构实现 |
|---|---|
| 代理永远不主动 | WebSocket Handler：仅在收到 `agent_wake` 消息时才创建 Agno Agent 并运行，连接建立时完全静默 |
| 确定性操作止步 | 确定性按钮（"登录"、"去支付"）不注册为 Agno Tool，Agent 在物理层面无法调用；白名单机制前端双重校验 |
| 权限一事一授 | 敏感操作工具（`fill_field_sensitive`、`read_sms`）在 `agent_core.py` 中列入 `_SENSITIVE_TOOLS` 集合；`ws_handler` 在工具调用前检查该集合，命中则暂停 Agno 执行、推送 `permission_request` 到前端，等待用户当场授权后恢复 |

#### 4.1.3 WebSocket 消息协议

所有消息遵循统一格式：`{ "type": "...", "payload": {...}, "ts": "ISO8601" }`

核心消息类型：

| 消息类型 | 方向 | 含义 |
|---|---|---|
| `agent_wake` | 前→后 | 用户点击"助手"按钮，唤醒代理 |
| `agent_ready` | 后→前 | 代理就绪，推送问候语 + 草稿提示 |
| `asr_result` | 后→前 | 语音识别结果（文字） |
| `agent_thinking` | 后→前 | LLM 推理中，前端显示等待动画 |
| `agent_reply` | 后→前 | 代理回复（文字 + TTS 音频） |
| `user_confirm` | 前→后 | 用户确认/拒绝代理复述 |
| `cmd_navigate` | 后→前 | 指令：跳转页面 |

### 4.2 前端开发

#### 4.2.1 技术选型与工具链

- 框架：Flutter Web（Dart），构建跨平台 Web 应用，在 Chrome 浏览器上运行
- 路由：GoRouter 14（17 条路由，代理指令通过 `context.go()` 触发页面跳转）
- 状态管理：Riverpod 2（`flutter_riverpod ^2.4.0`）——全局状态（登录态、当前模式）通过 `NotifierProvider` 管理，代理面板内部状态采用 `StatefulWidget` 管理，两层分离保证面板可独立复用
- 本地存储：IndexedDB（通过 `idb_shim` Dart 包），用于草稿箱和操作记录的持久化
- 语音唤醒：Web Speech API（`dart:html` 调用浏览器原生 `SpeechRecognition`），持续监听"小浙"热词，触发代理面板打开
- 语音录制：浏览器 MediaRecorder API，长按麦克风按钮录音，松开后将音频 base64 编码经 WebSocket 发送至后端讯飞 ASR 识别
- 开发工具链：项目级 Flutter SDK（`./bin/flutter`），镜像源 `pub.flutter-io.cn`，与系统环境隔离

状态管理架构说明：全局状态（`loginProvider` 登录态、`modeProvider` 长辈/标准模式切换）使用 Riverpod `NotifierProvider`，在 `ProviderScope` 根节点注入，全应用响应式更新。代理面板（`AgentPanel`）内部维护独立的 `SessionState` 对象（会话 ID、对话历史、已授权权限集合、WebSocket 连接状态），面板关闭时随之销毁，不污染全局状态——这在架构上保证了"权限随会话消亡"的语义。

#### 4.2.3 高亮机制实现原理

高亮机制通过以下方案实现：

1. 业务页面中每个可被代理高亮的元素使用语义化 `GlobalKey`（如 `Key('btn_go_payment')`）注册到全局 `agentElementRegistry`；
2. 收到 `cmd_highlight` 消息后，通过 `BuildContext` + `GlobalKey` 查找元素的 `RenderBox`，获取其屏幕坐标和尺寸；
3. 在屏幕全局 Overlay 上绘制高亮描边（橙色脉冲动画），覆盖于蒙层之上；
4. 蒙层（`IgnorePointer + ColorFiltered`）在高亮元素区域挖空，使其可见且可点击。

代理只知道语义 key（如 `btn_go_payment`），不知道 DOM 结构——这是前端安全边界，未注册的元素代理无法操作。

#### 4.2.4 草稿箱与操作记录

草稿箱（IndexedDB `store: drafts`）存储页面级表单快照，最多 10 条，超出删除最旧。每条草稿包含：`page_id`、`fields`（已填字段 map）、`created_at`、`updated_at`、`status`。

操作记录（IndexedDB `store: operation_logs`）保留最近 50 条，每条包含：操作场景名、一句话摘要（由代理生成）、步骤列表（区分代理操作 vs 用户操作），涉及敏感字段的步骤值替换为 `[已隐藏]`。

### 4.3 后端开发

#### 4.3.1 后端结构

后端采用 FastAPI + Agno Agent，主要文件：

| 文件 | 职责 |
|---|---|
| `main.py` | FastAPI 入口，定义 WebSocket 端点 `/ws/session/{session_id}` |
| `ws_handler.py` | WebSocket 状态机（idle / listening / confirming / executing / done），消息路由 |
| `agent_core.py` | Agno Agent 创建、场景工具注册、session 管理 |
| `models.py` | Pydantic 消息模型（所有 WS 消息类型的 schema） |
| `deepseek_client.py` | DeepSeek API 直接调用封装（意图分类、复述话术等轻量推理） |
| `asr_adapter.py` | 讯飞 ASR 适配器，接收音频 bytes，返回识别文字 |
| `tts_adapter.py` | TTS 适配器，讯飞 TTS（主）+ Edge TTS（备），返回 mp3 bytes |
| `tools/` | 操作工具函数（navigate、highlight、fill_field、press_button、read_sms） |
| `prompts/` | 各场景 system prompt（意图分类、复述话术、超出能力应答等） |

#### 4.3.2 Agno Agent 配置

```python
# 以医保缴费场景为例（L2-L3）
agent = Agent(
    model=DeepSeek(id="deepseek-chat"),
    tools=[
        cmd_navigate,
        cmd_highlight,
        fill_field_normal,            # 普通字段代填（L2）
        fill_field_sensitive,          # 敏感字段代填（L3，触发 HITL）
    ],
    session_id=session_id,
    response_model=AgentInstruction,   # Pydantic schema，保证结构化输出
    system_prompt=prompts["yibao_jiaofei"],
)
```

场景工具集白名单：每个场景在 `agent_core.py` 中维护独立的工具列表（`SCENE_TOOLS`），代理只能调用当前场景注册的工具——这是能力分级矩阵在代码层面的直接映射。L1 场景（如 `login_face`）只注册 `cmd_navigate` 和 `cmd_highlight`，从物理上排除填字段的可能；L2 场景增加 `fill_field_normal` 和 `read_sms`；L3 场景再增加 `fill_field_sensitive`。

`fill_field_sensitive` 触发 HITL 暂停机制：Agno 调用此工具前自动暂停代理执行，`ws_handler` 推送 `permission_request` 消息到前端，前端弹出授权卡片；用户说"可以"后推送 `permission_response`（`granted=true`），后端恢复代理执行；用户拒绝则跳过该步骤并语音告知。

需要说明的是，本原型中部分工具使用模拟数据演示交互流程。例如 `read_sms` 工具返回预设的验证码（`"123456"`），`fill_field_sensitive` 使用预设的脱敏身份证号。这是因为原型运行在浏览器环境中，无法真实读取手机短信或访问用户证件信息。在实际产品化部署中，这些工具需对接系统级 SMS API（如 Android Intent）和用户预存的加密个人信息库。模拟数据不影响对代理交互流程、授权机制和止步边界的验证——这些才是本研究关注的核心设计问题。

确定性按钮（`cmd_press_button` 对应的目标按钮名）在场景配置中通过白名单区分：只有查询类按钮（"查询"、"搜索"）被允许代按，支付/提交/登录类按钮永远不在白名单内，无论代理生成什么指令，前端执行层会在 `AgentCommandExecutor` 中二次校验并拒绝执行。

#### 4.3.3 WebSocket 状态机

后端 `ws_handler.py` 实现了一个五状态有限状态机，完整管理一次代理会话的生命周期：

```
idle（连接建立，代理静默）
  → [收到 agent_wake] → listening（等待用户语音或文字输入）
  → [text_input / asr_result] → confirming（代理生成复述，等待用户确认）
  → [user_confirm yes] → executing（Agno Agent.run() 在此状态运行）
  → [task_done / out_of_scope] → done → idle（重置，等待下次唤醒）

HITL pause（在 executing 状态内的子流程）：
  Agno 调用敏感工具 → ws_handler 推送 permission_request
  → 等待 permission_response
  → granted=true：恢复 Agno 执行
  → granted=false：语音告知，跳过该步骤继续
```

五状态设计的关键约束：在 `idle` 状态下收到任何指令都不激活代理，只有 `agent_wake` 能将状态机推进到 `listening`——这在协议层面保证了"代理永远不主动"原则，即便前端因 Bug 意外发送消息，后端也不会触发代理行为。

### 4.4 API 集成

#### 4.4.1 DeepSeek-V3 集成

通过 Agno 的 `DeepSeek` model 类接入，API key 存储在 `.env` 文件中（不入版本库）。使用 `deepseek-chat` 模型（DeepSeek-V3），具备强中文理解能力，接口响应时间通常在 1—3 秒，适合实时对话场景。

意图分类 prompt 通过 few-shot 示例覆盖每个场景的 3—5 种口语表达变体，确保"缴医保"、"交医保"、"帮我缴医保"等变体都能正确分类到 `yibao_jiaofei` 场景。

#### 4.4.2 语音识别与合成集成

**语音识别（ASR）**：前端使用浏览器 MediaRecorder API 录制音频，以 200ms 为间隔生成音频块，松开麦克风后将所有块 base64 编码，通过 `audio_chunk` / `audio_end` 消息序列发送至后端。后端 `ws_handler.py` 累积音频字节后调用 `get_asr_adapter().recognize(chunks)`，讯飞 ASR 返回识别文字，后端推送 `asr_result` 消息并进入意图理解流程。

**语音合成（TTS）**：后端 `tts_adapter.py` 实现讯飞 TTS（主）和 Edge TTS（备）两个适配器，通过 `get_tts_adapter()` 工厂函数按环境变量配置自动选择。讯飞 TTS 通过 WebSocket 接入，语速 40（慢约 15%），音色"晓燕"；Edge TTS 使用 `zh-CN-XiaoxiaoNeural` 音色，语速 `-15%`。合成后的 mp3 bytes 经 base64 编码嵌入 `agent_reply` / `task_done` 消息推送至前端播放。

#### 4.4.3 代理面板触发方式

代理面板通过界面上的悬浮按钮（`AgentFab`）手动触发。用户点击右下角的"浙"字橙色圆形按钮后，气泡聊天窗展开，用户可通过文字输入或长按麦克风说话与代理交互。

热词唤醒（Web Speech API 持续监听"小浙"）在原型阶段暂未启用，原因如下：浏览器的 `SpeechRecognition` 需要持续占用麦克风权限，在未建立 HTTPS 的局域网演示环境下权限申请流程复杂，且后台持续识别会增加额外功耗。在面向实际部署的版本中，可通过 `WakeWordListener` 组件（已在代码库中实现）重新启用，无需修改业务逻辑。

### 4.5 测试部署

#### 4.5.1 测试策略

测试分三层：

单元/组件测试：对核心 Flutter Widget（AgentPanel、PermissionCard）和后端工具函数进行单元测试，覆盖正常路径和边界条件（如 `permission_response granted=false` 的处理）。

E2E 集成测试：在后端编写 7 条场景测试用例（对应 N4 任务表中 TC01—TC07），使用 `httpx` 模拟 WebSocket 连接，验证从 `agent_wake` 到 `task_done` 的完整消息序列是否符合预期。

真机测试：在 Android 手机 Chrome 上运行所有 4 个核心场景，验证 Web Speech API 语音识别准确率（目标 ≥ 80%），以及高亮动效、TTS 播放、面板动画的实际体验。

#### 4.5.2 部署方案

本研究采用局域网部署方案：开发机（Windows）与测试手机连接同一 Wi-Fi，手机通过开发机局域网 IP 直接访问前后端服务，无需云服务器。

- 后端服务：`uvicorn main:app --host 0.0.0.0 --port 8000`，监听所有网卡，局域网内可达；
- 前端静态文件：`flutter build web` 后由 Python `http.server` 或 Nginx 在本机 8765 端口托管；
- 手机访问地址：`http://<开发机局域网IP>:8765`，WebSocket 连接至 `ws://<IP>:8000/ws/session/{id}`；
- HTTPS：局域网 HTTP 环境下，Chrome for Android 将 `<局域网IP>` 视为非安全源，MediaRecorder 麦克风权限受限。解决方案：使用 mkcert 生成本机自签证书，在手机上安装根证书后信任，即可通过 `https://<IP>` 访问并获取麦克风权限。

此方案无需域名和公网服务器，适合答辩演示场景；如需公网访问，可通过 ngrok 等内网穿透工具临时暴露本机端口。

## 6 结论

### 6.1 主要研究贡献

本研究从理论与实践两个维度做出了如下贡献：

理论层面，提出受控响应型智能代理这一交互模式，系统回应了老年用户"信任人胜过信任系统"的深层心理需求。该模式的核心贡献有两点：

其一，分级授权的能力矩阵（L1/L2/L3）。区别于既有适老化智能代理研究以"最大化自主完成任务"为目标的设计取向，本研究将代理能力划分为三个边界清晰的层级，每个层级对应不同的用户授权要求，使代理行为的边界可被用户直观感知、可被测试验收。这为适老化代理的能力设计提供了可操作的分级框架（详见 §3.5.2）。

其二，三原则的架构固化。本研究将设计原则从"对话层面的约定"升级为"架构层面的约束"——通过工具白名单物理隔离、协议入口唯一化、权限状态会话级生命周期三项技术手段，使"代理不会越权"成为可验证的技术保障而非运行时判断（详见 §4.1.2）。这对于消解老年用户对智能代理"失控感"的焦虑具有根本意义。

实践层面，基于上述理论框架实现了一款可运行的浙里办长辈版高保真原型，覆盖 4 个政务服务核心场景、17 个页面，实现了从语音意图理解到多模态结果呈现的完整功能链路。原型可在手机 Chrome 浏览器上直接访问体验，技术栈全部为开源或免费方案（FastAPI、Agno、DeepSeek API、Edge TTS、Flutter Web），具备推广复制价值。

### 6.2 多模态协同的设计价值

本研究通过语音、触控、视觉三模态协同的冗余互补策略，回应了现有研究中指出的"多模态交互在适老化场景中形成有效协同尚不成熟"这一不足（王斯薇子等，2025；Bhattacharya 等，2025）。

本研究不是简单叠加多个通道，而是建立了通道间的时序对齐与语义融合机制：代理说话的同时精确高亮对应界面元素（语音-视觉时序同步）；查询结果语音播报的同时屏幕大字显示（听觉-视觉语义重叠，互为兜底）；用户既可语音表达意图也可点击面板按钮（语音-触控等价互换，消除输入障碍）。这种"通道互为兜底"的策略有效应对了老年用户感官功能的个体差异性，避免了因单一通道故障（听力退化、视力退化、发音不标准）导致整个交互失败的脆弱性。

三原则与多模态设计的关系值得特别说明：多模态协同的目标是让信息更有效地送达用户，而受控响应原则的目标是让用户对代理的行为保持掌控感。二者相辅相成：代理说出"帮您填身份证号，可以吗"的同时弹出可视化授权卡片，语音通道传递信息、视觉通道提供确认，使"一事一授权"的体验更加自然流畅，而非令人紧张的权限弹窗。

### 6.3 设计局限与反思

本研究存在以下局限，需在结论中诚实呈现：

第一，可用性测试尚未完成。受时间约束，本研究的可用性测试仍处于计划阶段，§3.9 中的预期结论（任务完成率 ≥ 80%、SUS ≥ 75 分）尚未得到实测验证。这是本研究最主要的不足，后续应补充 5—8 位老年用户的真机测试数据，用实测数据替代推演预期。

第二，代理唤醒词识别依赖网络环境。Web Speech API 在 Chrome 上需要 HTTPS 环境，本地测试（localhost）可用，但真机部署需要云服务器 + HTTPS 证书才能完整验证语音识别链路。

第三，场景覆盖有限。本研究聚焦 4 个政务场景，能够系统演示受控响应型代理的能力矩阵和三原则边界，但无法覆盖浙里办全部服务场景。未覆盖场景的代理能力需要逐场景扩展 prompt 和工具集。

第四，方言识别未实现。当前语音识别基于 Web Speech API 的普通话模型，对口音较重的老年用户（尤其是第一代移民农村的杭州郊区老人）识别准确率可能下降。

### 6.4 未来展望

未来研究可在以下方向延伸：

近期可实施（答辩后）：完成 5—8 位老年用户的可用性测试，补充任务完成率、SUS 量表、信任感评分的实测数据；在云服务器上完成 HTTPS 部署，验证真机语音识别链路；根据测试结果优化 prompt 话术，提升意图分类准确率至 ≥ 80%。

中期研究方向：探索"偏好记忆"机制——如何在不牺牲用户控制感的前提下，为高频操作（如每年医保缴费）提供"上次您授权过，这次继续吗"的渐进式信任积累，减少重复授权的摩擦；接入方言 ASR 模型提升老年用户语音识别准确率；将受控响应型代理模式扩展到医疗挂号、公积金查询等其他信息服务场景。

长期理论价值：本研究提出的"分级授权能力矩阵 + 三原则架构固化"框架，不局限于政务 APP 场景，可迁移到任何涉及老年用户与智能代理交互的领域——医疗健康数据录入、银行金融操作、社区服务预约等。这一框架的核心洞察（代理的信任不是口头承诺而是架构约束）对代理式 AI 的产品设计具有普遍参考价值。
