## 1 设计背景

### 1.1 绪论

#### 1.1.1 研究背景

我国已步入中度老龄化社会。截至 2025 年底，60 岁及以上老年人口达 3.23 亿，占总人口比例 23.0%。与此同时，移动互联网的普及使各类信息服务 APP（政务、医疗、健康、生活服务等）成为公众日常生活不可或缺的工具。然而，老年群体在使用这些 APP 时面临严峻的"数字鸿沟"困境。

本研究的核心洞察来自于现实观察：老年用户并非不愿线上办事，而是**不想在线上自己冒险办事**。相比独自操作，他们更倾向于去线下服务大厅寻求工作人员帮助。老年用户最渴望的不是更清晰的界面，而是可信赖的帮助。这种"信任人胜过信任系统"的心理，是阻碍老年用户跨越数字鸿沟的深层障碍，要求信息服务从传统的自助服务模式向**协助服务模式**转变。

近年来，以大语言模型为基础的智能代理技术快速发展。Gartner 将代理式人工智能列为 2025 年十大技术趋势之首，预测到 2028 年全球 15% 的日常工作决策将由具备自主决策能力的 AI 代理完成。智能代理依靠感知能力、规划能力、记忆能力及工具使用能力，能够理解用户意图、拆解复杂任务、调用工具执行操作，为构建"线上工作人员"提供了技术可能。

在多模态交互层面，研究显示 78% 的老年人倾向于支持多种输入方式（如语音或触控）的界面，针对认知老化的个性化多模态界面能将老年人可用性提升 30%。语音、触控、视觉等模态的协同整合为老年用户提供了更自然、更包容的交互方式。

#### 1.1.2 国内外研究现状

**信息服务 APP 适老化设计方面**，国外研究起步较早，形成了较系统的理论体系。Farage 等经典综述系统梳理了老年人在视觉、听觉、触觉、运动和记忆方面的功能变化，并提出具体设计指南，如提高对比度、控制声音频率范围、提供多通道反馈、简化任务流程等。Bhattacharya 等的叙事性综述提供了关键数据支撑：相比单一模态系统，78% 的老年人更倾向于支持多种输入方式的界面；AI 驱动的预测模型在自适应系统中预测用户偏好的准确率可达 95%；针对认知老化的个性化多模态界面能将老年人可用性提升 30%。国内方面，张晓宇等构建了包含系统易用性、系统包容性、情感支持度、服务有效度四个维度的适老化设计评价指标体系，指出当前适老化改造多停留在放大字体、简化界面等表层优化，而交互逻辑、容错机制、情感化设计仍有不足。

**多模态适老化设计方面**，国际人机交互领域的最新综述分析了 31 项关于老年人智能交互界面的研究，将交互形式分为手势交互、语音交互、多模态交互和人工智能交互四类。研究发现，语音交互因其能够绕开视觉与操作障碍，被认为是适老化设计的重要突破口，但目前许多产品只是形式上接入语音模块，却未形成真正可用的交互闭环。王斯薇子等提出以交互情境、模态信息、用户意图三要素为导向的闭环设计模型，通过建立多模态信息间的动态映射关系实现场景与用户状态的智能适配。

**智能代理方面**，Lee 等提出的任务导向型多模态代理系统（TOMAS）通过简化用户界面和整合语音命令，帮助老年人更独立地使用智能手机。CareBuddy 等多智能体协同系统在老年照护领域展现出良好的可用性。Cuadra 等开发的多模态大语言模型代理系统用于健康数据录入，揭示了多模态交互如何以互补方式提供替代路径。张诗棋等通过用户访谈深入了解了老年人对代理型适老软件的需求和体验，为设计提供了重要参考。

#### 1.1.3 现有研究不足

综合国内外研究现状，当前主要存在三方面不足：

第一，**表层改造与深层需求脱节**。现有适老化改造多局限于放大字体、简化首页等视觉层面调整，对交互逻辑、容错机制和情感化设计缺乏深入考量，未能有效降低老年人的认知负荷与操作门槛。

第二，**信任与控制的双重缺失**。传统智能代理强调自主性和主动性，其设计逻辑是代理替用户做决定、替用户执行。然而老年人对智能代理的失控存在深层焦虑——担心代理理解错误、操作错误、泄露信息，却又无法干预。这种"不信任系统"叠加"不放心代理"的双重心理，导致老年人既不愿独自操作，也不敢委托代理，最终形成对线上服务的抵触和回避。

第三，**多模态交互的适老化应用尚不成熟**。在适老化场景中，语音、触控、视觉等模态未能形成有效协同——语音输入缺乏视觉反馈确认、视觉引导缺少语音解说、触控操作无法获得多通道响应。模态间的割裂导致老年人需在不同交互方式间频繁切换，反而增加了认知负担。

针对上述三方面不足，本研究聚焦于受控响应型智能代理的设计与多模态交互的融合，探索从"自助服务"到"协助服务"的范式转变路径。

### 1.2 选题来源

#### 1.2.1 政策背景

2020 年国务院办公厅印发《关于切实解决老年人运用智能技术困难的实施方案》，明确要求推动各类移动互联网应用适老化改造。2021 年工信部发布《移动互联网应用（APP）适老化通用设计规范》，从技术标准层面对适老化设计作出规范要求。2026 年国务院办公厅印发《政务移动互联网应用程序规范化管理办法》，鼓励利用人工智能等新技术提升服务体验。广东省已发布《移动政务"适老化"用户体验设计标准》，标志着信息服务 APP 适老化设计正从实践探索走向标准化。政策层面的密集出台，为本研究提供了重要的外部驱动力。

#### 1.2.2 实地调研触发

选题直接来源于对杭州市西湖区老年用户的实地调研。研究者深入港湾家园、三坝雅苑、古荡新村三个社区，以及古荡菜市场和西湖区政务大厅两个公共服务场景，与 7 位 60 岁左右老年用户进行深度访谈，并结合参与式观察记录真实操作行为。

调研中发现的典型用户反馈是选题的直接触发点：

- "能不能像微信一样，我说查养老金它就给我查了？"
- "刷脸认证要我缓慢摇头，可怎么摇都说我超时"
- "一上来就要身份证号，我不敢填，怕是骗子"
- "我觉得自己弄肯定弄不好，还是去街道办保险"

这些反馈共同揭示了一个核心矛盾：老年用户不是不想用，而是**没有一个可信赖的人帮他们用**。这一洞察成为"受控响应型智能代理"设计概念的原始源头。

#### 1.2.3 技术时机

以大语言模型为基础的智能代理技术（2023—2025 年）恰好在此阶段成熟到可以支撑"线上工作人员"角色的技术落地——能够理解自然语言指令、拆解复杂任务、调用工具操作界面、与用户进行对话确认。Web Speech API 的浏览器原生支持使语音交互无需额外设备安装即可实现。这一技术时机使本研究的设计构想具备了切实的实现路径。

## 2 方案构思

### 2.1 设计创新点及设计源点

#### 2.1.1 核心创新：受控响应型智能代理

本研究的核心创新是提出并实现**受控响应型智能代理**这一交互模式。其定义包含两个关键词：

**"受控"** 意味着权限受控与行为受控的双重约束：
- **权限受控**：代理不持有永久授权，每次执行需要特别许可的操作（如填写身份证号、代读短信）时，必须当场向用户请求"一事一授权"，任务结束授权自动失效；
- **行为受控**：确定性操作（凡按下就会产生真实后果且难以撤销的动作，如"登录"、"提交"、"去支付"）代理永远不代为执行，必须由用户亲手完成。

**"响应型"** 意味着代理永远不主动——在用户主动唤醒之前，代理完全沉默，不说话、不弹窗、不推送。代理只在用户有需求时才介入，介入后以用户确认为前提推进。

这一设计定位与传统智能代理的"强自主性"范式形成对比：传统代理追求最大化自主完成任务；受控响应型代理追求最大化用户可信任感与控制感，将"代理能做什么"的边界设计得比技术上限更保守、更透明。

[图：受控响应型代理与传统代理的设计定位对比图]

#### 2.1.2 设计源点：线下窗口服务的数字复刻

设计源点来自老年人线下求助行为的观察。在政务大厅，当老年人遇到不会操作的环节时，窗口工作人员的协助方式是：

1. 先听清楚老人要办什么（意图理解）
2. 复述确认"您是要办医保缴费对吧？"（复述确认）
3. 引导老人跟着步骤做，必要时帮忙填写（按需协助）
4. 在提交和支付的关键节点，仍然让老人自己确认和操作（止步于确定性动作）
5. 操作完成后告知结果（结果反馈）

本研究将这一线下服务模式原样映射到线上，通过代理"小浙"实现数字化复刻。代理的交互逻辑完全按照"有耐心、有礼貌、会引导、懂边界"的线下工作人员标准设计。

#### 2.1.3 三条横切全局原则

以下三条原则贯穿所有设计决策，优先级最高，不因具体场景而妥协：

- **原则 1：代理永远不主动**——未被唤醒时完全沉默；
- **原则 2：确定性操作代理止步**——凡按下即产生不可撤销后果的按钮，代理只引导不代按；
- **原则 3：权限一事一授**——没有持久化全局授权，每个敏感操作当场问当场用。

#### 2.1.4 多模态协同策略

本研究探索语音、触控、视觉三模态的协同整合，核心策略是**冗余互补**：同一信息通过多个通道同时传递，确保任一感官通道退化不影响信息接收。

- 代理**说话**的同时**高亮**对应界面元素（语音 + 视觉同步）；
- 用户可以**说话**也可以**点击**表达意图（语音 + 触控可互换）；
- 查询结果**语音播报** + **屏幕大字显示**（听觉 + 视觉双通道）。

### 2.2 设计概念草案

#### 2.2.1 产品定位

以浙江省政务服务 APP"浙里办"为应用场景，面向老年用户，提供集成受控响应型智能代理"小浙"的适老化长辈版。通过语音对话为主的多模态交互方式，将线下"向窗口工作人员求助"的体验复现到线上，帮助老年人跨越数字鸿沟。

产品愿景：成为老年用户在政务 APP 中**可信赖的线上工作人员**——用户只需说出需求，代理就能引导、协助完成操作，同时用户始终保持最终控制权。

#### 2.2.2 代理形象概念

代理命名为"**小浙**"，取浙里办的"浙"字，名字直白易记，贴近政务服务语境。形象为**卡通化"浙"字小人图标**：简洁不复杂，橙色圆形底，白色字形小人，既有亲和感又有辨识度，不走拟人化路线以避免老年用户对真实性产生困惑。

[图：小浙形象设计稿]

#### 2.2.3 核心场景框定

研究聚焦 4 个老年人高频使用、痛点突出的核心政务服务场景：

| 场景 | 代理能力 | 设计价值 |
|---|---|---|
| 账号登录·刷脸分支 | L1（纯语音引导） | 展示代理对系统弹窗"不介入"的边界 |
| 账号登录·验证码分支 | L2（代读短信、代填验证码） | 展示跨应用操作协助与确定性止步 |
| 医保缴费 | L2-L3（代填表单、敏感字段单独授权） | 展示权限一事一授的完整流程 |
| 医保/养老金查询 | L1-L2 全链路 | 展示查询类无确定性操作的全自动路径 |

4 个场景合起来，系统地演示了受控响应型代理在不同能力等级和不同授权情境下的行为边界。

#### 2.2.4 设计概念关键词

**协助而非替代**：代理是助手，用户始终是主角；**透明而非黑箱**：每步操作前都有语音告知，操作后有记录可查；**渐进而非全权**：信任通过一次次授权自然累积，而非一开始就全盘委托；**冗余而非单一**：同一信息多通道传递，确保老年人无论习惯哪种方式都能接收。

### 2.3 用户画像

#### 2.3.1 主要用户：独自使用政务 APP 的老年人

基于实地调研，构建如下用户画像：

**基本信息**：
- 年龄：60—75 岁
- 学历：初中至高中为主
- 智能手机使用年限：3—7 年
- 设备：安卓手机为主，屏幕尺寸 5.5—6.5 英寸

**数字生活特征**：
- 会用微信语音、视频通话、收发红包；
- 会用美团、支付宝扫码支付；
- 对政务 APP 操作有困难，主要障碍在多级菜单导航、验证码输入、人脸识别；
- 遇到困难时倾向求助子女或去线下办。

**核心心理**：
- "信任人胜过信任系统"——有人帮才放心；
- 操作出错担心"弄坏"——心理负担大于技术障碍；
- 不是懒得用，而是没有信心单独用。

**典型用户故事（赵阿姨，68 岁，西湖区古荡新村）**：

> 赵阿姨退休前是工厂车间组长，智能手机使用 5 年，微信用得很熟练。每年医保缴费季，她都要让女儿帮忙在浙里办上操作，"我自己搞不来，万一填错了，那年的保险不就没了"。访谈中她表达了对语音助手的期待："要是能像跟人说话一样，说'帮我交医保'，然后有人一步步告诉我怎么做，那就好了。"

[图：主要用户画像卡片]

#### 2.3.2 次要用户：子女（监护角色）

子女作为监护角色，不直接使用代理功能，但需要了解老年父母的操作情况。本产品中，子女通过拿老人手机查看"操作记录"页面实现**被动监护**——看代理帮父母做了什么、每步是谁操作的、有没有敏感信息被填写。这一设计满足子女"放心"的诉求，同时避免引入复杂的远程监控机制。

## 3 初步设计

### 3.1 用户需求

#### 3.1.1 用户调研方法

本研究采用结合参与式观察与半结构式访谈的质性研究方法。

调研场域：杭州市西湖区港湾家园、三坝雅苑、古荡新村三个典型老年社区，以及古荡菜市场（高频数字支付场景）与西湖区政务大厅（政务服务高频场景）。样本：7 位 60 岁左右老年用户，涵盖不同年龄段、文化程度、智能手机使用年限与居住状况。

访谈围绕：日常数字生活习惯、对信息服务 APP 的认知与使用体验、操作过程中的主要障碍与需求展开，并设置任务测试环节（如查找养老金发放记录、完成医保缴费记录查询）。

#### 3.1.2 核心痛点发现

调研梳理出三个层面的痛点：

**层面一：导航入口迷失**

高频功能入口隐蔽，搜索体验欠佳。老年用户在查找养老金认证功能时，由于首页入口不明显，被迫依赖搜索栏。手写输入识别率低、语音搜索对方言支持不足，搜索功能未将高频功能进行显性前置。

**层面二：认知障碍**

- **返回逻辑混乱**：多位受访者对页面连续返回逻辑不适应，习惯性点一次返回退出流程，但部分服务需多次返回，导致误认为操作失败；
- **验证码输入流程中断率高**：几乎所有受访者在短信验证码环节遇到困难，需切换 APP 查看短信再返回填写，过程中页面可能刷新失效；
- **人脸识别容错性差**：部分功能要求完成眨眼或摇头动作，老年人动作幅度小、速度慢，常被识别为失败；
- **弹窗打断流程**：系统升级提示或跳转应用商店导致流程中断、信息丢失。

**层面三：情绪冲突**

- **缺乏操作安全网**：误触返回键即导致表单数据全部丢失，无容错机制；
- **依赖线下，缺乏引导**：系统缺乏有效的引导机制，老年人形成"去线下办才保险"的行为定势。

#### 3.1.3 需求优先级

| 需求 | 优先级 | 来源 |
|---|---|---|
| 语音对话唤起代理，代理引导完成操作 | P0 | 用户明确表达 + 痛点层面 2 |
| 代理帮忙处理验证码（代读代填） | P0 | 高频痛点，100% 受访者提及 |
| 代理帮忙找入口（跳转页面） | P0 | 痛点层面 1 |
| 操作有安全感，出错可恢复 | P0 | 痛点层面 3 |
| 关键步骤仍由用户自己完成（不失控感） | P0 | 信任心理需求 |
| 操作记录可查看（子女放心） | P1 | 次要用户需求 |
| 表单中断后可恢复（草稿箱） | P1 | 痛点层面 3 容错延伸 |

### 3.5 交互设计

#### 3.5.1 代理交互核心流程

所有场景共用同一套核心交互流程：

```
用户点击"助手"按钮
  → 代理面板从底部滑上
  → 代理语音问候"您好，有什么可以帮您？"
  → 用户按住麦克风说出需求
  → 代理复述确认（"帮您XXX，对吗？"）
  → 用户确认（语音或点击"对"）
  → 代理执行任务（跳转/高亮/代填…）
  → 涉及敏感操作时当场请求授权
  → 确定性操作前代理停止，高亮按钮提示用户亲手点击
  → 任务完成，代理语音播报结果，面板延迟关闭
```

#### 3.5.2 场景 1：登录·刷脸分支（L1 纯引导）

代理在此场景中只扮演"解说员"角色：语音引导每一步应做什么，高亮对应元素，但不帮用户点击任何按钮。

关键交互节点：
- 摄像头权限弹窗前：代理提前语音解释"马上会弹出一个使用摄像头的请求，这是为了刷脸认证，请您点同意"，消除用户对弹窗的陌生感和恐惧；
- 刷脸动作引导：代理语音引导"请您看着摄像头，慢慢眨眨眼"，用通俗语言替代系统的技术性提示；
- 登录按钮和同意条款始终由用户自己点，代理只高亮提示。

[图：场景 1 交互流程图]

#### 3.5.3 场景 2：登录·验证码分支（L2 代读代填）

代理在此场景中升级为"操作员"：代读短信验证码并代填到输入框，前提是获得用户当场授权。

关键交互节点：
- 进入验证码页面后，代理不直接代填，而是先弹出授权卡片"这次我帮您读验证码并填上去，可以吗？"；
- 用户授权后，填充动画逐位显示数字填入过程（透明而非突然出现）；
- 代理填完后主动说"验证码已经帮您填好了，您看一下对不对"，给用户核验空间；
- "登录"按钮始终由用户亲手按，代理高亮提示止步于此。

[图：场景 2 授权弹窗交互设计稿]

#### 3.5.4 场景 3：医保缴费（L2-L3，含敏感字段授权）

代理在此场景展示完整的权限一事一授流程：普通字段直接代填，敏感字段（身份证号）需单独授权，支付按钮代理永远不代按。

关键交互节点：
- 逐字段代填时，每填一个字段都有语音播报，让用户实时了解代理在做什么；
- 身份证号字段弹出专门的授权卡片，突出强调字段名称（24sp 粗体）；
- 身份证号填入后脱敏显示（如"330…**…6"），在面板上对用户可视；
- "去支付"按钮高亮后代理不发出任何进一步指令，静候用户操作。

**草稿箱机制**：若用户在填写过程中关闭面板或离开页面，已填字段自动快照保存到草稿箱。下次唤醒代理时，代理在问候后提示"您上次有个未完成的医保缴费，要继续吗？"。

[图：场景 3 完整交互流程图]

#### 3.5.5 场景 4：养老金查询（L1-L2 全链路）

此场景无确定性操作，代理可以全链路完成——从唤醒到结果播报，无需用户手动操作任何步骤。

关键交互节点：
- "查询"按钮属于非确定性操作，代理可直接代按（高亮 1 秒提示后自动触发）；
- 查询结果出来后，屏幕大字显示的同时，代理语音播报关键数字（"您的养老金本月发放了 3280 元"）；
- 双通道同步：屏幕通道与语音通道互为补充，老年人任意通道失效都能接收信息。

[图：查询结果双通道呈现示意图]

#### 3.5.6 错误处理与容错设计

| 错误情况 | 代理行为 | 设计原则 |
|---|---|---|
| 没听清（ASR 失败） | 说"不好意思没听清，您再说一遍"，最多重试 3 次 | 不让用户感到责备 |
| 连续 3 次没听清 | 提示"您也可以试试手动操作"，面板关闭 | 优雅退出，不强迫 |
| 理解意图错误 | 复述确认机制拦截：用户说"不对"则重新听取 | 执行前必有确认 |
| 网络断开 | 面板顶部红色横幅"网络已断开"，麦克风不可用 | 明示状态，不静默失败 |
| LLM 推理等待 | 显示"小浙正在想…"三点跳动动画，上限 8 秒 | 避免老年用户误判为死机 |

### 3.6 视觉设计

#### 3.6.1 长辈版色彩系统

| 用途 | 色值 | 说明 |
|---|---|---|
| 主色（按钮/高亮/代理强调） | `#FF6D00` | 橙色，温暖、亲切、有活力 |
| 代理面板底色 | `#FFFFFF` | 纯白，保持清晰 |
| 小浙气泡背景 | `#FFF0E6` | 浅橙色，区分小浙发言 |
| 用户气泡背景 | `#F5F5F5` | 浅灰，区分用户输入 |
| 文字主色 | `#1A1A1A` | 高对比，正文 |
| 文字辅助色 | `#888888` | 状态行、副标题 |
| 危险/网络断开 | `#FF3B30` | 警示红，仅用于错误状态 |
| 蒙层 | `rgba(0,0,0,0.4)` | 面板打开时上方页面遮罩 |

选择橙色作为长辈版主色的理由：橙色在色彩心理学上兼具温暖感和行动感，比冷色系更适合老年群体的情感预期；高亮橙色对比度满足 WCAG AA 级标准；与浙里办原有品牌色系有差异，便于老年用户直观识别"这是长辈版"。

[图：长辈版色彩系统色板]

#### 3.6.2 字体规范

全局最小字号 **14sp**，正文最小 **18sp**，核心交互文字 **20sp**。

| 用途 | 字号 | 字重 |
|---|---|---|
| 页面主标题 | 22sp | Bold |
| 模块标题、字段标签 | 20sp | Medium |
| 代理/用户对话气泡 | 20sp | Regular |
| 授权卡片权限名称 | 24sp | Bold |
| 状态行（"正在听您说话…"） | 16sp | Regular |
| 底部 Tab 标签 | 14sp | Regular |

字号选择依据认知负荷理论：大字号减少老年用户识别文字的认知消耗，使注意力可以集中在理解内容而非辨认文字上。

#### 3.6.3 触控目标规范

| 元素 | 最小触控区 | 说明 |
|---|---|---|
| 底部 Tab 按钮 | 64×56dp | 覆盖图标和标签整块 |
| "助手"圆形按钮 | 88dp（含外圈） | 视觉 56dp，触控区放大 |
| 面板内麦克风大按钮 | 88dp | 核心操作，最大化设计 |
| 授权弹窗"可以"/"不用了" | 120×56dp 各 | 两按钮间距 ≥16dp 防误触 |
| 关闭按钮 × | 44×44dp | 右上角，满足 HIG 最小尺寸 |

所有可交互元素触控区不小于 48dp，参考 WCAG 移动端无障碍设计规范。

### 3.8 技术可行性分析

#### 3.8.1 前端可行性

Flutter Web 已有成熟的生产级应用，场景画布 v1 已完成 11 个页面的原型实现，技术路径经过验证。GoRouter 路由体系适配代理的页面跳转指令，Widget 树的 GlobalKey 机制支持精确的元素高亮操作，AnimationController 支持所有动效需求。

Web Speech API（SpeechRecognition）是 Chrome 原生支持的浏览器 API，在项目内已通过 `wake_word_listener.dart` 验证可用，无需额外 SDK 或服务器端音频处理。HTTPS 环境下（非 localhost）Chrome 可调用麦克风权限。

IndexedDB 支持草稿箱和操作记录的本地持久化存储，前端无需账号体系即可实现数据保存与查询。

#### 3.8.2 后端与代理可行性

Agno 轻量级代理框架（Python）原生支持 DeepSeek-V3 集成，`@tool(requires_confirmation=True)` 装饰器直接对应"权限一事一授"机制，structured output（`response_model=` + Pydantic schema）保障前端收到的指令格式可靠。Agno session 机制与 WebSocket 连接生命周期天然对齐，session 关闭时授权自动失效，无状态残留风险。

DeepSeek-V3 API 中文理解与生成质量高，月均用量估计成本 < ¥10，适合毕设场景。

#### 3.8.3 语音合成可行性

Edge TTS（微软免费语音合成服务）已在项目中验证可用，支持中文语速调节（慢 15% 适合老年用户），音色自然亲切。接入方式为 HTTP 调用，后端直接获取 mp3 音频流后经 WebSocket 推送给前端播放，链路简单可靠。

#### 3.8.4 关键技术风险

| 风险项 | 等级 | 应对方案 |
|---|---|---|
| Web Speech API 在 Safari 兼容性 | 中 | 主力演示环境为 Android Chrome，iOS Safari 作备选 |
| DeepSeek API 响应延迟 | 低 | `agent_thinking` 状态反馈消除等待焦虑；备用文字输入框兜底 |
| 服务器部署 HTTPS 配置 | 低 | Let's Encrypt 免费证书 + Nginx 反向代理，流程成熟 |
| 老年用户语音识别准确率 | 中 | Prompt few-shot 调优 + 文字输入框备选 |

### 3.9 可用性研究

#### 3.9.1 测试设计

**测试对象**：5—8 位 60 岁以上老年用户，招募来源为调研社区，覆盖不同数字素养层次。

**测试方法**：任务式可用性测试 + 半结构式访谈。测试者采用"边做边说"协议，观察者记录操作行为与情绪反应。

**测试任务**（覆盖 4 个核心场景）：

| 任务 | 说明 |
|---|---|
| T1 | 唤醒小浙，让它帮你完成刷脸登录 |
| T2 | 用验证码方式登录，让小浙帮你填验证码 |
| T3 | 让小浙帮你缴今年的医保 |
| T4 | 让小浙帮你查上个月的养老金发放情况 |

**评估指标**：

| 指标 | 说明 |
|---|---|
| 任务完成率 | 在无辅助情况下独立完成任务的比例 |
| 任务完成时间 | 从唤醒代理到任务结束的时长 |
| 操作错误率 | 偏离预设路径的操作次数 / 总操作次数 |
| 用户满意度 | SUS 量表（System Usability Scale）得分 |

#### 3.9.2 预期测试结论（待实测验证）

基于设计原则推演，预期：任务完成率 ≥ 80%（老年用户在有代理协助下）；用户满意度（SUS）≥ 75 分（良好水平）；信任感评分高于市场上现有适老化 APP 参照组。

## 4 详细设计与实现

### 4.1 技术架构

#### 4.1.1 系统整体架构

系统由前端、后端代理服务、外部 API 三层构成，通过 WebSocket 长连接实时通信。

```
用户设备（Chrome 浏览器）
  └── Flutter Web 前端
        ├── 业务页面层（12 个长辈版页面）
        ├── 代理 UI 层（底部面板、麦克风、高亮覆盖层）
        ├── 本地存储层（IndexedDB：草稿箱、操作记录）
        └── WebSocket Client
              │  WebSocket 长连接
后端代理服务（FastAPI on 云服务器）
  ├── WebSocket Handler（消息路由 + 状态机）
  ├── Agno Agent（意图理解、工具调用、HITL 确认）
  └── TTS 适配器（Edge TTS → mp3）
              │  HTTPS API
外部服务
  ├── DeepSeek-V3 API（LLM 推理）
  └── Edge TTS（语音合成）
```

[图：系统架构图]

#### 4.1.2 三条原则在架构中的落地

| 原则 | 架构实现 |
|---|---|
| 代理永远不主动 | WebSocket Handler：仅在收到 `agent_wake` 消息时才创建 Agno Agent 并运行，连接建立时完全静默 |
| 确定性操作止步 | 确定性按钮（"登录"、"去支付"）**不注册为 Agno Tool**，Agent 在物理层面无法调用；白名单机制前端双重校验 |
| 权限一事一授 | 敏感操作工具（`fill_field_sensitive`、`read_sms`）使用 `@tool(requires_confirmation=True)`，调用前自动暂停，推送 `permission_request` 到前端，等待用户当场授权后继续 |

#### 4.1.3 WebSocket 消息协议

所有消息遵循统一格式：`{ "type": "...", "payload": {...}, "ts": "ISO8601" }`

核心消息类型：

| 消息类型 | 方向 | 含义 |
|---|---|---|
| `agent_wake` | 前→后 | 用户点击"助手"按钮，唤醒代理 |
| `agent_ready` | 后→前 | 代理就绪，推送问候语 + 草稿提示 |
| `asr_result` | 后→前 | 语音识别结果（文字） |
| `agent_thinking` | 后→前 | LLM 推理中，前端显示等待动画 |
| `agent_reply` | 后→前 | 代理回复（文字 + TTS 音频） |
| `user_confirm` | 前→后 | 用户确认/拒绝代理复述 |
| `cmd_navigate` | 后→前 | 指令：跳转页面 |
| `cmd_highlight` | 后→前 | 指令：高亮元素 |
| `cmd_fill_field` | 后→前 | 指令：填写字段 |
| `cmd_press_button` | 后→前 | 指令：代按非确定性按钮（查询场景） |
| `permission_request` | 后→前 | 请求用户当场授权 |
| `permission_response` | 前→后 | 用户授权结果 |
| `task_done` | 后→前 | 任务完成 |
| `agent_error` | 后→前 | 错误（ASR 失败等） |

#### 4.2.3 高亮机制实现原理

高亮机制通过以下方案实现：

1. 业务页面中每个可被代理高亮的元素使用语义化 `GlobalKey`（如 `Key('btn_go_payment')`）注册到全局 `agentElementRegistry`；
2. 收到 `cmd_highlight` 消息后，通过 `BuildContext` + `GlobalKey` 查找元素的 `RenderBox`，获取其屏幕坐标和尺寸；
3. 在屏幕全局 Overlay 上绘制高亮描边（橙色脉冲动画），覆盖于蒙层之上；
4. 蒙层（`IgnorePointer + ColorFiltered`）在高亮元素区域挖空，使其可见且可点击。

代理只知道语义 key（如 `btn_go_payment`），不知道 DOM 结构——这是前端安全边界，未注册的元素代理无法操作。

#### 4.2.4 草稿箱与操作记录

草稿箱（IndexedDB `store: drafts`）存储页面级表单快照，最多 10 条，超出删除最旧。每条草稿包含：`page_id`、`fields`（已填字段 map）、`created_at`、`updated_at`、`status`。

操作记录（IndexedDB `store: operation_logs`）保留最近 50 条，每条包含：操作场景名、一句话摘要（由代理生成）、步骤列表（区分代理操作 vs 用户操作），涉及敏感字段的步骤值替换为 `[已隐藏]`。

### 4.3 后端开发

#### 4.3.1 后端结构

后端采用 FastAPI + Agno Agent，主要文件：

| 文件 | 职责 |
|---|---|
| `main.py` | FastAPI 入口，定义 WebSocket 端点 `/ws/session/{session_id}` |
| `ws_handler.py` | WebSocket 状态机（idle / listening / confirming / executing / done），消息路由 |
| `agent_core.py` | Agno Agent 创建、工具注册、session 管理 |
| `models.py` | Pydantic 消息模型（所有 WS 消息类型的 schema） |
| `tts_adapter.py` | Edge TTS 调用，返回 base64 mp3 |
| `tools/` | 操作工具函数（navigate、highlight、fill_field、press_button、read_sms） |
| `prompts/` | 各场景 system prompt（意图分类、复述话术、超出能力应答等） |

#### 4.3.2 Agno Agent 配置

```python
# 以医保缴费场景为例（L2-L3）
agent = Agent(
    model=DeepSeek(id="deepseek-chat"),
    tools=[
        cmd_navigate,
        cmd_highlight,
        fill_field_normal,            # 普通字段代填
        fill_field_sensitive,          # 敏感字段代填（HITL）
    ],
    session_id=session_id,
    response_model=AgentInstruction,   # Pydantic schema，保证结构化输出
    system_prompt=prompts["yibao_jiaofei"],
)
```

`fill_field_sensitive` 使用 `@tool(requires_confirmation=True)` 装饰，调用时 Agno 自动触发 HITL 暂停，ws_handler 此时推送 `permission_request` 到前端，等待 `permission_response` 后恢复执行。

确定性按钮操作永远不注册为工具，物理上杜绝被调用的可能。

#### 4.3.3 WebSocket 状态机

```
idle
  → [收到 agent_wake] → listening
  → [asr_result is_final] → confirming
  → [user_confirm yes] → executing （Agno Agent.run() 在此状态运行）
  → [task_done / out_of_scope] → done → idle

HITL pause（在 executing 状态内）：
  收到 requires_confirmation → 推送 permission_request → 等待 permission_response
  → granted=true：恢复 Agno 执行
  → granted=false：代理语音告知，跳过该步骤
```

### 4.4 API 集成

#### 4.4.1 DeepSeek-V3 集成

通过 Agno 的 `DeepSeek` model 类接入，API key 存储在 `.env` 文件中（不入版本库）。使用 `deepseek-chat` 模型（DeepSeek-V3），具备强中文理解能力，接口响应时间通常在 1—3 秒，适合实时对话场景。

意图分类 prompt 通过 few-shot 示例覆盖每个场景的 3—5 种口语表达变体，确保"缴医保"、"交医保"、"帮我缴医保"等变体都能正确分类到 `yibao_jiaofei` 场景。

#### 4.4.2 Edge TTS 集成

Edge TTS 通过 Python 的 `edge-tts` 包调用，配置中文女声音色（`zh-CN-XiaoxiaoNeural`），语速设置为 `-15%`（适合老年用户），输出 mp3 格式。

TTS 生成为异步操作，在代理生成回复文本的同时并行触发 TTS，减少用户等待时间。生成后音频 base64 编码嵌入 `agent_reply` 消息推送给前端。

#### 4.4.3 Web Speech API 集成（前端语音识别）

```dart
class SpeechRecognizer {
  html.SpeechRecognition? _recognition;
  void Function(String text)? onResult;

  void start() {
    _recognition = html.SpeechRecognition()
      ..lang = 'zh-CN'
      ..interimResults = false
      ..continuous = false;
    _recognition!.onResult.listen((e) {
      final text = e.results?.item(0)?.item(0)?.transcript ?? '';
      if (text.isNotEmpty) onResult?.call(text);
    });
    _recognition!.start();
  }

  void stop() => _recognition?.stop();
}
```

识别结果通过 `text_input` 消息类型发送到后端（替代 `audio_chunk` 方案），后端无需 ASR 服务，架构大幅简化。

#### 4.5.1 测试策略

测试分三层：

**单元/组件测试**：对核心 Flutter Widget（AgentPanel、PermissionCard）和后端工具函数进行单元测试，覆盖正常路径和边界条件（如 `permission_response granted=false` 的处理）。

**真机测试**：在 Android 手机 Chrome 上运行所有 4 个核心场景，验证 Web Speech API 语音识别准确率（目标 ≥ 80%），以及高亮动效、TTS 播放、面板动画的实际体验。

#### 4.5.2 部署方案

- **服务器**：云服务器（2C2G，Ubuntu 22.04 LTS，上海/杭州节点）；
- **后端服务**：`uvicorn main:app --host 127.0.0.1 --port 8000`，由 systemd 管理；
- **前端静态文件**：Flutter Web build 后上传到服务器，由 Nginx 托管；
- **HTTPS**：Let's Encrypt 免费证书（需域名）或 mkcert 自签证书（仅 Android Chrome 可信任）；
- **Nginx 配置**：静态文件托管 + `/ws/session/` 路径 WebSocket 反向代理（含 `wss://` 升级头）。

HTTPS 是 Web Speech API 在非 localhost 环境下获取麦克风权限的必要条件。

## 6 结论

### 6.1 主要研究贡献

本研究从理论与实践两个维度做出了如下贡献：

**理论层面**，提出**受控响应型智能代理**这一交互模式，系统回应了老年用户"信任人胜过信任系统"的深层心理需求。该模式通过三条横切原则（代理永远不主动、确定性操作止步、权限一事一授）在技术层面将"边界"固化，而非依赖对话逻辑的运行时判断，使代理行为可预测、可审查、可信任。这为人机交互领域的适老化研究提供了"协助服务"范式的理论框架。

**实践层面**，基于受控响应型代理实现了一款可运行的浙里办长辈版高保真原型，覆盖 4 个政务服务核心场景，实现了语音意图理解、跨页面导航、多模态字段代填、敏感操作授权确认、查询结果双通道呈现等完整功能链路。原型可在手机 Chrome 浏览器上直接访问体验，技术栈全部为开源或免费方案，具备推广复制价值。

### 6.2 多模态协同的设计价值

本研究通过语音、触控、视觉三模态协同的冗余互补策略，实证了多模态融合对老年用户体验的提升路径：不是简单叠加多个通道，而是建立通道间的时序对齐和语义融合——代理说的同时高亮对应元素，结果语音播报的同时屏幕大字显示，用户既可说话也可点击同等完成任务。这种"通道互为兜底"的策略有效应对了老年用户感官功能的个体差异性。

### 6.3 研究局限与未来展望

本研究存在以下局限：第一，可用性测试样本较小（5—8 人），结论的代表性有限，需要后续更大规模的定量研究验证；第二，原型聚焦 4 个场景，能力范围受限，未能覆盖更广泛的政务服务场景；第三，方言识别尚未实现，制约了对普通话能力有限的老年用户的适用性。

未来研究可在以下方向延伸：扩展代理协助场景（公积金查询、异地就医备案等）；接入方言 ASR 模型；研究"常做任务免问"的偏好记忆机制如何在不牺牲用户控制感的前提下减少重复授权；探索受控响应型代理模式在医疗、金融等其他信息服务场景的迁移适用性。
