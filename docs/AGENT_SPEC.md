## 1 项目背景

## 3 横切全局原则

以下三条原则对所有设计决策生效，优先级最高。

### 原则 1：代理永远不主动

代理在用户主动唤醒前，**完全沉默**——不说话、不出现、不弹窗、不响铃。

- 代理**未被唤醒时** → 完全沉默
- 代理**被唤醒后** → 正常应答、复述、确认、执行、反馈（属于用户主动开场后的对话，不违反原则）
- 任何系统层面的提醒需求（草稿、操作记录等）→ 统一走**静态按钮入口**，用户看得到、够得着、想用才点

**派生纪律**：未来任何"代理应该主动告诉用户什么"的设计提议，默认答案是"不"；改用"静态按钮入口 + 用户点击查看"。

### 原则 2："确定性操作"代理止步

**定义**：确定性操作 = 一旦按下就会**对外产生真实结果、不可轻易撤销**的动作。

典型例子：登录、提交表单、去支付、同意条款、确认修改、删除、发送申请。

**规则**：代理可以引导、填充、准备前置条件，但**按下那一下必须由用户亲手完成**。

**推论**：
- 横跨所有 L 等级——哪怕 L3 也不能替用户按确定性按钮
- 查询类按钮（"查询"、"搜索"、"下一步"等无对外后果的操作）→ 不是确定性操作，代理可按
- 确定性按钮（"登录"、"提交"、"去支付"、"同意"等）→ 代理永远不按

### 原则 3：权限一事一授

**定义**：代理权限不是持久化的全局等级，而是**任务级的当场授权**。

**工作方式**：
1. 用户发起任务（"帮我登录" / "帮我查养老金"等）
2. 代理在对话中遇到需要升级能力的点**当场问一句**（如"这次我帮您填验证码可以吗"）
3. 用户当场回答，授权**只对当前任务生效**
4. 任务结束，授权自动失效
5. 下次同类任务重新问一次

**推论**：
- 用户不会看到"权限等级"这个前台概念——只会听到"这次要不要我帮您做 X"
- L1/L2/L3 能力矩阵在代码中存在，用途为"能力敏感度分类"——决定什么时候该停下来问
- 信任推进由用户一次次说"可以"自然累积

## 4 核心流程

用户与代理对话 → 代理复述确认 → 用户确认 → 代理自动跳转对应页面 → 引导 + 操作完成易失误环节。

## 5 能力矩阵（L1 / L2 / L3）

| 能力 | L1 | L2 | L3 |
|---|:-:|:-:|:-:|
| 语音对话、理解意图 | ✓ | ✓ | ✓ |
| 复述确认（执行前必做） | ✓ | ✓ | ✓ |
| 页面跳转（帮用户找入口） | ✓ | ✓ | ✓ |
| 语音引导用户操作（"请您点这个绿色按钮"） | ✓ | ✓ | ✓ |
| 屏幕高亮目标元素 | ✓ | ✓ | ✓ |
| 代读外部内容（短信、屏幕文字） | ✗ | ✓ | ✓ |
| 代填非敏感字段（查询条件、缴费金额、手机号、地址等） | ✗ | ✓ | ✓ |
| 代填敏感字段（身份证号、密码、银行卡号） | ✗ | ✗ | ✓ |
| 按确定性按钮（登录/提交/去支付等） | ✗ | ✗ | ✗ |

**记忆口诀**：
- **L1 = "只用嘴和眼"**（代理只说话、只高亮；手始终是用户的）
- **L2 = "嘴眼手，但手只碰普通信息"**
- **L3 = "嘴眼手，手可以碰敏感字段，但不能按 Send"**

### 5.1 敏感字段白名单（硬编码）

"敏感"由硬编码白名单定义，不由 LLM 判断：
- 身份证号
- 密码
- 银行卡号

其余（手机号、地址、验证码、查询条件等）一律算非敏感。如后续需扩展，直接加进白名单。

## 6 业务场景

### 6.1 聚焦 4 个核心场景

| # | 场景 | 代理能力上限 | 代理止损点 |
|---|---|---|---|
| 1 | 账号登录 · 刷脸分支 | L1（语音引导） | 摄像头权限弹窗前语音解释；刷脸动作（眨眼/摇头）永远用户自己做 |
| 2 | 账号登录 · 验证码分支 | L2（代读短信 + 代填验证码） | 填完后停在"登录"按钮前，用户亲手按 |
| 3 | 医保缴费 | L2–L3（代填表单、代确认缴费对象） | 止步于"去支付"按钮前，用户亲手按；后续支付链路不碰 |
| 4 | 医保查询 / 养老金查询 | L1–L2 全链路 | 纯查询无确定性操作，代理可一气到底 |

**学术价值**：刷脸分支展示"纯 L1 多模态引导"；验证码分支展示"L2 代填 + 确定性按钮不代做"——两条合起来把受控响应型在两种输入模态下的分级控制演示到位。

### 6.2 不做的内容

- 4 场景之外的浙里办页面（代理答"这个我不会"）
- 支付环节（明确退出代理能力范围）
- 纯生物识别动作（刷脸、指纹等肉身动作）

### 6.3 超出能力时的应答

| 情景 | 代理反应 |
|---|---|
| 用户问的事浙里办根本没有（订火车票、看天气） | 明说"浙里办没有这个服务" |
| 用户问的事浙里办有但代理不会（改头像等） | 指路 + 坦白"这个功能在这里，我暂时帮不了您，您可以自己试试" |

## 7 代理 UI 形态

> 本节为倾向性描述，视觉细节留待原型绘制阶段锁定。

### 7.1 代理面板

底部滑上来一块对话区，占屏幕下半部分，内含麦克风图标和"正在听您说话"的视觉反馈。
- 类似微信"语音输入条"的形态
- 不是右下角小气泡，不是全屏接管
- 长辈版页面继续保留在上半部分（用户仍可看见当前场景）

### 7.2 唤醒与消失

- **唤醒**：用户主动点"助手"按钮
- **热词唤醒**：留待后续决定（倾向默认关闭）
- **消失时机**：留待原型阶段定（候选：任务完成后自动收起 / 用户下拉关闭 / 用户点空白处关闭）

### 7.3 并行操作

当前假设：**不锁屏幕**。上半部分长辈版页面用户仍可操作，下半部分对话区是代理的"地盘"。具体交互在原型阶段验证。

## 8 三项机制

### 8.1 容错 · 草稿箱

**保存粒度**：页面级——存表单中所有已完成字段的快照。不存半截字段，不存对话历史。

**适用范围**：主要服务于医保缴费等有表单的流程。查询类场景中断后重查成本极低，不需要草稿。

**找回方式**：
- 界面上常驻"草稿箱 (N)"按钮，用户自己点进去恢复
- 用户唤醒代理时，代理在问候语后顺带提一句"您上次有个未完成的缴费，要继续吗？"（不违反原则 1——是被唤醒后的正常应答）

### 8.2 信任分级推进

不存在用户可见的"权限等级"概念。信任推进通过"权限一事一授"（原则 3）自然实现——用户一次次授权的累积。

### 8.3 操作记录（子女视角）

**查看方式**：子女拿到老人的手机，在 APP 内点进"操作记录"页面直接查看。
- 不涉及多账号、远程查看、扫码共享
- 不需要独立子女端

**权限**：只看记录，不能介入（纯被动监护）。

**知情机制**：不需要额外告知。递手机本身就是知情同意。

## 9 错误恢复

### 9.1 代理理解错意图

由 B3 复述确认机制兜底：代理执行任何操作前先复述"我帮您查 XX，对吗？"，用户确认后才动。理解错的指令在执行前即被拦截。

### 9.2 代理执行中填错字段

不需要专门的撤销机制。处理方式：
- 用户手动点击填错的字段自己修正
- 或语音告诉代理"那个年度选错了，改成 2026"，代理重新填

### 9.3 代理没听清

- 代理说"不好意思没听清，您再说一遍"
- 连续 3 次仍听不清 → 提示"您也可以试试手动操作"，退出语音识别等待

### 9.4 网络断开

- 界面显示"网络已断开"提示
- 代理停住，不做任何操作
- 网络恢复后用户重新唤醒代理继续
- 不做自动重连、不做离线缓存
