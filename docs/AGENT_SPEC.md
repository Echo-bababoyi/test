## 1 项目背景

### 1.1 学术定位

本科毕业设计：**《信息服务 APP 的适老化设计与多模态交互》**。

### 1.2 要解决的问题

老年用户使用信息服务 APP 的**交互问题**与**数字鸿沟**。

### 1.3 现有方案不足

1. 适老化改造只停留在页面层，未触及交互逻辑
2. APP 内置 AI 助手能力局限，只给文字建议
3. 老年人最终仍倾向线下求助工作人员

### 1.4 设计定位

以**受控响应型智能代理**为核心，围绕它实现适老化设计与多模态交互。

### 1.5 能力隐喻

**模拟线下客服**——复刻老年人向窗口工作人员求助的场景。

## 3 横切全局原则

以下三条原则对所有设计决策生效，优先级最高。

### 原则 1：代理永远不主动

代理在用户主动唤醒前，**完全沉默**——不说话、不出现、不弹窗、不响铃。

- 代理**未被唤醒时** → 完全沉默
- 代理**被唤醒后** → 正常应答、复述、确认、执行、反馈（属于用户主动开场后的对话，不违反原则）
- 任何系统层面的提醒需求（草稿、操作记录等）→ 统一走**静态按钮入口**，用户看得到、够得着、想用才点

**派生纪律**：未来任何"代理应该主动告诉用户什么"的设计提议，默认答案是"不"；改用"静态按钮入口 + 用户点击查看"。

### 原则 2："确定性操作"代理止步

**定义**：确定性操作 = 一旦按下就会**对外产生真实结果、不可轻易撤销**的动作。

典型例子：登录、提交表单、去支付、同意条款、确认修改、删除、发送申请。

**规则**：代理可以引导、填充、准备前置条件，但**按下那一下必须由用户亲手完成**。

**推论**：
- 横跨所有 L 等级——哪怕 L3 也不能替用户按确定性按钮
- 查询类按钮（"查询"、"搜索"、"下一步"等无对外后果的操作）→ 不是确定性操作，代理可按
- 确定性按钮（"登录"、"提交"、"去支付"、"同意"等）→ 代理永远不按

### 原则 3：三级信任模型 + 任务级一次性授权（v1.1 更新）

**三级信任模型**：用户在登录后可显式选择代理的工作方式：

| 等级 | 前台名称 | 工作方式 |
|---|---|---|
| 引导级（guide，**默认**） | 我自己做，小浙提醒我 | 代理只语音提示 + 高亮元素，不代操作任何字段或按钮，不代跳页面 |
| 半委托级（semi） | 小浙帮我填，我自己点提交 | 代理代填普通字段 + 代跳页面；敏感字段逐次弹卡授权 |
| 全委托级（full） | 小浙全程代办，关键步骤我确认 | 代理可填普通 + 敏感字段；首次遇敏感字段弹一次总授权卡 |

**默认保守原则**：未选择或关闭弹卡 = 引导级；未登录 = 强制引导级（忽略设置值）。

**任务级一次性授权（全委托专属）**：全委托级下，代理在任务内首次遇到敏感操作前弹一次总授权卡。用户同意后，本次任务内不再弹卡；任务结束自动失效，下次同类任务重新询问。用户拒绝 = 该字段跳过，任务不取消，下一个敏感字段再次弹卡。

**最终能力 = min（用户选级，登录态联动级，场景固有上限）**：三道约束同时生效，取最保守值。

**两条硬性底线（任何级别均不可逾越）**：
- **密码类字段永不代填**：登录密码、支付密码及字段名含 `password`/`pwd` 的字段，代理直接语音提示用户自填，不走授权流程
- **确定性按钮永不代按**：同原则 2，跨所有等级生效

## 4 核心流程

用户与代理对话 → 代理复述确认 → 用户确认 → 代理自动跳转对应页面 → 引导 + 操作完成易失误环节。

## 5 能力矩阵（L1 / L2 / L3）

| 能力 | 引导级 L1（默认） | 半委托级 L2 | 全委托级 L3 |
|---|:-:|:-:|:-:|
| 语音对话 / 理解意图 | ✓ | ✓ | ✓ |
| 复述确认（执行前必做） | ✓ | ✓ | ✓ |
| 语音提示 cmd_say | ✓ | ✓ | ✓ |
| 元素高亮 cmd_highlight | ✓ | ✓ | ✓ |
| 页面跳转 cmd_navigate | **✗** | ✓ | ✓ |
| 代填普通字段（查询条件、手机号、地址等） | ✗ | ✓ | ✓ |
| 代按非确定性按钮（"查询"/"下一步"等） | ✗ | ✓ | ✓ |
| 代读短信 read_sms | ✗ | ✓（逐次授权） | ✓（一次总授权） |
| 代填敏感字段（身份证号、银行卡号等） | ✗ | ✓（逐次授权） | ✓（一次总授权） |
| 代填密码类字段 | ✗ | ✗ | ✗ **硬底线** |
| 代按确定性按钮（登录/提交/支付） | ✗ | ✗ | ✗ **硬底线** |

**记忆口诀**：
- **L1 = "只用嘴和眼"**（只语音提示 + 高亮；不跳页、不填表、不按钮——手始终是用户的）
- **L2 = "嘴眼手，手只碰普通信息；敏感信息每次问"**
- **L3 = "嘴眼手，敏感信息任务内一次授权；密码和提交永远不碰"**

### 5.1 字段分类（硬编码）

分类由硬编码白名单定义，不由 LLM 判断：

**普通字段**（代理在 L2/L3 直接填写，无需授权）：手机号、地址、查询条件、缴费金额等

**敏感字段**（需授权才能代填，L2 逐次 / L3 一次总授权）：身份证号、银行卡号、短信验证码等

**密码字段**（任何级别均不代填，硬底线）：登录密码、支付密码，以及字段名含 `password`/`pwd` 的字段

如需调整分类，直接修改代码白名单，不由 LLM 判断。

### 5.2 三级权限矩阵 vs 原 L1/L2/L3 能力矩阵对照（v1.1 新增）

| 信任级别 | 对应原矩阵 | 与 v1.0 的主要差异 |
|---|---|---|
| 引导级 guide（默认） | L1 | **v0.6 移除 cmd_navigate**：引导级不再代跳页面，仅语音提示 + 高亮 |
| 半委托级 semi | L2 | 新增 cmd_navigate + fill_field_normal + cmd_press_button；敏感字段逐次授权 |
| 全委托级 full | L3 | 同 semi + 敏感字段一次总授权；密码字段仍为硬底线 |

> **v0.6 核心变更**：引导级的"L1"原含页面跳转能力，v0.6 移除该能力——引导级代理仅能语音提示 + 高亮元素，不替用户跳转任何页面。

## 6 业务场景

### 6.1 聚焦 4 个核心场景

| # | 场景 | 代理能力上限 | 代理止损点 |
|---|---|---|---|
| 1 | 账号登录 · 刷脸分支 | **L1**（场景固有上限；语音提示 + 高亮） | 摄像头权限弹窗前语音解释；刷脸动作（眨眼/摇头）永远用户自己做 |
| 2 | 账号登录 · 验证码分支 | **L1**（场景固有上限；引导用户自查短信 + 自填验证码） | 代理不代填任何字段，用户亲手完成所有输入 |
| 3 | 医保缴费 | L2–L3（代填表单、代确认缴费对象） | 止步于"去支付"按钮前，用户亲手按；后续支付链路不碰 |
| 4 | 医保查询 / 养老金查询 | L1–L2 全链路 | 纯查询无确定性操作，代理可一气到底 |

**学术价值**：**登录场景统一限定为 L1 引导级**，体现敏感入口的最高保守态势；**医保缴费场景**承担 L2/L3 切换演示职责（普通字段直填 + 敏感字段授权 + 不可逆操作止步）。

**三道 AND 机制（场景固有上限）**：最终能力 = min(用户选级, 登录态联动级, 场景固有上限)。login_face / login_verify 场景固有上限为 L1，无论用户在设置页选何级别，代理在登录场景均只能语音提示 + 元素高亮，**不向用户展示"被降级"提示**，UI 表现与用户主动选 L1 一致。

### 6.2 不做的内容

- 4 场景之外的浙里办页面（代理答"这个我不会"）
- 支付环节（明确退出代理能力范围）
- 纯生物识别动作（刷脸、指纹等肉身动作）

### 6.3 超出能力时的应答

| 情景 | 代理反应 |
|---|---|
| 用户问的事浙里办根本没有（订火车票、看天气） | 明说"浙里办没有这个服务" |
| 用户问的事浙里办有但代理不会（改头像等） | 指路 + 坦白"这个功能在这里，我暂时帮不了您，您可以自己试试" |

## 7 代理 UI 形态

> 本节为倾向性描述，视觉细节留待原型绘制阶段锁定。

### 7.1 代理面板

底部滑上来一块对话区，占屏幕下半部分，内含麦克风图标和"正在听您说话"的视觉反馈。
- 类似微信"语音输入条"的形态
- 不是右下角小气泡，不是全屏接管
- 长辈版页面继续保留在上半部分（用户仍可看见当前场景）

### 7.2 唤醒与消失

- **唤醒**：用户主动点"助手"按钮
- **热词唤醒**：留待后续决定（倾向默认关闭）
- **消失时机**：留待原型阶段定（候选：任务完成后自动收起 / 用户下拉关闭 / 用户点空白处关闭）

### 7.3 并行操作

当前假设：**不锁屏幕**。上半部分长辈版页面用户仍可操作，下半部分对话区是代理的"地盘"。具体交互在原型阶段验证。

### 7.4 登录后首次主动询问机制（v1.1 新增）

用户**首次**登录成功跳转长辈首页（`/elder`）时，系统自动弹出三级权限选择卡（底部上滑 BottomSheet，不可点空白关闭）。

**触发条件**：`xiaozhe_first_choice_shown != "true"` 且当前已登录（仅触发一次，重新登录不重复弹）

**交互规则**：
- 展示三张权限卡，默认三卡均未选中（用户主动选择，不强制默认某级）
- 用户选中某张 → 立即生效 + 关闭弹卡 + Toast「已设为 XX」
- 用户点「✕」关闭不选 → 落地引导级（默认），不再弹，无 Toast

**不违反原则 1 的原因**：登录成功是用户主动触发的状态转换，弹卡是该转换的配套询问，不属于"主动挑事"。

**设置页随时调整**：
- 升级（L1→L2 / L1→L3 / L2→L3）：弹二次确认 dialog，取消回滚
- 降级（任意→更低）：直接生效，无需确认
- 未登录态：三卡灰显（opacity 0.4），顶部提示「登录后可调整小浙的工作方式」+ 去登录按钮

## 8 三项机制

### 8.1 容错 · 草稿箱

**保存粒度**：页面级——存表单中所有已完成字段的快照。不存半截字段，不存对话历史。

**适用范围**：主要服务于医保缴费等有表单的流程。查询类场景中断后重查成本极低，不需要草稿。

**找回方式**：
- 界面上常驻"草稿箱 (N)"按钮，用户自己点进去恢复
- 用户唤醒代理时，代理在问候语后顺带提一句"您上次有个未完成的缴费，要继续吗？"（不违反原则 1——是被唤醒后的正常应答）

### 8.2 信任分级推进

**v1.1 更新**：引入用户可见的三级信任模型（详见原则 3 与 §5.2）。用户登录后通过首次弹卡主动选择信任级别，可在设置页随时调整。

任务级一次性授权保留作为**全委托级的子机制**：全委托级下，任务内首次遇到敏感字段弹一次总授权卡，同意后本任务内免弹；任务结束失效，下次同类任务重新建立授权。这延续了"授权不持久化、每次任务重新建立信任"的原始精神。

### 8.3 操作记录（子女视角）

**查看方式**：子女拿到老人的手机，在 APP 内点进"操作记录"页面直接查看。
- 不涉及多账号、远程查看、扫码共享
- 不需要独立子女端

**权限**：只看记录，不能介入（纯被动监护）。

**知情机制**：不需要额外告知。递手机本身就是知情同意。

## 9 错误恢复

### 9.1 代理理解错意图

由 B3 复述确认机制兜底：代理执行任何操作前先复述"我帮您查 XX，对吗？"，用户确认后才动。理解错的指令在执行前即被拦截。

### 9.2 代理执行中填错字段

不需要专门的撤销机制。处理方式：
- 用户手动点击填错的字段自己修正
- 或语音告诉代理"那个年度选错了，改成 2026"，代理重新填

### 9.3 代理没听清

- 代理说"不好意思没听清，您再说一遍"
- 连续 3 次仍听不清 → 提示"您也可以试试手动操作"，退出语音识别等待

### 9.4 网络断开

- 界面显示"网络已断开"提示
- 代理停住，不做任何操作
- 网络恢复后用户重新唤醒代理继续
- 不做自动重连、不做离线缓存
