# 产品需求文档（PRD）

> **版本**：v1.0（2026-04-28）
> **产品名称**：浙里办长辈版 · 智能代理"小浙"
> **关联文档**：`docs/AGENT_SPEC.md`（代理设计规范）、`docs/信息服务APP的适老化设计与多模态交互_开题报告初稿.txt`（开题报告）

## 1 产品概述

### 1.1 产品定位

以浙江省政务服务 APP"浙里办"为应用场景，面向老年用户，提供集成**受控响应型智能代理"小浙"**的适老化长辈版。通过语音对话为主的多模态交互方式，将线下"向窗口工作人员求助"的体验复现到线上，帮助老年人跨越数字鸿沟。

### 1.2 要解决的核心问题

老年用户使用信息服务 APP 时面临**交互问题**与**数字鸿沟**：

1. **表层改造与深层需求脱节**——现有适老化改造停留在放大字体、简化首页，未触及交互逻辑、容错机制和情感化设计
2. **信任与控制的双重缺失**——老年人"信任人胜过信任系统"，既不愿独自操作，也不敢委托传统 AI 助手（担心失控）
3. **多模态交互的适老化应用不成熟**——语音、触控、视觉等模态之间割裂，未形成有效协同

### 1.3 产品愿景

成为老年用户在政务 APP 中**可信赖的线上工作人员**——用户只需说出需求，代理就能引导、协助完成操作，同时用户始终保持最终控制权。

## 2 目标用户

### 2.1 用户画像

| 维度 | 描述 |
|---|---|
| **年龄** | 60 岁及以上 |
| **数字素养** | 会基本使用智能手机（微信、电话），但对复杂 APP 操作存在困难 |
| **核心心理** | 信任人胜过信任系统；不是不愿线上办事，而是不想在线上自己冒险办事 |
| **典型行为** | 倾向线下求助工作人员或子女；操作中遇到障碍容易放弃 |
| **生理特征** | 可能存在视力下降、听力减退、手指灵活度降低、记忆力减退等 |

### 2.2 用户调研核心发现

来源：开题报告中对杭州市西湖区 3 个社区（港湾家园、三坝雅苑、古荡新村）7 位老年用户的深度访谈 + 古荡菜市场与政务大厅参与式观察。

**三层痛点**：

| 层面 | 痛点 |
|---|---|
| **导航入口迷失** | 高频功能入口隐蔽、搜索体验差（手写识别率低、语音对方言支持不足） |
| **认知障碍** | 返回逻辑混乱、验证码切换 APP 中断率高、人脸识别容错性差、弹窗打断流程 |
| **情绪冲突** | 误操作导致数据丢失无安全网、依赖线下帮助缺乏系统引导 |

**典型用户反馈**：
- "能不能像微信一样，我说查养老金它就给我查了？"
- "刷脸认证要我缓慢摇头，可怎么摇都说我超时"
- "一上来就要身份证号，我不敢填，怕是骗子"
- "我觉得自己弄肯定弄不好，还是去街道办保险"

### 2.3 次要用户：子女

子女作为监护角色，需要了解老年父母的操作情况。在本产品中，子女通过拿老人手机查看"操作记录"页面实现被动监护。

### 3.2 核心场景用户故事

#### 场景 1：账号登录 · 刷脸分支

> **作为**一位不熟悉刷脸认证的老年用户，
> **我想要**有人告诉我每一步该怎么做，
> **以便**我能顺利完成刷脸登录，不用去找子女帮忙。

**代理能力**：L1（纯语音引导 + 视觉高亮）

**流程概要**：
1. 用户唤醒小浙，说"帮我登录"
2. 小浙复述确认"帮您登录，对吗？"
3. 用户确认后，小浙跳转到登录页面
4. 小浙语音引导 + 高亮"同意条款"按钮 → 用户点击
5. 小浙语音解释摄像头权限请求 → 用户授权
6. 小浙语音引导刷脸动作（"请您对着摄像头眨眨眼"）→ 用户自己完成
7. 登录成功，小浙语音播报"登录成功了"

**验收标准**：
- 代理全程不代替用户做任何操作，只引导和高亮
- 刷脸动作由用户自己完成
- 摄像头权限弹窗前有语音解释

#### 场景 2：账号登录 · 验证码分支

> **作为**一位看不清短信验证码、不会切换 APP 的老年用户，
> **我想要**有人帮我把验证码填进去，
> **以便**我不用来回切换手机页面。

**代理能力**：L2（代读短信 + 代填验证码）

**流程概要**：
1. 用户唤醒小浙，说"帮我登录"/ 在刷脸页点"其他方式认证"
2. 小浙跳转到验证码页面
3. 小浙询问"这次我帮您读验证码并填上去，可以吗？"（一事一授）
4. 用户同意后，小浙代读短信中的验证码并代填到输入框
5. 小浙语音播报"验证码已经填好了，您看一下对不对"
6. 小浙高亮"登录"按钮 → **用户亲手按**（确定性操作）
7. 登录成功，小浙语音播报确认

**验收标准**：
- 代理代读验证码前征得用户授权
- 代理填完后主动让用户确认
- "登录"按钮由用户亲手按，代理不代按

#### 场景 3：医保缴费

> **作为**一位需要缴纳医保但不会在手机上操作的老年用户，
> **我想要**有人帮我找到缴费入口、帮我填好表单，
> **以便**我不用跑去社区街道排队缴费。

**代理能力**：L2–L3（代填表单字段，敏感字段需单独授权）

**流程概要**：
1. 用户唤醒小浙，说"帮我缴医保"
2. 小浙复述确认 → 用户确认
3. 小浙跳转到医保缴费页面
4. 小浙逐个字段询问并代填（缴费对象、缴费年度、金额等）
5. 若需填身份证号（敏感字段）→ 小浙额外问"这次我帮您填身份证号可以吗？"
6. 所有字段填完后，小浙语音播报"表单已经填好了，请您确认后点'去支付'"
7. 小浙高亮"去支付"按钮 → **用户亲手按**（确定性操作）
8. 后续支付链路（支付宝/微信/输密码）代理不介入

**验收标准**：
- 敏感字段代填前有单独授权
- 代理在"去支付"按钮前止步
- 表单中断后可从草稿箱恢复

#### 场景 4：医保查询 / 养老金查询

> **作为**一位想查自己养老金发了多少的老年用户，
> **我想要**直接说一句话就能看到结果，
> **以便**我不用在各种菜单里翻找入口。

**代理能力**：L1–L2 全链路（纯查询无确定性操作）

**流程概要**：
1. 用户唤醒小浙，说"帮我查养老金"
2. 小浙复述确认 → 用户确认
3. 小浙跳转到养老金查询页面
4. 小浙代填查询条件（如有）→ 代按"查询"按钮（非确定性操作）
5. 结果出来后，小浙语音播报"您的养老金本月发放了 XXX 元"
6. 任务完成

**验收标准**：
- 代理可全链路完成（从唤醒到结果播报），无需用户手动操作
- 查询按钮代理可代按（非确定性操作）
- 结果以语音播报 + 屏幕显示双通道呈现

## 4 非功能需求

### 4.1 适老化设计要求

| 维度 | 要求 |
|---|---|
| **视觉** | 大字体（≥18sp）、高对比度、大触控目标（≥48dp）、长辈版主色橙色 `#FF6D00` |
| **语音** | 语速偏慢、用词通俗、避免专业术语、关键信息重复确认 |
| **导航** | 扁平化层级、高频功能前置、返回逻辑一致 |
| **容错** | 误操作可恢复（草稿箱）、代理填错可手动改或语音纠正 |
| **情感** | 代理语气亲切耐心、不催促、不流露不耐烦 |

### 4.2 多模态协同策略

同一信息通过多个通道同时传递（冗余互补）：
- 代理**说**的同时**高亮**对应元素（语音 + 视觉）
- 用户可以**说**也可以**点**（语音 + 触控）
- 结果**语音播报** + **屏幕显示**（听觉 + 视觉）

### 4.3 技术约束

| 约束 | 说明 |
|---|---|
| **前端** | Flutter Web（已有技术基础） |
| **代理后端** | LLM 驱动的智能代理服务（具体框架待技术设计阶段确定） |
| **语音** | 需接入 ASR（语音识别）+ TTS（语音合成）服务 |
| **通信** | 前后端通过 WebSocket 实时通信 |

## 5 代理行为规范（摘要）

以下为 `docs/AGENT_SPEC.md` 中定义的核心规则，PRD 层面不再重复完整内容：

- **三条横切原则**：代理永远不主动 / 确定性操作代理止步 / 权限一事一授
- **能力矩阵**：L1（嘴和眼）/ L2（嘴眼手，不碰敏感字段）/ L3（嘴眼手，可碰敏感字段，不按确定性按钮）
- **敏感字段白名单**：身份证号、密码、银行卡号（硬编码，不由 LLM 判断）
- **错误恢复**：复述确认兜底 + 手动/语音改错 + 网络断开提示

完整规则见 `docs/AGENT_SPEC.md`。

## 6 产品边界

### 6.1 明确不做

| 内容 | 原因 |
|---|---|
| 支付环节（支付宝/微信/输密码） | 超出代理能力范围，涉及第三方支付安全 |
| 生物识别代做（替用户刷脸/按指纹） | 肉身动作，技术上也无法代做 |
| 4 场景之外的政务服务 | 毕设范围聚焦，代理会回答"这个我不会" |
| 多账号/远程子女查看 | 简化为子女拿手机直接看 |
| 自动重连/离线缓存 | 毕设原型不需要生产级可靠性 |
| 方言识别 | 技术复杂度高，留待未来 |

### 6.2 未来可扩展

| 方向 | 说明 |
|---|---|
| 更多政务场景 | 公积金查询、异地就医备案、社保认证等 |
| 热词唤醒 | "小浙小浙"免按钮唤醒（需考虑隐私） |
| 方言支持 | 接入方言 ASR 模型 |
| 常做任务免问 | 记住用户偏好，减少重复授权 |
| 独立子女端 | 远程查看操作记录 + 通知 |
| 跨 APP 服务 | 从政务扩展到医疗、生活缴费等 |

## 7 验收与测试计划

### 7.1 可用性测试

- **对象**：5–8 位 60 岁以上老年用户
- **方法**：任务测试 + 半结构式访谈
- **测试任务**：覆盖 4 个核心场景
- **指标**：
  - 任务完成率
  - 任务完成时间
  - 操作错误率
  - 用户满意度（SUS 量表或自定义）
  - 信任感评分

### 7.2 场景验收标准汇总

| 场景 | 关键验收点 |
|---|---|
| 登录 · 刷脸 | 代理全程只引导不代操作；刷脸用户自己做 |
| 登录 · 验证码 | 代读代填前征得授权；"登录"按钮用户按 |
| 医保缴费 | 敏感字段单独授权；"去支付"按钮用户按；草稿可恢复 |
| 医保/养老金查询 | 代理可全链路完成；结果语音 + 屏幕双通道呈现 |
| 超出能力 | 浙里办没有的说"没有这个服务"；有但代理不会的指路 + 坦白 |
