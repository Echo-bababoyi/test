## 一、产品定位与终极目标

### 1.1 一句话定位

**一款以浙里办长辈版为蓝本、注入"受控响应型智能代理"的高保真仿真 Web 原型，用于验证"代理协助而非替代老年用户"这一设计假设在真实老年用户身上的可用性。**

> **受控响应型智能代理**：代理仅在用户明确唤醒并发出指令后响应，不主动打扰；所有关键操作前必须预告并征得用户同意；用户随时可叫停。与"全自动代理"的根本区别在于：**用户始终是决策主体，代理是执行工具**。

### 1.2 终极目标演示场景

答辩当天，指导老师打开 Chrome 浏览器，看到如下演示：

1. 一位老年用户对着浏览器说"帮我查养老金"；
2. 屏幕下方浮出气泡："我准备帮您打开养老金查询页面，请点确认。"
3. 用户点"确认，帮我做"；
4. 页面自动导航到养老金查询页，语音播报"已打开，您看到三张险种卡片"；
5. 用户再问"这个灵活就业是什么意思"，代理语音解释，不跳页、不打断浏览；
6. **全程：老年用户听得懂、看得清、知道代理下一步要做什么、随时可以叫停。**

配套产出：
- 一份**毕业设计论文**（含用户测试报告）
- **5-8 位老年用户可用性测试**的数据 + 视频记录
- 一份**交互设计规范**（论文附录形式）

### 1.3 与开题报告的对齐

开题报告 3.1 列出四项预期成果（论文 / 多模态交互方案 / 高保真原型 / 可用性测试报告）——本 PRD 全量承接。

**当前实际技术栈**（描述现状，不做规定）：Flutter Web（Chrome）+ FastAPI + DeepSeek-V3 + agno Agent 框架。技术栈选型权属于架构师，此处仅记录事实。

## 二、用户研究

### 2.1 调研概述

本项目开展**两阶段用户调研**，均已完成。

| 阶段 | 时间 | 方法 | 人数 | 目标 |
|---|---|---|---|---|
| 阶段一 | 前期 | 参与式观察 + 半结构访谈 + 任务测试 | **7 人** | 识别使用"浙里办"的核心痛点 |
| 阶段二 | 补充 | 半结构深度访谈（智能代理专题） | **6 人**（回访 5 + 新增陈伯伯） | 探索老年用户对智能代理的期待与顾虑 |

**阶段一样本**：王阿姨（62，退休教师）/ 张叔叔（63，退休工人）/ 黄阿姨（56，裁缝）/ 李阿姨（62，退休后勤）/ 陆叔叔（58，保安）/ 吴叔叔（59，保安）/ 郑阿姨（54，保洁）。地点：古荡菜市场、西湖区政务大厅、港湾家园 / 三坝雅苑 / 古荡新村。

**阶段二样本**：回访受访者 A-E（王阿姨、张叔叔、李阿姨、郑阿姨、陆叔叔），新增受访者 F（陈伯伯，68 岁，退休工人，从未用过政务 APP）。

### 2.2 阶段一：老年用户使用浙里办的核心痛点

| # | 痛点 | 典型描述 | 设计影响 |
|---|---|---|---|
| P1 | **导航入口迷失**：高频功能隐蔽，语音搜索对方言支持不足 | 张叔叔查养老金认证找不到入口，被迫依赖搜索 | → F1 语音查询导航 |
| P2 | **返回逻辑混乱**：一次返回以为退出，实际在流程中间 | 王阿姨、张叔叔对"连续返回"逻辑不适应 | → UI6 停止按钮 / F7 一键回首页 |
| P3 | **验证码输入中断**：需切换 APP 查短信，切回后页面可能失效 | 几乎所有受访者遇到困难 | → F2 验证码自动填写 |
| P4 | **刷脸识别容错差**：眨眼 / 摇头动作幅度小常被识别失败 | 李阿姨、王阿姨需多次尝试 | → F3 人脸识别语音引导 |
| P5 | **弹窗打断流程**：系统升级提示 / 跳转应用商店中断操作 | 王阿姨、李大叔遭遇流程中断 | → 交互设计准则四 |
| P6 | **缺乏操作安全网**：误触返回键导致表单数据全部丢失 | 王阿姨填表时误触返回，数据清空 | → F8 草稿自动保存 |
| P7 | **依赖线下帮助**：平台缺乏有效引导机制 | 张叔叔、李阿姨仍需亲属陪同 | → 代理"受控响应"核心定位 |

**用户原话摘录**：
- 王阿姨："刷脸认证要我缓慢摇头，可怎么摇都说我超时"
- 张叔叔："能不能像微信一样，我说'查养老金'它就给我查了"
- 陆叔叔："一上来就要身份证号，我不敢填，怕是骗子"

### 2.3 阶段二：对智能代理的期待与顾虑

#### 2.3.1 代理形象（**已决策**）

用户希望代理像"耐心的工作人员"或"客服"，而非"管家"或"家人"。王阿姨："像个耐心的客服，说话慢一点"；陈伯伯："像个老邻居，和和气气的"；张叔叔："不要搞成卡通人物，太幼稚"。

**决策**：不设具象虚拟人物形象；以简洁**助手图标**作为视觉锚点，颜色使用**温暖的蓝色或绿色**，区别于其他功能按钮。（来源：交互逻辑设计文档 §二.2.4）

#### 2.3.2 唤醒方式

用户同时需要**按键唤醒**和**语音唤醒**两种方式：
- 王阿姨："最好有个大按钮，按一下就能说话"
- 张叔叔："最好直接对着手机说'帮我办'"
- 李阿姨："怕手机一直听着费电"，倾向先按键再说话

#### 2.3.3 能力边界（用户愿意 / 不愿意委托）

| 操作类型 | 意见 | 处理方式 |
|---|---|---|
| 查找功能入口 | 普遍愿意 | F1 直接支持 |
| 填写短信验证码 | 多数愿意 | F2（L2 权限，首次询问）|
| 填写身份证号 | 有分歧（陈伯伯坚决拒绝）| L3，每次单独确认 |
| 刷脸认证 | 有分歧 | F3（L2，引导为主，代填可选）|
| 支付操作 | 普遍不愿意 | 不进入代理能力范围 |

#### 2.3.4 授权机制需求

- 敏感操作（身份证、支付）：**每次必须事先询问**
- 非敏感操作（验证码）：首次授权后可不重复询问
- 授权说明必须包含"为什么要填"和"会不会有风险"

#### 2.3.5 叫停与控制需求（**3 种中断方式**）

| 用户偏好 | 对应设计 |
|---|---|
| 王阿姨："按一下屏幕上的'停'字" | 屏幕固定位置红色停止按钮 |
| 张叔叔："大声说'不办了'" | 语音中断，识别"停 / 停下 / 不办了 / 取消" |
| 陆叔叔："按返回键" | 物理返回键触发中断确认弹窗 |
| 陈伯伯："手机晃一晃" | P2 可选（体感中断）|

#### 2.3.6 不信任情境（需重点规避）

1. 代理**未授权自主操作**（张叔叔："自己弹出来帮我弄，没叫它"）
2. 代理**动作过快**，用户看不清（郑阿姨）
3. 代理**乱跳页面或卡住**（李阿姨）
4. **出错后无法纠正**（陆叔叔）

#### 2.3.7 反馈偏好

多数用户首选**语音 + 视觉双重反馈**：
- 王阿姨："做完事最好说话告诉我，屏幕上也要闪一下"
- 郑阿姨："屏幕上显示步骤，让我知道它做到哪一步了"

#### 2.3.8 透明与追溯需求

约半数用户希望查看操作记录（王阿姨、李阿姨、郑阿姨）；另半数认为做完告知即可，不需查记录（张叔叔、陆叔叔、陈伯伯）。**设计上需提供记录入口，但设为非显眼位置，不强迫查看**。

#### 2.3.9 方言需求

陈伯伯希望支持杭州方言。其他用户普通话可接受。（列为 P2 可选项，见 §三 F-P2）

### 2.4 用户画像（3 类）

#### 画像一：王阿姨型 ——"愿意学但怕错"

| 维度 | 内容 |
|---|---|
| **年龄段** | 60-65 岁 |
| **典型描述** | 退休教师 / 教职人员，熟练微信+支付宝，有政务 APP 使用经验但体验不佳；识字率高，有学习意愿，但担心填错信息或误操作带来后果 |
| **核心诉求** | 代理耐心引导，操作前告知；出错能纠正；有记录可查 |
| **抗拒点** | 刷脸认证不敢委托（"怕替我摇头摇不好"）；支付坚持自己来 |
| **验收标准映射** | T1 场景B（语音查询）应在代理提示下主动唤起；填验证码时能理解"帮填"是什么意思并授权 |
| **代理偏好** | 女声 / 慢速 / 按键唤醒；填身份证必须每次确认 |

#### 画像二：张叔叔型 ——"信任人胜过信任系统"

| 维度 | 内容 |
|---|---|
| **年龄段** | 63-68 岁 |
| **典型描述** | 退休工人，仅掌握微信基础功能，从未独立使用政务 APP；对线上办事持怀疑态度，习惯线下窗口 |
| **核心诉求** | 代理像"会办事的小伙子"，用简单直接的话告诉他怎么做 |
| **抗拒点** | 支付绝不委托；代理不能"自己弹出来"（未经唤醒的主动介入会让他不信任）|
| **验收标准映射** | 代理介入前必须由用户主动唤醒；IntentProposalCard 不应在用户未开口时出现 |
| **代理偏好** | 男声 / 大声 / 慢速；最想代理帮找功能入口；不用查记录，出错能改就行 |

#### 画像三：郑阿姨型 ——"想用但没时间学"

| 维度 | 内容 |
|---|---|
| **年龄段** | 54-60 岁 |
| **典型描述** | 仍在务工的保洁人员，熟练使用微信和工作软件，对新技术接受度高但缺乏探索耐心；时间有限，希望办事快 |
| **核心诉求** | 代理帮她把事办完以节省时间；愿意授权较多，但交钱前需最终确认 |
| **抗拒点** | 代理动作太快看不清（会触发不信任）；交钱环节需自己最后看一眼 |
| **验收标准映射** | 代理执行进度必须实时屏幕可见（UI5）；执行速度有视觉缓冲（不允许闪跳）|
| **代理偏好** | 女声 / 正常速度 / 按键或语音均可；愿意委托大部分操作；首次询问后续可免重复 |

## 三、功能需求

> **说明**：涉及代理行为（唤醒/交互模式/权限/反馈）的细节规格见 §六；本节只定义"用户需要什么"。

| 编号 | 需求 | 优先级 | 验收标准（输入 → 操作 → 期望结果）|
|---|---|---|---|
| **F1** | **语音查询养老金或医保**：用自然语言说出需求，系统理解意图并自动跳转目标页面、播报查询结果 | **P0** | 输入"帮我查养老金" → 代理复述+确认 → 页面跳转至养老金查询页 + 语音播报最近到账记录 |
| **F2** | **验证码自动填写**：收到短信验证码后，代理主动询问是否帮填，用户确认后自动读取并填入 | **P0** | 触发验证码发送 → 代理弹出询问卡 → 用户点"帮填" → 输入框逐位填入 + 代理提示"填好了您确认一下" |
| **F3** | **人脸识别语音引导**：进入刷脸环节时，语音说明具体动作，屏幕播放示范动画；失败时提供具体建议而非只说"失败" | **P1** | 进入 FaceAuthPage → 代理语音"现在像我这样眨一下眼睛" + 播放动画 → 失败时"没关系，可能是动得快了，我们再试一次" |
| **F4** | **关键操作授权**：执行敏感操作（填身份证号）前，代理必须事先询问并等待明确确认；说明包含"为什么要填"和"风险是否存在" | **P0** | 触发身份证填写 → 代理弹窗"需要填身份证号来核实身份，是否允许" → 用户点"允许"才执行 |
| **F5** | **操作中断与叫停**：用户可随时中断代理，方式包括：语音"停"、点击停止按钮、按返回键（详见 §六.7）| **P1** | 代理执行中 → 用户说"停" → 代理立即停止 + 语音"已停止" + 屏幕返回中断前状态 |
| **F6** | **操作记录与追溯**：记录代理历史操作（时间/内容/结果），支持查看；出错时可撤销 | **P2** | 打开"助手记录" → 显示倒序操作列表 → 点击某条可查详情和"撤销"按钮（仅支持页面跳转和表单填写的撤销，不含已提交操作）**【Phase 3 范围：`InteractionLogger` 仅写 LocalStorage（D3），记录列表 UI 留 Phase 4】**|
| **F7** | **一键返回首页**：用户说"回首页"或"不办了"，代理直接返回长辈版主页，而非逐层返回 | **P1** | 用户在任意页说"不办了" → 代理确认"好的，已回到首页" → 直接导航至长辈版首页 |
| **F8** | **草稿自动保存**：用户填写表单时代理自动保存草稿；误触返回时询问是否恢复 | **P1** | 用户填写表单时误触返回键 → 代理弹出"您刚才在填表，要不要先保存？" → 点"保存草稿"后下次进入提示恢复。**实现方式：前端 LocalStorage（单设备持久化）；多设备同步能力属未来扩展项，不在本毕设范围。TTL 30 天（与 UI7 对齐）。** |

**明确不做**：
- ❌ 真实接入浙里办后端 API（所有数据走 mock）
- ❌ 原生 Android / iOS 安装包（Web 端唯一平台）
- ❌ 支付操作委托（用户调研显示普遍不愿意，风险超出研究范围）
- ❌ 个性化推荐 / 跨会话记忆（单次会话，不持久化用户数据）

## 四、非功能需求

| 编号 | 需求 | 指标 | 备注 |
|---|---|---|---|
| **NF1** | 语音响应速度 | 用户说完指令到代理开始响应 **≤ 2 秒** | 含 STT 返回 + 意图理解 + TTS 首音 |
| **NF2** | 语音识别准确率 | 普通话识别率 **≥ 90%**；对杭州话提供基础支持（P2）| 第三方 STT API |
| **NF3** | 语音播报速度 | 默认 **慢速约 120 字 / 分**；用户可在设置中调节 | 老年友好音色 |
| **NF4** | 界面字体大小 | 关键文字（确认按钮、验证码等）字号 **≥ 16pt**；代理响应文字 ≥ 18pt | 符合适老化标准 |
| **NF5** | 界面对比度 | 前景与背景对比度 **≥ 4.5:1**（WCAG AA 级）| 含代理气泡 / 确认卡 |
| **NF6** | 触控目标尺寸 | 可点击按钮最小尺寸 **≥ 10mm × 10mm**；代理授权按钮 ≥ 48dp | 特别注意返回按钮固定位置 |
| **NF7** | 信任感 | 可用性测试后信任度评分（5 分量表）**≥ 4 分** | 验证"受控响应型"设计假设 |

## 五、界面需求

| 编号 | 需求 | 规格说明 |
|---|---|---|
| **UI1** | **代理唤醒入口**（助手按钮）| 首页底部固定设置；颜色区别于其他按钮（温暖蓝 / 绿色）；始终可见，不随页面滚动隐藏；≥ 10mm × 10mm |
| **UI2** | **语音交互状态条** | 用户说话时屏幕底部弹出语音条；显示收音波形动画 + 实时文字（用户说的话）；明确显示"正在听您说话" |
| **UI3** | **意图确认卡** | 代理复述用户指令时屏幕弹出；包含复述文字 + "确认" / "重说" 两个按钮；等待 5 秒超时后代理询问"还在吗" |
| **UI4** | **执行预告** | 代理执行操作前屏幕高亮即将操作的目标区域（呼吸灯效果，持续 3-5 秒）；同步语音说明 |
| **UI5** | **进度提示** | 代理执行操作时，屏幕顶部或底部显示进度文字（如"正在打开养老金查询页面"）+ 加载动画 |
| **UI6** | **停止按钮** | 屏幕固定位置设置红色"停止"按钮；任何时候点击即可中断代理操作；代理激活期间始终可见 |
| **UI7** | **操作记录入口** | 首页或助手设置中提供"助手记录"入口；操作列表按时间倒序排列；每条记录含时间 / 内容 / 结果；最近 30 天 |

> 各组件详细视觉规格（布局位置 / 尺寸 / 颜色 token / 圆角 / 交互态 / 动效）见 `docs/AGENT_UI_SPEC.md`（architect review 通过后为 frontend 实施依据）。

## 六、代理设计规范

### 6.1 角色定位

代理定位为**"耐心的工作人员"**，模拟政务大厅窗口办事员的职业形象：专业、耐心、可靠，但不越权。

**不是**：管家 / 家人 / 私人秘书（不主动替用户做决定）。

**视觉呈现**（已决策）：
- 不设置具象的虚拟人物形象（避免"恐怖谷"效应，用户调研已确认）
- 以简洁的**助手图标**作为视觉锚点
- 颜色：**温暖的蓝色或绿色**（区别于长辈版橙色主色，不产生颜色冲突）
- 交互状态下以语音条、进度提示、高亮框等形式呈现，而非拟人头像

### 6.2 声音设计

| 维度 | 规格 |
|---|---|
| 默认音色 | 同时提供**女声和男声**两个选项，用户可在设置中切换 |
| 语速 | 默认**慢速，约 120 字 / 分**；用户可调节 |
| 音质 | 温暖、清晰，避免机械感电子音 |
| 方言 | 支持杭州话作为**P2 可选项**，默认普通话；用户可在设置中切"杭州话模式"（Baidu STT `dev_pid` 切换）；不作验收硬指标，"能切就行" |

### 6.3 唤醒机制

#### 6.3.1 按键唤醒（P0，必做）

- 入口：首页底部固定"助手"按钮（对应 UI1）
- 点击后立即唤醒，屏幕底部弹出语音条
- 代理语音反馈："我在，请说"
- 视觉反馈：收音波形动画 + "正在听您说话"提示

#### 6.3.2 语音唤醒（P2，可选）

- 唤醒词："你好助手"
- **默认关闭**，需用户在设置中手动开启
- 开启后低功耗后台监听，30 秒无操作后自动休眠
- 理由：部分用户担心费电 / 误触发（李阿姨："怕手机一直听着费电"）

### 6.4 核心交互模式："预告 - 确认 - 执行"三阶段

所有关键操作均按此三阶段流程执行。

#### 第一阶段：预告

- 代理理解意图后，**先用语音告知**即将执行的操作（例："好的，我帮您查养老金到账记录"）
- 屏幕同步显示代理复述的内容
- 涉及敏感信息时，额外说明目的（例："需要填身份证号来核实身份，我只帮您填这一次"）

#### 第二阶段：确认

- 预告后代理等待用户确认（弹出 UI3 意图确认卡）
- **确认方式**：语音说"对 / 好的 / 行"，或点击"确认"按钮
- **拒绝 / 纠正方式**：说"不对"或点击"重说"，代理重新询问
- 等待超时（5 秒）：代理询问"还在吗？需要我继续吗？"
- 超过 30 秒无响应：代理自动退出并告知"我先退出了，需要的时候再叫我"

#### 第三阶段：执行

- 用户确认后代理开始执行
- 执行中屏幕显示进度提示（UI5）
- 执行完成后代理语音告知结果，屏幕高亮结果区域（UI4）
- 执行失败时进入 §八 异常处理框架

### 6.5 权限分级模型（L1 / L2 / L3 完整三级）

代理默认处于最低权限，用户根据信任程度可逐步授权升级。

| 级别 | 名称 | 能力范围 | 授权要求 | 适用场景 |
|---|---|---|---|---|
| **L1** | **引导模式** | 仅用语音提示和屏幕高亮引导用户操作；不自动点击、不自动填写 | 无需授权，唤醒即可 | 用户想自己操作但需要指引 |
| **L2** | **代填模式** | 在 L1 基础上，可代填**非敏感信息**（短信验证码、地址等）| 首次使用时弹窗询问，用户同意后同类操作后续自动执行 | 用户信任代理处理重复性、非敏感任务 |
| **L3** | **代办模式** | 在 L2 基础上，可代填**敏感信息**（身份证号）、代提交申请 | **每次敏感操作前必须单独弹窗确认**，用户点"同意"方可执行 | 用户高度信任代理，愿意委托复杂任务 |

> **重要**：支付操作**不进入任何权限级别**（用户调研中普遍拒绝委托，设计上不提供此能力）。

**权限升降级**：
- 升级：用户在"助手设置"中手动升级，系统说明各级能力和授权要求
- 降级：用户随时降级，或在某次操作选"这次我自己来"（单次手动，不永久降级）

> **Phase 3 实施范围（D4）**：Phase 3 实施 L1 + L2；L3（身份证级敏感操作，每次单独弹窗确认）留 Phase 4 实施。答辩核心场景 A（刷脸引导）/ B（语音查询）/ C（验证码填写）均不依赖 L3。

### 6.6 反馈机制

#### 6.6.1 语音反馈规范

以下关键节点必须有语音反馈：
- 唤醒时："我在" / "我在，请说"
- 意图确认后：复述用户指令并等待确认
- 执行操作前：预告即将执行的操作
- 执行完成后：告知结果
- 出错时：先安抚再给建议（不责备用户）

语速默认慢速（120 字 / 分）；音量适中；语调平稳温和。

#### 6.6.2 视觉反馈规范

与语音反馈**同步进行**，确保信息冗余传达：

| 元素 | 触发时机 | 表现 |
|---|---|---|
| 语音条（UI2）| 代理被唤醒 | 屏幕底部弹出，显示收音波形 + 用户说话实时文字 |
| 意图确认卡（UI3）| 代理预告后 | 屏幕中央弹出，含复述文字 + "确认" / "重说" |
| 高亮引导（UI4）| 代理引导点击某按钮 | 目标按钮呼吸灯光晕，持续 3-5 秒 |
| 进度提示（UI5）| 代理执行操作中 | 屏幕顶部 / 底部进度文字 + 加载动画 |
| 结果高亮 | 执行完成后 | 高亮显示结果区域（如养老金金额、缴费记录）|

#### 6.6.3 触觉反馈（可选）

用户可在设置中关闭。关键节点短震动（约 20 毫秒）：
- 代理唤醒成功时
- 代理预告后等待确认时
- 操作执行完成时

### 6.7 中断与控制机制

#### 6.7.1 三种中断方式

| 方式 | 操作 | 代理响应 |
|---|---|---|
| **语音中断** | 说"停" / "停下" / "不办了" / "取消" | 立即停止当前操作 |
| **按键中断** | 点击屏幕固定位置红色"停止"按钮（UI6）| 立即停止 |
| **物理返回键** | 按手机返回键 | 代理询问"是否停止当前操作"，用户确认后中断 |

#### 6.7.2 中断后处理

- 代理语音确认"已停止"
- 屏幕返回中断前状态
- 若已执行部分操作（如已跳转页面），代理告知已完成内容，用户可选"继续"或"我自己来"

### 6.8 透明与追溯机制

#### 6.8.1 实时透明

代理执行每一步时，屏幕同步显示当前步骤（进度文字简洁，避免术语，如"正在帮您找养老金"而非"正在执行 NavigateTo intent"）。

#### 6.8.2 操作记录（对应 UI7）

- 首页 / 助手设置提供"助手记录"入口
- 倒序排列，每条记录含：操作时间 / 操作内容 / 执行结果
- 保存最近 30 天，用户可手动清空

#### 6.8.3 纠错机制

| 方式 | 说明 |
|---|---|
| 撤销 | 在操作记录中点"撤销"，回退到该操作之前状态；仅支持页面跳转和表单填写，不支持撤销已提交的申请 |
| 手动修改 | 用户直接点击代理填写的输入框，手动修改后代理不再覆盖 |
| 重新执行 | 说"重新帮我查"或点"重试"，代理重新执行当前任务 |

### 6.9 对话风格规范

#### 6.9.1 称呼

代理称呼用户为"**您**"，代理自称为"**我**"。示例："我帮您查一下" / "您看看对不对"。

#### 6.9.2 指令用语

指令用"请"字开头，语气柔和。示例："请点一下允许" / "请看着屏幕"。

#### 6.9.3 安抚用语

失败时先说"没关系"或"没成功"，再给具体建议，**不责备用户**。示例："没关系，可能是动得快了，我们再试一次，眨眼慢一点就好。"

#### 6.9.4 术语替换对照表

| 技术术语（禁用）| 口语替换（使用）|
|---|---|
| 识别失败 | 没成功 |
| 网络超时 | 网络不太稳 |
| 授权 | 允许 |
| 提交 | 交上去 |
| 验证 | 确认一下身份 |

### 场景 A：登录与人脸识别引导

**触发**：用户进入长辈版主页，点击"立即登录"并进入刷脸认证环节。

#### 主流程

| 步骤 | 用户动作 | 代理 / 系统响应 |
|---|---|---|
| A1 | 进入长辈版主页，点击"立即登录" | 跳转登录页，手机号默认已填入 |
| A2 | 点击"同意隐私条款" | 进入身份验证页，默认显示刷脸入口 |
| A3 | 点击"开始验证" | 代理介入，语音："您好，需要刷脸验证身份。我教您做，很简单。" |
| A4 | — | 弹出应用内刷脸请求弹窗；代理语音："请点一下同意并继续。"屏幕高亮"同意并继续" |
| A5 | 点击"同意并继续" | 弹出手机系统摄像头权限请求；代理语音："请点一下允许，打开摄像头。"屏幕高亮"允许" |
| A6 | 点击"允许" | 进入人脸识别页；代理语音："现在看屏幕，像我这样眨一下眼睛。"屏幕播放眨眼示范动画 |
| A7 | 完成眨眼动作 | 代理语音："很好。接下来请保持不动，屏幕会闪一下，不用紧张。" |
| A8 | 保持不动完成颜色验证 | 代理语音："验证通过，已经帮您登录了。"系统返回长辈版主页 |

#### 异常分支

- **A-Ex1**（眨眼失败）：→ 见 §八 AE5（执行失败）；代理安抚"没关系，可能是动得快了"，重播动画，返回 A6
- **A-Ex2**（连续 3 次失败）：→ 代理提示"刷脸暂时没通过，需要我帮您切换到验证码方式吗"；屏幕显示"切换验证码" / "再试一次"
- **A-Ex3**（用户拒绝授权）：→ 代理"好的，那您需要的时候再叫我"；代理退出，用户可手动操作

### 场景 B：语音查询养老金 / 医保

**触发**：用户在任意页面唤醒代理，说出查询指令。同一场景覆盖两个查询子类型（养老金到账记录 / 医保缴费记录），流程相同，目标页面不同。

#### 主流程

| 步骤 | 用户动作 | 代理 / 系统响应 |

| B1 | 点击首页底部"助手"按钮 | 代理语音："我在，请说。"屏幕底部弹出语音条（UI2）|
| B2 | 说："帮我查一下医保交了没有" / "我想看看养老金发了没有" | 语音条实时显示用户说的话 |
| B3 | — | 代理语音复述："好的，您是要查医保缴费记录，对吗？"屏幕弹出意图确认卡（UI3）|
| B4 | 说"对"或点击"确认" | 代理语音："行，我帮您打开。"屏幕显示进度"正在打开医保缴费页面"（UI5）|

#### 异常分支

- **B-Ex3**（查询无记录）：→ 代理语音"暂时查不到记录，可能是今年还没交。需要我帮您打开缴费页面吗？"屏幕显示"打开缴费" / "返回首页"

### 场景 C：验证码自动填写

> **介入模式**：B-半主动（代理上下文感知触发，用户确认后才执行；详见 §九 D1 豁免）。

**触发**：用户在登录（VerifyPage）或业务办理时，系统发送短信验证码。

> **仿真说明**：Web 端浏览器无法读取手机短信，本场景使用固定 mock 验证码（如 `123456`）。用户测试前需告知被试"假装手机收到了 123456"。

#### 主流程

| 步骤 | 用户动作 | 代理 / 系统响应 |
|---|---|---|
| C1 | 触发短信验证码发送 | 手机收到短信，通知栏显示验证码 |
| C2 | — | 代理检测到验证码到达；自动弹出**非阻塞建议卡（InAppOverlay）**，不打断用户当前操作；代理语音："您收到一条验证码，需要我帮您填上去吗？"显示"帮填" / "自己填"；用户不理则卡片静默消失，不强制响应 |
| C3 | 说"好，你帮我填"或点击"帮填" | — |
| C4 | — | 代理语音先播报数字"验证码是一二三四五六，是这个吗"（让用户先确认）；用户确认后代理自动填入输入框，逐位高亮填入 |
| C5 | — | 代理语音："填好了，您确认一下对不对。"屏幕高亮验证码输入框 |
| C6 | 核对后说"对的"或**自己点**提交按钮 | **代理不自动提交**；由用户自行点击"登录" / "确认"按钮（L2 权限边界）|

#### 异常分支

- **C-Ex1**（用户拒绝帮填）：→ 代理"好的，您自己填"；询问卡消失，用户手动输入
- **C-Ex2**（验证码过期）：→ 见 §八 AE5；代理"验证码已经过期了，需要重新发送吗？"屏幕显示"重新发送" / "取消"

### 辅助场景：中途退出与草稿保存

此场景为辅助流程（F7 / F8），不列为 3 个核心场景之一。

- **主动退出**：用户说"不办了"或"回首页" → 代理"好的，已回到首页" → 直接返回长辈版主页
- **误触返回（草稿）**：用户填写表单时误触返回键 → 代理弹出"您刚才在填表，要不要先保存？" → 用户选"保存草稿"后代理提示"已保存，下次进来我提醒您"
- **恢复草稿**：下次进入同一表单 → 代理"您上次填了一半，要继续填吗？" → 用户选"继续填写"则自动回填

## 八、异常处理框架

> 所有场景中遇到异常，按以下框架处理。§七 场景流程中的异常分支用 AE 编码引用此节，不在场景内重复逻辑。

| 编号 | 异常类型 | 触发条件 | 代理响应 | 用户操作入口 |
|---|---|---|---|---|
| **AE1** | **未听清指令** | 语音识别置信度低于阈值 | 语音："没听清，请再说一遍"；屏幕显示"没听清" + "重说" / "取消" | 用户可重说或取消 |
| **AE2** | **识别错误（意图偏差）** | 代理理解内容与用户意图不符 | 复述后询问"对吗"；屏幕显示"确认" / "不对"；用户说"不对"后代理重新收音 | 用户可纠正或重说 |
| **AE3** | **操作超时** | 代理等待用户确认超过 5 秒 | 语音："还在吗？需要我继续吗？"屏幕显示"继续" / "结束" | 用户可选继续或结束；30 秒无响应代理自动退出 |
| **AE4** | **网络异常** | 代理执行中检测到网络中断 | 语音："网络不太稳，建议稍后再试"；屏幕显示"重试" / "取消" | 用户可重试或取消 |
| **AE5** | **执行失败** | 代理无法完成指令（如页面改版找不到入口 / 刷脸动作未被识别）| 语音说明失败原因（用口语，见 §六.9）+ 给出替代建议；屏幕显示备选选项（如"查社保局电话" / "换验证码登录"）| 用户可选备选方案或返回首页 |

## 九、设计准则

以下 5 条准则来自交互逻辑设计文档（§一.2），为核心设计指导原则，优先级高于下方补充列表。

| 编号 | 准则 | 含义 |
|---|---|---|
| D1 | **受控响应** | 代理仅在用户明确唤醒并发出指令后响应，不主动打扰；所有关键操作前必须预告并征得同意 |
| D2 | **透明可见** | 代理的每一步操作在屏幕上同步显示，用户始终知道代理在做什么、做到哪一步 |
| D3 | **冗余互补** | 关键信息通过语音和视觉双通道同时传递，确保任一感官通道退化不影响信息接收 |
| D4 | **容错可逆** | 允许用户犯错并提供恢复路径；所有关键操作可撤销或修改 |
| D5 | **渐进信任** | 代理默认权限最低，用户根据使用体验逐步授权升级，而非一次性获取全部权限 |

> **D1 豁免 — 场景 C 上下文触发特例**：当用户已进入验证码输入页且系统检测到验证码到达，代理可主动显示**非阻塞建议卡**（InAppOverlay，用户不理即不响应）；用户须点"帮填"方可执行。此为上下文感知触发，非随机打扰，豁免边界：① 当前页为验证码输入页 ② 非阻塞 ③ 仍需显式确认。三个边界条件同时满足方可适用，不得援引此特例扩大其他场景的主动介入范围。

**与用户需求文档 5 条准则的对照**：

| 交互逻辑文档准则 | 对应用户需求文档准则 | 说明 |
|---|---|---|
| D1 受控响应 | 准则一（流程适配优先于界面适配）| 互补，D1 强调代理行为，准则一强调适老化整体方向 |
| D2 透明可见 | 准则二（提供安全网）| 互补，D2 是实时透明，准则二含二次确认 / 草稿 |
| D3 冗余互补 | 准则三（多模态互补）| **一致**，直接对应 |
| D4 容错可逆 | 准则五（操作可逆）| **一致**，直接对应 |
| D5 渐进信任 | 准则四（非打断式提醒）| 补充关系；准则四约束系统弹窗行为 |

> **冲突处理**：两套准则无实质冲突。交互逻辑文档更聚焦代理交互，以 D1-D5 为主；准则四（非打断式）是对系统层面的额外约束，并行适用。
