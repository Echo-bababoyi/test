## Git 推送策略：只 commit，不自动 push

做完 `git commit` 就停在本地。只有当用户显式说"推送 / 同步 / push / 推到 github"才推送。

**Why**：用户（2026-04-22 指示）希望批量控制远端更新时机，偏好"本地积累多个 commit → 需要时一次性推送"。

**How to apply**：
- 工作流里只 `git commit`，不要自动补 `git push`
- 远端仓库为 `https://github.com/Echo-bababoyi/test.git`（本地 repo 目前尚未配置 remote）
- 首次推送前需 `git remote add origin <url>`；注意分支名：本地 `master`，远端默认 `main`，用 `git push origin master:main` 或先本地 `git branch -m main`
- 推送前例行：`git log --oneline origin/main..HEAD` 预览、`git status` 确认、force-push 类操作先二次确认

## 不考虑时间进度压力

做计划、PRD、gap 分析、建议、风险评估时**不要把"时间不够"作为约束**，不要因时间降级方案、砍功能、推荐妥协。按"该怎么做就怎么做"输出。

**Why**：用户（2026-04-18 指示）明确说"以后你就不要考虑时间上的问题"。此前 PRD 中"4/23 来不及/buffer 为 0/压缩时间线"让用户觉得被时间绑架。用户对毕设答辩时间自有安排。

**How to apply**：
- 按功能完整度排序，不按"能在 X 天内完成"排序
- 不写"由于时间限制建议砍掉 XX"
- 不在文档开头/结尾渲染"时间紧、buffer 少"的紧迫感
- 开题报告里的日期（4/23 测试、5/15 答辩）仅作参考，不作硬约束
- 分期任务按"依赖关系"或"P0/P1/P2"分层，不按"第 X 周做"分层
- 用户主动问"来得及吗"再评估

## TODO 三态流转

**🔲 待修复 → 🧪 待测试（已修复，等用户验证）→ ✅ 已修复（用户确认）**

提交后只能标 🧪，必须等用户真人测试确认才标 ✅。

**Why**：代码修完 ≠ 问题解决，必须人验。

**How to apply**：
- 每个项目一个 TODO 文件（本项目 `docs/TODO.md`），不主动并行建第二个
- 接手遗产项目且用户明确说"另起一个"才走双文件流程
- 用户提新问题时（"还有一个"、"另外"、"顺便"），**先加入 TODO 为 🔲 再进入调查流程**

## 删除文件必须单独确认

不要在用户回复其他问题时顺带删除文件。删除操作必须单独确认，尤其是 untracked 文件（git 无法恢复）。

**Why**：用户说"继续吧"是确认执行任务，不是同意删文件。擅自删除 untracked 截图无法恢复。

**How to apply**：批量操作里如果牵涉 `rm`，必须单独列出清单问一次。

## 测试前四段说明

让用户测试之前，必须主动整理并报告：
1. **问题描述**：问题本身是什么
2. **根因**：为什么会出现
3. **解决方法**：具体改了什么、为什么能解决
4. **测试方法**：用户如何操作验证

**Why**：用户多次要求提供完整说明才测试，缺任何一部分都会被追问。

**How to apply**：提交后、让用户测试前，从成员或 git diff 收集信息再报告。不要让用户自己猜怎么测。

## 文档→代码边界要等命令

文档层（PRD / SPEC / schema / review）可连续自主推进到准备就绪。一旦要跨入"实际写产品代码"（Phase 实施开工、任何 feat 级改动），**必须停下等用户明确说"开始实施 / 开工"**。铺好流水线的水闸要用户开。

**Why**：用户（2026-04-18）明确：文档阶段可多走几步不用问，代码落地必须用户拍板。

**How to apply**：看到文档 review 通过时，**不要**立刻派 frontend/backend 写代码，先汇报"文档就绪，可以开工时说一声"。
