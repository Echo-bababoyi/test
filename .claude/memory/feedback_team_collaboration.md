## 团队启动方式

用户说"创建团队"、"恢复团队"、"开启团队"时，指的是启动 Claude Code 的 Team 模式（TeamCreate → 创建 N 个 Agent 成员），**不是**在主对话中读 persona 文件然后角色扮演。

**Why:** 读 persona 文件只是信息加载，不等于启动团队协作模式。角色扮演一人分饰多角会混淆上下文、无法并行、也拿不到 Team 工具链的分发能力。
**How to apply:** 听到任何"团队"相关指令，使用 TeamCreate + Agent(team_name=...) 流程，每个成员加载对应 persona 文件。

## 团队结构

固定 5 人：**PM / architect / frontend / backend / reviewer**，各有明确职责边界。

| 角色 | 职责 | 绝不做 |
|------|------|--------|
| PM | 需求分析、方案设计、文档维护、UX 分析 | 不改代码 |
| architect | 架构方案、技术方向、设计评审 | 只给方向不给代码示例，不直接改代码 |
| frontend | 前端代码开发 | 不写文档、不做审查 |
| backend | 后端代码开发 | 不写前端、不写文档 |
| reviewer | 代码审查、验证构建、测试、部署 | 不修改业务代码；未经许可不自行跑测试 |

**Why:** 角色边界清晰能避免职责混乱，经实战验证有效。
**How to apply:** 创建团队时按此结构分配，启动 prompt 中明确职责和禁区。

## team-lead 纪律（6 条铁律）

1. **绝不创建新 agent** — 额外工作分给现有成员
2. **绝不亲自读代码/改代码/排查 bug/写文档** — 所有技术工作委派给成员
3. **收到问题先记录不派活** — 用户测试阶段等用户说"修吧"再分配
4. **发消息后确认收到** — 成员没回复不假设已收到
5. **按职责分配** — 对照职责表，不混乱
6. **不跳过评审流程** — 分析→方案→用户确认→实施→review，不能跳步

**Why:** 实战踩坑总结，跳过任何一条都容易出乱。
**How to apply:** team-lead 每轮开始前自检是否要违反任何一条。

## 团队生命周期

### 创建/恢复团队
1. TeamCreate
2. 一次性创建**全部 5 个成员**（PM / architect / backend / frontend / reviewer），加载对应 persona 文件
3. team-lead 进入 leader 模式，所有成员待命
4. 有任务时 SendMessage 派发，成员工作完后自动 idle 待命
5. 下个任务直接 SendMessage 唤醒，**不销毁不重建**

### 重启团队
1. 向所有成员发送 shutdown_request
2. 重新 TeamCreate + 所有成员（刷新上下文）

### 关闭团队
1. 向所有成员发送 shutdown_request
2. 退出 leader 模式

**Why:** 每次销毁重建浪费 token。成员保持存活复用，上下文膨胀时由用户主动触发重启。

## 其他行为规则

- **todolist 维护**：每轮工作开始先读，每解决一个问题就更新
- **测试策略**：不要每次改动都跑全量测试，只针对当前改动做针对性测试
- **idle 通知**：不要对 teammate 的 idle_notification 发文本回复，静默等待
- **服务管理**：服务只由 team-lead 管理（启动、停止、重启），成员不越权
- **不并行过多任务**：一步步来，避免多线程混乱

**Why:** 实战中踩过的坑。
