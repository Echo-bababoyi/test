## 三态流转

三种状态：**🔲 待修复 → 🧪 待测试（已修复，等用户验证） → ✅ 已修复（用户确认）**。

提交后只能标 🧪，必须等用户真人测试确认后才标 ✅。

**Why:** 代码修完不等于问题解决，必须经过真人验证。
**How to apply:** team-lead 在 git commit 后同步更新 todolist 为 🧪，绝不在用户确认前标 ✅。

## 用户提出新问题时的流转

用户提出新问题 → 第一时间加入 todolist 为 🔲，再进入调查/修复流程。

**Why:** 问题在对话中提及但不落入文件，对话中断后丢失。
**How to apply:** 听到"还有一个问题"、"另外"、"顺便"这类前缀时，优先记录再回应。

## todolist 文件约定

**默认：一个项目一个 todolist 文件。** 命名与位置由项目约定（例如 `docs/TODO.md`），新问题一律写入此文件。不要主动创建第二个并行的待办文件。

**例外——接手遗产项目：** 当新加入一个已有旧 `TODO.md` 的项目、且用户明确要求"新建一个"时，才保留旧文件作为只读历史，另建活文档。这是**特例**，不是默认姿势。

**Why:** 历史上踩过"两个并行 todolist 互相脱同步"的坑，把规则泛化成了"项目通常有多个"——反而导致新项目也莫名其妙多建一个文件制造混乱。
**How to apply:** 接手项目先问：当前有几个 todolist 文件？只有一个就直接用；没有就建一个；只有在用户明确要求"保留旧的、另起一个"时才走双文件流程。
