## 功能增强开发流程

### 一、需求分析（PM）
1. PM 分析需求：用户故事、验收标准、优先级
2. PM 输出需求文档到 docs/，向 team-lead 汇报

### 二、技术方案（架构师）
3. 架构师读需求文档，设计技术方案（文件/函数/改动点）
4. 架构师输出方案文档，评估改动量和风险

### 三、用户确认
5. team-lead 向用户汇报：需求摘要 + 技术方案 + 改动范围
6. 用户确认后进入实施

### 四、实施（开发）
7. 派成员按方案执行
8. 成员完成后汇报

### 五、代码 Review（架构师）
9. 架构师 review 代码改动
10. 有问题 → 修正；通过 → 提交

### 六、提交 + 测试
11. team-lead 提交，todolist 标 🧪
12. 用户测试确认 → 标 ✅

**与 bug 修复的区别**：前面多了 PM 需求分析 + 架构师方案设计两个阶段，确保先想清楚再动手。

## Bug 修复完整流程

### 一、任务选择
1. team-lead 读 todolist，评估最应优先处理的问题
2. 向用户汇报推荐理由，用户同意后进入下一步

### 二、调查阶段
3. 派相关成员（前端/后端）调查根因并提出方案
4. 成员报告：什么导致了问题、具体要改哪些文件的哪些位置、为什么这样改能解决

### 三、方案审查
5. 派架构师审查方案（是否合理、有无遗漏、边界情况、风险点）
6. 架构师给出 review 意见（✓ 通过 / ⚠️ 需修正）
7. 向用户报告方案 + 架构师意见，用户确认后执行

### 四、实施阶段
8. 派成员按审查通过的方案执行代码修改
9. 成员完成后汇报改动摘要 + 编译/构建验证

### 五、代码 Review
10. 派架构师 review 实际代码改动（git diff 逐文件检查）
11. 有问题 → 回到步骤 8 修正；通过 → 进入提交

### 六、提交
12. team-lead 执行 git commit
13. 更新 todolist：状态表标 🧪，提交记录区**插入最上面**（新记录在前，旧记录在后）

### 七、用户测试
14. 用户真人测试验证
15. 用户确认无误后 → team-lead 将 todolist 状态从 🧪 改为 ✅
16. 用户发现问题 → 记录问题，回到步骤 3

## 辅助规则

### 修复后必须解释
成员修完代码只说"已修"不行，必须说明成因、改了哪里、为什么能解决。

**Why:** 只说"已修"无法判断修得对不对、全不全。

### 反复 bug 升级
同一问题第 1 次正常修；第 2 次排查根因；第 3 次必须实际运行验证附日志证明。

**Why:** 避免表面修复、反复返工。

### 提交前检查
.gitignore 是否覆盖敏感文件、未跟踪文件是否该提交。

**Why:** 避免提交垃圾文件或遗漏关键文件。

### 报告写文件
大篇幅内容写到 docs/，消息中只简短汇报结论。

**Why:** 用户可以自己读文件，完整转述浪费 token。

### 不堆积改动
每个问题修完 review 通过后立即提交，不攒多个一起提交。todolist 等元数据文档不单独 commit，随功能修复一起提交。

**Why:** 大量改动堆积在工作区容易丢失变更历史，也难以回滚。文档单独 commit 会产生无意义的提交记录。

### 测试前必须提供完整说明
在让用户测试之前，team-lead 必须向用户报告四部分内容：
1. **问题描述**：详细描述问题本身是什么
2. **根因**：详细描述为什么会出现这个问题
3. **解决方法**：详细描述具体改了什么、为什么这样改能解决
4. **测试方法**：详细描述用户如何操作来验证修复

信息从成员处收集整理后再报告，不要让用户自己去猜怎么测试。

**Why:** 用户多次要求提供完整说明才测试，缺少任何一部分都会被追问。
**How to apply:** 提交后、让用户测试前，主动整理并报告这四部分。

### 删除文件必须明确确认
不要在用户回复其他问题时顺带删除文件。删除操作必须单独确认，尤其是 untracked 文件（无法从 git 恢复）。

**Why:** 用户说"继续吧"是确认执行任务，不是同意删文件。擅自删除 untracked 截图会无法恢复。

### 用户提出问题立即记录
用户提出新问题时，第一时间加入 todolist 的待解决问题列表，再进入调查流程。

**Why:** 避免问题只在对话中提及但没落入追踪系统，对话中断后丢失。
