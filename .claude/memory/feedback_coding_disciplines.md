## Config 保护

绝不通过放宽 linter / formatter 配置来"修复"警告。遇到 ESLint / ruff / pyproject.toml 规则报错，**修代码而不是改规则**。

**Why**：规则通常有其道理，放宽会让问题在更大范围内蔓延。

**How to apply**：看到 lint 错误先理解规则，再改代码。改规则前必须先和用户对齐。

## 编辑前先查调用方

修改 service/util 层函数（尤其是加参数、改签名）之前，必须先 grep 所有调用点。

**Why**：实战踩坑——加全局参数时容易漏掉内部辅助函数，导致逻辑部分失效。

**How to apply**：引入全局参数（user_id、tenant_id、context 等）时做全量 grep 审计，列完整清单，一次性改完。不"修到哪算哪"。

## Search-first 决策矩阵

写工具函数/新依赖之前，依次查：
1. **项目内** — 有没有类似实现？
2. **包生态**（PyPI / npm / cargo / pub.dev）— 有没有成熟包？
3. **组合** — 能用现有工具组合解决吗？
4. **自建** — 以上都不行才自己写

**Why**：自建代码增加维护负担，重复造轮子不如成熟库可靠。

## Checkpoint 习惯

大改动前（重构、跨模块修改）先打 checkpoint：`git tag checkpoint-<描述>`。验证失败时快速回退。

**Why**：大改动出错没 checkpoint 只能一点点 revert，容易引入新问题。

**How to apply**：判断"大改动"标准——涉及 3+ 文件或核心业务逻辑。

## 改代码必须重启服务（无热加载场景）

uvicorn / vite preview 等不热加载的服务，改代码必须重启，否则新旧代码混跑出幽灵 bug。

**How to apply**：任何代码改动后服务必须显式重启（团队模式下由 team-lead 负责）。

## 验证用实际调用，不用本地 import

后端功能验证用 curl / HTTP 请求打真实运行的服务，不在 REPL 里 import 模块跑。

**Why**：本地 import 绕过依赖注入、中间件、认证等实际路径，假通过。

**How to apply**：验证 API 用 curl，不用 `python -c "from x import y; y()"`。

## 反复 bug 升级

同一问题第 1 次正常修；第 2 次排查根因；第 3 次必须实际运行验证附日志证明。

**Why**：避免表面修复、反复返工。

## 修完必须解释

修完代码只说"已修"不行，必须说明**成因、改了哪里、为什么能解决**。

**Why**：只说"已修"无法判断修得对不对、全不全。

## 提交前检查

提交前核查：`.gitignore` 是否覆盖敏感文件、未跟踪文件是否该提交。

**Why**：避免提交垃圾文件或遗漏关键文件。

## 大篇幅报告写文件

大段内容（分析、方案、review）写到 `docs/`，对话中只简短汇报结论。

**Why**：用户可以自己读文件，完整转述浪费 token。

## 不堆积改动

每个问题修完 review 通过立即提交，不攒多个一起提交。TODO 等元数据文档不单独 commit，随功能修复一起。

**Why**：堆积改动容易丢历史、难回滚；文档单独 commit 产生无意义提交记录。

## 全链路字段一致性

新增/修改 API 字段时检查**前端 → api 层 → service 层 → DB**（以及反向）全链路透传一致。改动后 grep 该字段名在各层出现次数确认一致。

**Why**：只改一端容易字段名不一致，前后端契约断裂。
